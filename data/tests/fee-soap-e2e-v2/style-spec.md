# v2 カルテ文体仕様

目的: 実際の電子カルテSOAP記載に近い文体で書く。v1の「採点情報を含む説明文」を排除する。

## 必須(リント対象)

1. **メタ文の禁止**: 「当日確認した主な診療内容は」「確認すべき論点は」「点数に直結する名称」「会計前に」「算定へ反映」「算定に反映」等の採点メタ文を含まない。
2. **点数表正式名称の禁止**: 全角の正式名称(ＣＴ撮影（１６列以上…）、処方箋料（リフィル以外・その他）等)を本文に書かない。例外: 薬剤名・検査略称など臨床で実際に使う名称(ゲーベンクリーム、HbA1c、インフル迅速)は可。
3. **施設属性の禁止**: CT列数・電子画像管理体制・施設基準届出をカルテに書かない(施設fixtureが持つ)。
4. **実施根拠の述語**: 当日実施は「施行/実施/行った/(検査名)提出」で書く。「〜を確認した」だけを実施根拠にしない。
5. **数値根拠**: 面積・量・日数が点数区分に効くものは数値を書く(「約8×6cm」「10g」「7日分」)。

## 画像属性の正準表

| 属性 | 書き方 | 期待 |
| --- | --- | --- |
| 造影 | 造影への言及なし | 非造影・造影確認なし |
| 造影 | 「造影CTを施行」等、使用明記 | 造影あり |
| 造影 | 「造影剤使用の有無は撮影部門の記録で確認する」等、有無が本文上不確定 | `造影確認` |
| 電子画像管理 | カルテ本文には書かない。fixtureがtrue | 加算あり |
| 電子画像管理 | fixtureがfalse | 加算なし・電子保存確認なし |
| 電子画像管理 | fixtureがunknown | `電子保存確認` |
| CT/MRI機器区分 | カルテ本文には書かない。fixtureが既知 | fixtureの区分で点数確定 |
| CT/MRI機器区分 | fixtureがunknown | `機器区分確認` |

## 推奨(リアリズム軸 — ケースごとに2つ以上)

- 略語: BT/KT, BP, P, SpO2, p.o., Tab, Cap, ED(各科慣用)
- 体言止め・短文: 「咽頭発赤(+)。扁桃白苔なし。」
- 検査結果の生値: 「WBC 9800, CRP 2.1」
- 陰性所見・否定: 「嘔吐なし、髄膜刺激徴候なし」
- 過去・外部情報の混在: 「前医でセフェム処方済み」「健診でHbA1c 6.9指摘」(時制・帰属が分かる書き方で)
- 予定の混在: 「改善なければ次回採血予定」
- 患者の言葉: 「『夜眠れないくらい痒い』」
- 表記揺れ: 半角/全角、カナ薬剤名、「インフル」「溶連菌」等の略称

## 禁止されるリアリズム

- 実在患者由来の文章の転記(文体参考のみ可)
- 根拠の曖昧化が答えを変えるレベルの省略(exactケースでは期待コードの根拠文は必ず特定可能に書く)

## 長さ

S/O/A/P合計の目安(難易度別):

- L1(雑情報少なめ): 400〜600字
- L2(ディストラクタ混在): 550〜900字
- L3(時系列・帰属の混在が濃い): 700〜1100字

v1(平均1064字の説明調)とは異なり、行数は多く1行は短い実カルテの密度で書く。各ケースは distractors(本文に存在するが算定してはならない情報)を難易度に応じて2〜4件含める。

## 文体バリアントプロファイル(第2波)

同一の請求結果・claim context(検証済み)に対し、カルテの書き方だけを変える。`variantOf` で基底ケースを参照。

- `terse`: 体言止め・極端に短い行・略語最多。L1帯。
- `verbose_narrative`: 経過を文章で丁寧に記述。読点が多く行が長い。
- `disordered`: 記載順が前後する(Sに客観情報、Oに患者の言葉が混ざる等)。
- `external_heavy`: 他院・健診・家族由来の情報が多く、当日実施分の特定が相対的に難しい。
- `noisy_results`: 検査生値・バイタル数値が多い。

制約: バリアントでも期待コードの臨床アンカー(実施述語)は必ず本文に含める。ディストラクタはプロファイルに合わせて入れ替えてよい。

## 1000ケース拡張時の作成原則

1000ケース拡張では、まず算定blueprintを固定し、その後にSOAP本文を作る。SOAP本文はアルゴリズムで量産しない。各ケースは、実カルテを想定して1件ずつ作成する。

- `gold-blueprints.json` は算定設計の母集団であり、SOAP本文ではない。coverage matrixから計画生成してよい。
- `requiredClinicalAnchors` は、後続の手書きSOAPで必ず自然な臨床文に変換する。
- `sources/batch-xxx-*.mjs` には、手書きしたSOAP本文を置く。
- caseId、日付、患者属性、診療科だけを差し替えたものは別ケースとして扱わない。
- `caseTypeKey` は、診療科、請求ドメイン、算定状態、施設fixture、文脈罠、不足情報を含む一意の設計キーにする。
- `variantOf` は既存の表現揺れ確認用に残すが、1000ケースの主たる増分には使わない。
- 文章量は既存v2の品質に合わせ、L1/L2/L3の文字数目安を守る。
- exactケースでは、期待コードに対応する臨床アンカーを本文に自然な形で書く。
- review/safety/unsupportedケースでは、確認すべき理由が本文から自然に読めるようにする。
- 施設基準、機器区分、電子画像管理などの施設属性は本文に書かず、fixture側で表現する。
