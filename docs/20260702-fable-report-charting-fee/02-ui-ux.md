# 02. UI/UX

対象: `apps/fee-web`（算定・月次・レセプト・設定）、`apps/charting-web`（録音・逐語・SOAP・管理）。

---

## 総評

直近のリファクタ（月次のポップアップ受診カード化、設定の2ペイン統合、レセプト様式のリアル化、算定候補の「やること」インボックス化）で、**画面ごとの情報設計はSaaS標準に近づいている**。方向性は正しい。

弱点は横断的な一貫性——**ローディング/エラー/空状態の作り込み差、アクセシビリティ（フォーカス管理・ライブリージョン・キーボード操作）の抜け、そして「操作のたびに全体再取得」する応答性**にある。1画面ずつは良くても、アプリ全体としての「速くて予測可能」感が削がれている。

---

## 中-1: 状態変更のたびに全データ再取得（体感の重さ）

**場所**: `apps/fee-web/components/fee-workspace.js:99-118, 137-165`

月次画面で作業ステータス（病名出し/回収など）を1つ切り替えるだけで:

```js
async function updateMonthlyWork(session, patch) {
  ...
  await feeApi(`/v1/fee/sessions/${id}`, { method: "PATCH", ... });
  await loadSummary();   // ← 月次サマリ全体を再取得（= 高コストなフルスキャンAPI）
}
```

`loadSummary` は月次サマリ＋一括候補を並列再取得する。[03](03-performance.md) 高-1 のフルスキャンと合わさり、**トグル1つで全画面がローディングに戻る**。楽観的更新（optimistic update）なら一瞬で反映できる。

**推奨**: PATCH 成功時はローカル state を楽観更新し、失敗時のみロールバック＋再取得。少なくとも「該当患者/受診だけ」部分更新する。

## 中-2: ローディング/空/エラー状態の作り込みが画面でバラつく

- 月次には `MonthlySkeleton`（`fee-workspace.js:539`）があり good。だがセッション詳細・レセプト・設定に同等のスケルトンがあるか不揃い。
- エラーは `toUserFacingErrorMessage` で日本語化され統一されている（良い）が、表示位置・再試行導線がまちまち。
- 空状態（0件時の「まだ算定がありません」的な誘導）が薄い画面がある。

**推奨**: `loading / empty / error / ready` の4状態を**共通コンポーネント**（`packages/web-ui`）に切り出し、全リスト/詳細で統一。空状態には必ず「次の一手」CTAを置く（SaaSの定石）。

## 中-3: アクセシビリティ（医療現場＝キーボード多用/読み上げ要件）

現状の実測（`grep`）:

| 指標 | fee-web | charting-web |
|------|---------|-------------|
| `aria-label` | 47 | 51 |
| `aria-live` | 4 | 4 |
| `aria-modal` | 6 | — |
| `role=` | 38 | 26 |
| `Escape` ハンドラ | 少数 | 少数 |
| focus管理(`focus()`/autoFocus) | 数箇所のみ | 数箇所のみ |

- **モーダル/ポップアップのフォーカストラップが不十分**。`aria-modal` は付くが、Tabがモーダル外へ抜ける・閉じたあとフォーカスが起点に戻らない懸念。医事はマウスを持たず高速入力する職種で、ここは体験に直結。
- **動的更新のライブリージョン**が算定進捗・保存結果に一貫適用されていない（`aria-live` が4箇所のみ）。スクリーンリーダー利用者に算定完了が伝わらない。
- **カラーコントラスト/色依存**: 分類（算定もれ候補/要確認/検討）や状態バッジが色主体。色覚多様性向けにテキスト/アイコン併記の確認を推奨。

**推奨**: 共通モーダルにフォーカストラップ＋Escape＋復帰フォーカスを実装。算定/保存/差分完了は `aria-live="polite"` の共通トースト領域へ。WCAG AA コントラストを設定画面で機械チェック。

## 中-4: charting 録音フローの状態可視化（複雑さの割に説明が薄い場面）

**場所**: `apps/charting-web/components/encounter-workspace.js:247`（`getTranscriptStateCard`）

録音・逐語には状態機械（status × mobileConnectionState × audioSourceType × recordingMode × localRecorderState × isStalled …）があり、状態カードで表現している。設計は真面目だが**状態の掛け算が多く、ユーザには「今なぜこの表示か」が分かりにくい**局面がある（stalled/再接続/モバイル連携）。

**推奨**: 状態を「録音中/一時停止/処理中/完了/要対応」の**5つのユーザ向けメンタルモデル**に丸め、技術状態はツールチップ/詳細に格納。異常系（stalled・接続断）は必ず「何をすればよいか」を1文で提示。

## 低-1: モバイル/レスポンシブ

- fee-web は院内PC前提の密度の高いレイアウト。月次のポップアップ受診カードはレスポンシブ対応済み（CSS確認）。
- charting はモバイル録音（`mobile-join-client.js` 1,268行）があり、そちらは別体験として作り込まれている。PC↔モバイルの用語/アイコンの一貫性を点検すると良い。

## 低-2: 印刷/PDF（レセプト）

レセプト様式は `@media print` でアプリUIを隠す実装があり good。実提出前提なら、印刷プレビューと実PDF出力のピクセル差異（余白・改ページ）を実機プリンタで検証しておくと安心。

---

## SaaS比較で見える差（詳細は [08](08-saas-benchmark.md)）

- **良い**: 月次の「やること」インボックス、受診カードでの点数内訳の紐付けは、Reze等の「生成AIでレセ業務を自動化」系の思想に沿う。差分診断の3分類も MightyChecker/べてらん君の「算定漏れ・査定返戻」観点と重なる。
- **足りない**: これら成熟製品が持つ「一覧の即時フィルタ/ソート、キーボードだけで回せる点検、前月との比較・トレンド、点検理由のワンクリック根拠表示」。特に**キーボード操作性**と**根拠の即時提示**は医事の生産性を分ける。

---

## 対応の優先順位（UI/UX）

1. **中-1**: 楽観的更新の導入（体感速度の即効改善）
2. **中-3**: 共通モーダルのフォーカストラップ＋ライブリージョン（医事の実操作＋a11y）
3. **中-2**: loading/empty/error の共通4状態コンポーネント化
4. **中-4**: charting 録音状態のメンタルモデル簡素化
