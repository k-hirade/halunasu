# 07. コンプライアンス・PHI・運用（追加テーマ）

医療情報（PHI）を扱うプロダクトとして、技術セキュリティ（[01](01-security.md)）とは別に、法令・ガイドライン・運用の観点を立てる。

## 総評

PHIの技術的取り扱いは丁寧（監査ログはPHIを含まない `safePayload` 設計、送信フィールドの絞り込み、暗号鍵の本番強制）。残るのは**「外部AI事業者への提供」の制度的整理**と、**データ保持/削除・運用手順**の明文化。

参照すべき基準: 厚労省「医療情報システムの安全管理に関するガイドライン」、経産省・総務省の対事業者ガイドライン（3省2ガイドライン）、個人情報保護法（要配慮個人情報＝診療情報は原則同意取得・第三者提供制限が厳格）。

## 高-1: 外部AI（OpenAI）への要配慮個人情報の提供整理

**該当データフロー**:
- fee: 患者氏名＋カルテ全文＋診断名 → OpenAI（`openai-fee-clinical-facts.js:373,443`）
- charting: 診療音声PCM＋逐語テキスト＋SOAP → OpenAI（`charting-gateway/src/server.js:518-610`）

診療情報は**要配慮個人情報**であり、第三者提供・委託の扱いが厳格。技術的低減（送信量制限・監査ログ非記録）はあるが、制度面の担保が要る。

**推奨（実装＋運用）**:
1. **氏名の仮名化**: `patientDisplayName` を OpenAI へ送らない（算定・SOAP精度に実名は不要）。送るならセッション内トークンに置換。→ 実装で今すぐ低減できる部分。
2. **委託契約（DPA）とゼロデータ保持**: OpenAI のデータ保持オプト設定・API のトレーニング非利用・保持期間を契約で固定。米国リージョン経由なら越境移転の整理。
3. **患者への周知/同意**: 利用規約・院内掲示・同意書に「AIによる記録支援のため外部処理を行う」旨を反映（医療機関側の運用として要提示）。
4. **国内・医療特化の代替検討**: medimo など医療特化・国内処理を謳うSTTや、オンプレ/VPC内推論の選択肢を、要配慮情報の性質に応じて評価。

## 中-2: 監査証跡の網羅性

**良い点**: fee-api は算定・レビュー・設定変更・差分診断で `createAuditEvent` を広く発火（`server.js` 各所、`safePayload` はPHIなし）。charting gateway も `appendAuditEventSafe` で音声保存等を記録。

**推奨**: 
- 「誰がどのレセを提出可否判断したか」「差分診断の結果に基づく変更」まで証跡が追えるか確認（医事の意思決定の説明責任）。
- 監査ログの改ざん耐性（追記のみ・ハッシュチェーン等）を将来的に検討。診療報酬は自主返還・監査対応で証跡が効く。

## 中-3: STG 環境の分離と本番相当データ

**確認**: STG は Netlify Edge の IPアローリスト（`stg-gate.js`）＋ `isStgEnvironment` ゲートで差分診断等を限定。良い。

**注意点**: 差分診断は「本番相当データ（実レセUKE/CSV）」を STG で扱う運用。**STG に実PHIを置くなら本番と同等の管理**が要る（アクセス制御・保持・削除・監査）。「STGだから緩くていい」という運用にしないこと。

**推奨**: STGでの実データ利用は、匿名化データを原則とし、実データ投入時は期間限定＋利用後削除＋監査を必須化。

## 低: 秘密情報・鍵管理

- gateway は `replace-me` 等のデフォルト秘密を起動時に拒否（`assertRequiredSecret`）、本番 runtime bootstrap 禁止、特権MFA時は `APP_FIELD_ENCRYPTION_KEY` 必須。**設計が堅い**。
- `OPENAI_API_KEY` 等は環境変数。シークレットマネージャ（GCP Secret Manager）経由での注入とローテーション方針を文書化推奨。
