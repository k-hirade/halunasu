## 結論

新規デプロイ後のSTG revision `fee-api-stg-00157-mw2`に対し、患者1001、1006、1013をそれぞれ3回再走した。

評価経路は次のとおりである。

`カルテ本文 -> セッション算定候補 -> 患者 x 月の月次レセプト案 -> 既存UKEとの比較`

全9患者Run、24算定はAPIエラーなく完走し、24算定すべてでOpenAI抽出結果が利用された。確定月次明細は既存UKEの15コードに対して`0/15`一致のままだが、レビュー候補まで含めた検知は`4/15コード（26.7%）`となり、直前revisionの`2/15コード（13.3%）`から改善した。

## 残る主要ギャップ

1. 確定明細としての一致は依然`0/15`で、候補を人が採用するワークフローまで含めた評価が必要である。
2. 評価セッションは前回比較のため`outpatient`固定であり、在宅患者訪問診療料と在宅加算を正しく評価できない。
3. 1001では在宅3コードを候補にもできず、1006・1013でも在宅系コードが未検知である。
4. 確定点数は安定したが、確認事項数と抽出イベント数は不安定である。
5. 今回修正された年齢注加算、特例付き背反・包括、きざみ点数を直接含む入力ではないため、それらのSTG動作確認を本結果だけで完了扱いにはできない。
6. 施設基準キーは0件で、施設基準依存加算の採否は評価していない。
