# 作業依頼: サイドカー自動読み取り版(0.2.0)のフォローアップ (2026-07-22)

対象: 未コミットの自動読み取り実装(受診区分DOM判定・カルテ自動追従・デプロイスクリプトの
カンマ安全化)のレビュー結果。実装本体は承認。以下は検出した要修正1件と改善候補。

## R1. [必須・コミット前] e2eスモークのボタン名不整合

`clients/homis-sidecar/e2e/smoke.test.mjs:59` が旧ラベル
`getByRole("button", { name: "画面から読み取る" })` を参照しているが、ボタンは
「**再読み取り**」に改名済み。e2eは資格情報なし環境ではskipされるため
ユニット15/15緑でも検出されない(リリース前スモークが壊れている状態)。

修正:

1. ラベルを「再読み取り」へ更新。
2. 自動読み取り前提にフローを合わせる: パネル接続後は自動でプレビューが出るため、
   「プレビュー表示を待つ→再読み取りをクリック(冪等確認)→区分がDOM選択済みであることを検証」
   の順に変更する。あわせて**自動読み取りの回帰**(カルテ切替→プレビュー自動更新)を
   e2eに1ステップ追加すると、この機能の実機保証になる。

## R3. [改善・P2] 病名レーンの受診区分ドメインフィルタ(候補ノイズの削減)

実機確認(患者1001・定期訪問)で、追加提案に外来系管理料が並ぶ:
生活習慣病管理料1(660点)・生活習慣病管理料2・特定疾患療養管理料・
プログラム医療機器等指導管理料(90点)。出所は病名レーン
(cc_act_indications: 心不全→特定疾患療養管理料、高血圧→生活習慣病管理料等を裏取り済み)。

適応判定としては正しいが、**訪問診療の患者に外来系管理料の候補はノイズ**
(生活習慣病管理料は外来限定、在宅は在医総管系の世界)。

仕様:

1. `disease_act_candidates`(checks_api.py)に `setting` 由来のドメインフィルタを追加:
   - `home_visit` / `house_call`: **外来系管理料ファミリを抑制**。
     抑制対象は行為の性質で決める(実装前に告示で「外来限定」の管理料を確認し、
     対象ファミリのリストと出典をコードに記載。少なくとも生活習慣病管理料・
     プログラム医療機器等指導管理料は外来限定のはず。特定疾患療養管理料の在宅扱いは
     在医総管包括の関係を確認して決める)。
   - `outpatient`: 現行どおり。
2. 抑制した候補はtraceに `suppressedByEncounterSetting` として記録(消えた理由を追える)。
3. 外来基本料の在宅抑制(既存)と同じ思想である旨をコメントで相互参照。
4. テスト: 同一診断で setting=outpatient は提示 / home_visit は抑制、の対で固定。
5. 効果確認: 1001の追加提案が「在宅療養(曖昧)・在宅データ提出加算」中心に締まること。

## R4. [改善・小] 自動読み取りの連続失敗バックオフ

現在は状態変化ごとにdebounce(180/220ms)のみ。実HOMISで高頻度mutationが起きる画面だと
抽出リトライが連続しうる。連続失敗(または fingerprint不変の連続)時は次回自動読み取りの
間隔を指数的に延ばす(上限数秒)。mockでは問題が出ていないため優先度は低い。

## 判断記録(チケット化しない)

- 「自動化はDOM読み取りまで。算定API呼び出し・保存は明示操作のみ」という境界は
  実装記録に明文化されており妥当。**自動算定への拡張はしない**(安全設計の根幹)。
- 受診区分の自動対応を明示ラベル(定期/往診/外来系)に限定し、電話再診・未知ラベル・
  SOAP本文からの推測をしない設計は G-1 の方針どおり。電話再診はH3で扱う。
