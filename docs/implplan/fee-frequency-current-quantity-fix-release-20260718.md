# 頻度上限・再計算挙動 リリースノート (2026-07-18)

## 変更内容

- 入院など複数日請求の `quantity` を日内回数として扱わず、頻度上限だけを理由に入院基本料を除外しないよう修正した。
- 履歴のない単日請求で当該請求数量だけが上限を超える場合は、自動除外せず要確認として表示する。
- 再計算時に候補の採用・確認状態がリセットされることを、算定画面の実行ボタン付近に表示する。
- `receiptPolicy.blockExportOnErrors` の既定値は `true`。明示設定のない既存組織では、検証エラーを含むUKE出力が従来の許可からブロックへ変わる。

## デプロイ後確認

- STGの `nishiyama-demo-stg` と `yamamoto-demo-stg` で、検証エラーを含むレセプトのUKE出力がブロックされることを確認する。
- 業務上ブロックが不要な組織だけ、`receiptPolicy.blockExportOnErrors=false` を明示設定する。
- 入院基本料を複数日で算定したケースが0点にならないことを確認する。

上記STG確認は、対象サービスのデプロイ後に実施する。
