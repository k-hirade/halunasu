# 設計+作業依頼: 縦断患者コンテキスト(過去カルテ活用)による算定精度向上 (2026-07-22)

## 意図

現行エンジンは「1枚のカルテに書かれた事実」から算定する。実測(yamamoto 13患者)で残る誤差の
主因は、いずれも**1枚のカルテの外にある情報**である:

| 残ギャップ | 必要な情報 | 情報の在処 |
| --- | --- | --- |
| 在医総管/施医総管 4,485〜5,385点/月 | 対象契約・月2回以上訪問・重症度 | 契約状態+月次実績 |
| 機器・管理料(人工呼吸7,480点、酸素系5,271点等) | 機器を使い続けている常態 | **導入時の過去カルテに一度だけ書かれる** |
| 在宅移行早期加算等の起点系 | 退院日・導入日 | 過去カルテ |
| 抽出の揺れ(イベント数最大3倍) | — | 毎回ゼロから抽出していること自体が原因 |

よって「抽出精度の改善」ではなく**情報源の追加**として過去カルテを使う。
過去カルテは時間スケールの異なる3層に分けて扱う(混ぜない):

```
Layer A: 恒常事実    … 一度書かれたきりの事実(造設・機器導入・退院日)。患者プロファイル化
Layer B: 月次実績    … 訪問回数・同月既算定。ローリング集計
Layer C: 直前差分    … 前回カルテとの段落diff。抽出メモ化と変化点フォーカス
```

**Layer AはH4(患者恒常算定プロファイル)の入力問題の答え**である。プロファイルの手入力は
運用負荷で破綻するが、過去カルテの縦断スキャンで自動提案し人は確認だけにする
(施設基準の「算定実績からの逆算」の患者版)。

## 安全原則(全Phase共通の不変条件)

1. **時間汚染の禁止**: 過去カルテの行為を「今回実施」として算定・候補化しない。
   反例コーパスに時間汚染ケースを追加してテストで固定する
   (例:「前月に人工呼吸器を導入」という過去記載 → 今回の処置候補に出ない)。
2. **鮮度の減衰**: 恒常事実は `lastConfirmedAt` から一定期間(既定90日、env可変)で
   「継続確認」の確認事項に落とす。無期限の自動継続はしない(誤継続=過大算定側の防御)。
3. **出所必須**: プロファイル項目・メモ化結果の全てに 出典カルテID(不変レコードID)+日付 を持つ。
4. **HOMIS行為欄は使わない**(評価専用。算定入力へのカンニング禁止は既定方針)。
5. **LLM送信の最小化**: 縦断スキャンは辞書レーン(機械別名)の決定論検出を先行させ、
   LLMは検出行の周辺文脈の確認のみに使う。
6. 候補は常に承認前(candidateOnly)。過去文脈が増えても自動確定はしない。

## Phase 0: 前提整備(2026-07-22外部レビュー反映。Phase 1より先に必須)

レビューでP1指摘3件が「Phase 1の前提が現行コードに無い」ことを突いた。いずれもコードで裏取り済み。

### 0-1. 正規患者キー(縦断履歴の名寄せ)

サイドカードラフトはハッシュ化仮患者キー(`sidecarPatientKey`)、採用後セッションは通常
`patientId`、サイドカー算定時の履歴検索は**ドラフトのみ**(server.jsのドラフトstoreシム)を
見る。このままでは同一患者の「ドラフト+採用済みセッション+外部レセ履歴」を
1本の縦断履歴として取得できない。

- `canonicalPatientId` の解決層を追加する:
  `orgId + sourceSystem + externalPatientId` → 既存患者(externalPatientIds照合)
  → あれば patientId、なければ sidecarPatientKey を暫定キーとし、採用時に patientId へ
  **名寄せ記録付きで**統合する(採用フローの patientMatch を解決層に昇格)。
- 縦断履歴の読み出しAPIはこの解決層を必ず経由する(直接キー参照の禁止をテストで固定)。

### 0-2. 履歴取得の完全性ステータス

`loadPriorFeeSessionsForPatient` は例外時に空配列を返す(catch → [])。Firestore障害・
インデックス不足・権限エラーと「本当に履歴がない」を区別できず、月次回数判定に使うと
**履歴障害時に重複算定候補を出す**。

- 返り値を `{ sessions, completeness: "complete" | "partial" | "unavailable" }` に変更。
- 消費側の規約: `unavailable` のとき、履歴依存の判定(初再診・同月/週回数・メモ化・
  月次回数要件)は**安全側+要確認**に落とす(頻度判定の「確度が足りなければ確定除外しない」
  原則と同じ)。メモ化は不使用。
- 警告文言: 「受診履歴を取得できなかったため、履歴に依存する判定は未確定です」。

### 0-3. スナップショットの保存場所(PHI・読込量)

`calculationResult` 直下に置くと、既存の履歴取得がcalculationResult全体を読むため
Firestore読込量とドキュメントサイズが患者履歴参照のたびに膨らむ。

- 抽出スナップショットは**セッション本体と分離した専用ドキュメント**
  (例: サブコレクション `extractionSnapshots/{feeSessionId}`)に保存し、
  履歴取得(prior sessions)のペイロードに**含めない**。メモ化時のみ直近1件をピンポイント取得。
- ライフサイクル: サイドカードラフトと同じTTL方針(purgeAt)を適用。出典セッション削除時に連動削除。

## Phase 1: 直前カルテ差分と抽出メモ化(Layer C) — Phase 0完了後に着手

### 目的

- Do記載の多い在宅カルテで、**同一段落の再抽出をやめて前回結果を再利用**する。
  抽出の揺れ(積年の課題)とLLMコストを同時に下げる。
- 「新規/継続/消失」の段落タグを下流(候補生成・確認事項)へ渡す基盤を作る。

### 仕様(2026-07-22レビューで改訂)

対象: `services/fee-api/src/clinical-calculation-input.js` の抽出フロー +
`packages/medical-core/src/fee/openai-fee-clinical-facts.js`。

0. **【前提修正】真の行スコープ抽出モードを新設する**。
   既存のline_reviewサブセット再抽出は `preprocessedLines` だけ絞り、
   **`clinicalText` は全文のまま送っている**(clinical-calculation-input.js 1700行付近の
   spread)。このままでは「new行だけ抽出」のつもりでも継続行・消失行のイベントが返りうる。
   - medical-core側に `scope: "line_subset"` モードを追加: 送信する本文を対象行のみで
     再構成し、指示文で「提示行以外に言及しない」を明示。visit_facts / checklist_findings は
     このモードでは**返させない**(スキーマからも外す)。
   - 既存のline_review再抽出もこのモードに乗せ替える(現行の全文送信は同種の弱点を持つため)。
1. **抽出スナップショットの保存**(保存場所はPhase 0-3の専用ドキュメント):
   ```
   {
     promptVersion, extractedAt, canonicalPatientId, sourceRecordId,
     visitFacts: <全文抽出時のvisit_facts>,
     lines: [{ lineKey, has_billable_act,
               events: [行内に evidence が閉じる clinical_events のみ] }]
   }
   ```
   - **`lineKey` = SHA-256(SOAP区分 + 正規化テキスト + 同一テキストの出現順)**。
     行テキスト単独のハッシュでは「継続」等の同語反復や区分違いを誤同一視するため、
     区分と出現順を鍵に含める(行番号には依存しない)。
   - 複数行にまたがるイベントは**スナップショット対象外**(メモ化しない。安全側)。
2. **差分計算**: Phase 0-1 の正規患者キーで直近スナップショット(採用済みセッション・
   ドラフト横断)を1件取得し、現在の前処理行と lineKey で突合して
   `continued` / `new` / (前回側未マッチ=消失) に分類。
   Phase 0-2 の completeness が `unavailable` ならメモ化を使わず全文抽出。
3. **メモ化抽出と統合規約**:
   - `continued` 行: 前回の `has_billable_act` と行内イベントを**行ID再マップして再利用**。
   - `new` 行のみ行スコープ抽出(仕様0)でLLMへ。
   - 全行 new(初回患者・promptVersion不一致・スナップショット無し)→ 従来どおり全文抽出。
   - **フィールド別の統合規約を明文化**(mergeClinicalFactsSamplesはevents/diagnoses/
     line_reviewしか安全に統合できないため):
     - clinical_events / line_review / diagnoses: 和集合(既存規約)
     - visit_facts: メモ利用時は**前回スナップショットの値を引き継ぐ**
       (初再診等は決定論の履歴上書きが後段で優先されるため影響は限定的だが、
       引き継ぎである旨をtraceに記録)
     - checklist_findings: メモ利用時は生成しない(チェックリスト検証はメモ対象外と割り切る。
       必要なら全文抽出に切り替わる条件を設ける)
     - 【実装レビュー2026-07-22追記】visit_facts引き継ぎの既知の限界: line_subsetスキーマは
       visit_factsを返さないため、**新規行に処方箋発行の記載が初めて現れた場合**、
       snapshot由来の古い visit_facts(院外処方=unknown等)が使われる。
       STG計測で院内薬剤32点が確定明細へ残る実害を確認したため、新規/消失行に
       院内/院外処方・処方箋・一般名処方・リフィルの語彙を含む場合は全文抽出へ
       フォールバックする条件を追加した。スナップショットには本文を重複保存せず、
       行ごとの機微フラグだけを保持する。
   - line_review完全照合(既存契約)は統合後に従来どおり実行する(メモ化はこの契約の内側)。
4. **ゲート**: env `FEE_EXTRACTION_MEMO`(既定off)。promptVersion不一致・
   スナップショット構造version不一致で自動失効(v14→v15移行で自動的に全文抽出へ)。
5. **観測**: metrics に `memoHitLineRatio` / `newLineCount` / `removedLineCount` を追加し、
   trace に stage `extraction_memo` を記録。評価スクリプトにも記録を追加。
6. **消失タグ**: Phase 1 では trace 記録のみ(警告は出さない)。Phase 3 で
   恒常事実の継続確認に接続する。

### テスト・受け入れ

- ユニット: 同一行再訪でLLM呼び出しが new 行のみに絞られる(抽出器スパイ) /
  行の順序変更・行番号ズレでもハッシュ一致で再利用される /
  promptVersion不一致で全文抽出に戻る / 複数行イベントは再抽出される。
- 反例(時間汚染): 前回カルテにのみあった処置行が今回のイベントへ混入しない。
- 受け入れ: mockの縦断データ(同一患者の連続カルテ)で、2回目以降の算定の
  `memoHitLineRatio` が高く、**同一カルテ再実行の抽出結果が完全一致**(揺れゼロ)になること。
  gold 2系統・全スイート不変。

#### STG等価性計測の判定規約

`npm run eval:fee-longitudinal-context` で、メモ経路と同一入力の全文抽出対照を同一Cloud Run
revision上で比較する。対照は新規の合成患者へ同じ対象本文を投入して3回実行し、まず対照経路
自身のLLM揺れを測る。

- 確定明細: コード・名称・数量・点数が安定した全文抽出対照の全実行と完全一致すること。
- 候補集合: pending明細と追加提案を意味キーで集合比較する。差分は全件保存し、既知の
  `visit_facts` 制約に起因する処方関連差分だけを `pass_with_known_limit` として許容する。
- 対照経路自身が揺れた場合: メモの不具合とは断定せず
  `inconclusive_control_variability` とする。対照間の差分集合もレポートへ残す。
- **メモ不使用シナリオの帰属(2026-07-22計測で追加)**: メモ経路が実際にはメモを使わなかった
  シナリオ(all_lines_new等、フォールバックが期待動作のもの)の等価性差分は、
  全文抽出同士の比較でありメモ起因と断定できない。`inconclusive_llm_variability` として
  原因を帰属する。ただし意味等価性の合格へ読み替えず、acceptance(フォールバック動作)と
  意味等価性を別々に報告する。
  実例: 20260722_172416 の all_lines_new は評価経路の全文抽出1回だけが
  clinicalEventCount=0(outputTokens=404)となった単発のLLM取りこぼしで、
  メモの履歴汚染ではない(対照3回は毎回3イベント)。
- 完全一致再計算: 前回スナップショットを読む別セッション同一本文と、同じ対象セッションの
  再計算を分け、双方で `memoHitLineRatio=1`、新規/消失行0、OpenAI呼出し0を確認する。
- 一部変更: P段落だけを変更し、継続3・新規1・消失1を確認する。
- `visit_facts` 既知限界: P段落へ「院外処方箋を発行」を初めて追加し、引き継がれた
  `visit_facts` と全文抽出対照の差、処方箋料系の確定明細・候補への実害を記録する。
- 行為消失: 前回だけの創傷処置行が現在の明細・候補に無いことに加え、
  `removedLineCount` と `extraction_memo` traceへ記録されることを確認する。
- 履歴取得不能: STGへ意図的な障害を起こさない。fail-closedはユニットテストで担保し、
  STG計測では `completeness=unavailable` が発生していないことだけを受け入れ条件にする。

制御データはlineKey一致率を人工的に設定するため、ここで得られるLLM削減率は機構の上限値で
あり実運用値ではない。顧客カルテのcopy-forward率(Do記載率)は実データで再計測するまで不明と
明記し、削減率を過大に訴求しない。

## Phase 2: 月次実績の完成(Layer B) — **新規実装ではなく既存機能の拡張**

(2026-07-22レビュー反映)既存レセ履歴の取込(`fee_billing_history`)・同月/同日/同週コード集計・
既存レセ一括取込ルートは**実装済み**である。Phase 2はそれらへの次の拡張として扱う:

1. **受診区分と実訪問回数**: 履歴レコードに定期訪問/往診/外来の別と回数を持たせ、
   在医総管の回数要件(月2回以上/月1回)判定に使える集計を追加(Phase 4の入力)。
2. **日付精度**: 月単位しかない取込元(レセ由来)と日単位(カルテ由来)の精度差を
   レコードに明示し、日精度が必要な判定(同日・同週)には月精度履歴を使わない。
3. **取込元と履歴完全性**: 取込元(UKE/CSV/カルテ/手動)と対象期間をレコード化し、
   Phase 0-2 の completeness 判定(この期間はcomplete等)に接続する。
4. **冪等な再取込**: 同一期間の再取込が重複レコードを作らない(取込元+期間+患者の一意キー)。
5. **外部患者ID→正規患者IDの対応**: Phase 0-1 の解決層を取込にも適用する。
6. 取込データはLLM抽出を通さない「履歴専用レコード」のまま保つ(時間汚染の構造的防止)。
7. 受け入れ: 履歴が埋まった患者で初再診・同月制限判定が導入初月から正しく働き、
   completeness が期間単位で正しく報告されること。

## Phase 3: 恒常事実スキャン(Layer A) = H4プロファイルの自動提案

1. **事実スキーマ**(新コレクション `patientStandingFacts` 相当。2026-07-22レビューで拡張。
   設計レビューで最終確定):
   ```
   { orgId, facilityId, canonicalPatientId,            // スコープ(必須)
     factType: tracheostomy | gastrostomy | home_ventilator | home_oxygen |
               self_injection | opioid_management | discharge_date | zaiisokan_eligible_hint | ...,
     status: proposed | confirmed | rejected | superseded | ended,   // 却下・再提案・競合を表現
     effectiveFrom, effectiveTo,                        // 有効期間
     firstSeen: {recordId, date},
     lastObservedAt,                                    // カルテ上の最終観測(機械)
     lastHumanConfirmedAt, confirmedByMemberId,         // 人の確認(観測と分離)
     endedEvidence?,
     sourceQuotes: [{recordId, date, excerpt}],         // 件数上限5・各120字上限
     revisions: [{at, byMemberId, change}]              // 訂正履歴
   }
   ```
   - RBAC: 確認・訂正は fee admin / org_admin に限定。全変更を監査イベントに記録。
   - 出典連動: 出典レコード(セッション/ドラフト)削除時は該当 sourceQuote を落とし、
     引用が0件になった fact は `proposed` へ降格して再確認を要求する。
   - **鮮度はfactType別**(一律90日は不適切):

     | factType | 減衰ポリシー |
     | --- | --- |
     | discharge_date 等の一回性事実 | 減衰しない(日付事実) |
     | home_ventilator / tracheostomy / gastrostomy | 長期(例180日)で継続確認。中止語検出で即時要確認 |
     | home_oxygen / self_injection | 中期(例90日)で継続確認 |
     | opioid_management | 短期(例60日)で継続確認 |

     値はfactTypeテーブルとしてコード化し、根拠(見直し理由)をコメントに残す。
2. **縦断スキャナ**: 保存済みカルテ本文(セッション/ドラフト/取込)を対象に、
   辞書レーンの機械別名+機器語彙で候補行を決定論検出 → LLMで
   開始/継続/中止の別だけ確認 → `proposed` として登録。
3. **確認フロー**: 「過去カルテから検出した常態: 人工呼吸器使用中(初出2025-11-04)」を
   人が確認して `confirmed` へ。確認UIの置き場所(fee-web設定/承認コンソール様式)は
   設計時に決定(サイドカー本体には置かない。閲覧UI不要方針と整合させる)。
4. **候補生成への接続**: confirmed の常態から、該当する在宅指導管理料・機器加算を
   **月次候補**として決定論生成(点数はマスタ解決。曖昧区分はcodeCandidates)。
   鮮度減衰・消失タグ(Phase 1)で継続確認を出す。
5. 受け入れ: 1002(人工呼吸)・1007(在宅酸素)・1004(気管切開)で、過去カルテスキャンから
   該当常態が提案され、確認後に管理料+機器加算群が候補に出てUKEの該当コードを検知できること。

## Phase 4: 月次アセンブラ(在医総管ファミリ)

- 入力: Layer A(対象契約・重症度示唆) × Layer B(月2回以上) × 施設設定(施設種別・施設基準) ×
  同一建物人数(受診DOM実績)。
- 出力: 在医総管/施医総管の区分マトリクスを機械解決した**月次候補**
  (一意に決まらない軸はcodeCandidates+「区分選択が必要」)。付随加算
  (在宅データ提出・包括的支援・在宅医療情報連携・緩和ケア充実等)は親候補の子として提示。
- 受け入れ: yamamoto 13患者の在医総管/施医総管系コードが候補込み検知でUKEと突合できること
  (確定は人の承認。ここまで到達すると残ギャップは電話再診(H3)とコメント領域のみ)。

## データ入手経路(前提)

- 導入時: 過去カルテ一括取込(Phase 2)。運用中: 毎回の算定で自然蓄積(ドラフト/セッションに
  本文保存済み)。補助: 拡張の「前のカルテ」をユーザーがめくって読み取り追加(自動クロールはしない)。

## 評価計画(2026-07-22レビュー反映: 指標を2段に分離)

- **UKEは正解データではなく比較対象**。UKE一致は「既存請求の再現率」であり、
  医学的・算定要件上の正しさを保証しない(既存側の誤請求を再現しても一致は上がる)。
- 指標を分離して報告する:
  1. **UKE再現率**(候補込み検知/確定一致): 導入価値・移行安全性の指標
  2. **gold正解率**: 医療事務が確認した正解ラベル(seed-300/v2系および今後の在宅gold拡充)
     に対する精度。制度上の正しさの指標
- 追加計測: 抽出安定性(memoHitLineRatio・イベント数分散)、時間汚染ゼロ(反例コーパス)、
  Phase 0-2 completeness の分布(unavailableの発生率)。
- 行為欄の自動突合ハーネス(受診単位の比較ラベル)は評価専用として別途整備してよい
  (これもUKE同様「比較対象」であり正解ではない)。

## 実装状況 (2026-07-22)

### 完了: Phase 0

- 正規患者キー解決を追加した。HOMISの `sourceSystem + facilityId + externalPatientId` を
  Platform患者の識別子へ完全一致させ、採用前のサイドカー患者キーと採用後の `patientId` を
  aliasesで横断する。既存患者の高速検索用に `patientIdentifierKeys` を追加し、旧データには
  既存検索インデックスを使った完全一致フォールバックを残した。
- 通常Feeセッション、サイドカードラフト、外部レセ履歴を正規患者キーでまとめて取得する。
  患者名寄せまたは履歴クエリが取得不能なら `completeness=unavailable` とし、履歴を空として
  正常扱いしない。初再診等の履歴依存判定を確定入力から外し、規定の警告を返す。
- 抽出結果はセッション本体ではなく `fee_extraction_snapshots` に分離した。
  `purgeAt` TTL、患者+診療日の複合インデックス、出典単位の削除メソッドを追加した。
  現在Feeセッションの削除APIは存在しないため、現行ライフサイクルでは独立TTLで削除し、
  将来の出典削除処理は追加済み削除メソッドを必ず呼ぶ。
