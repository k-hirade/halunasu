# mock_partner カルテから月次レセプト比較 STG再走

## 結論

患者1001、1006、1013の合成カルテを使い、2026-07-13にSTGで次の経路を患者ごとに3回再走した。

`カルテ本文 -> セッション算定候補 -> 患者×月の月次レセプト案 -> 既存UKEとの比較`

全9患者RunがAPIエラーなく完走した。一方、既存UKEの15コード・25算定回・8,752点に対して、一致した既存コードはすべてのRunで0件だった。患者1001の月次結果は3回とも371点で安定したが、患者1006と1013は外来管理加算52点の有無により月次結果が変動した。

この結果は、確定コード付き`orders.csv`を入力した[日付補正後の再算定試験](../20260713_comparison_mockpartner_data_dateshift_stg/README.md)の15/15一致とは評価対象が異なる。今回の試験では`orders.csv`、UKEコード、期待値を算定処理へ渡していない。したがって、決定論エンジンへ正しいコードを渡した場合の再現性は確認済みだが、カルテから必要な算定条件を抽出・構造化して月次明細へ昇格させる経路には大きな不足が残る。

## 優先して直すべき点

### P0

1. セッション契約へ`home_visit / house_call`、訪問日、同一建物区分を追加し、外来として誤処理しない。
2. 在宅診療では外来管理加算を候補化しない排他条件を決定論化する。
3. 施設基準を有効期間付きで登録し、在宅データ提出加算・ベースアップ評価料等へ接続する。
4. 傷病手当金意見書・療養費同意書の作成交付イベントを診療行為候補へ接続する。

### P1

1. がん性疼痛緩和指導管理料などのレビュー候補を患者×月で重複排除する。
2. 薬剤総合評価調整管理料について、変更前後の薬剤数、対象薬、同月履歴を構造化する。
3. 同一カルテから確認事項の件数と種類が大きく変わらないよう、重複排除と決定論リカバリを強化する。
4. 月次集計・差分診断APIのレイテンシを継続計測し、Firestore read件数とCloud Run処理時間を分離する。
