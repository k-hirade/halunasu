# 診療報酬算定: 制度・算定ナレッジ層 実装計画

作成日: 2026-06-20

## 目的

特定疾患療養管理料のように、カルテ本文の単語抽出だけでは拾いにくい算定候補を扱う。

対象は「LLMに探させる項目」ではなく、病名、主病、処方、同月履歴、施設条件、診療日、診療報酬マスターをまたぐ制度判定である。そのため、LLM/NLPの外側に制度・算定ナレッジ層を置き、根拠付きの算定候補・増点候補・要確認・除外を生成する。

## 現状

既存実装には、特定疾患療養管理料と特定疾患処方管理加算の最小候補生成がある。

- `clinical-calculation-input.js` 内で、対象疾患、管理・指導記録、長期処方、同月履歴を見て候補を作る。
- 特定疾患療養管理料は候補として出し、自動点数算入はしない。
- 同月履歴から「当月2回目」「前回日」を出すテストがある。

ただし、現状は制度ナレッジ層としては不十分である。

- 対象疾患判定がほぼ `喘息/気管支喘息` の正規表現だけ。
- B000/特定疾患処方管理加算のロジックが巨大な入力変換ファイルに直書きされている。
- 傷病名マスターの特定疾患等対象区分や、医科診療行為マスターの傷病名関連区分を利用していない。
- B001、C100、I004 などの管理・指導・在宅系シグナルは体系化されていない。
- レセコン・会計上の算定済み履歴は、現状は同一アプリ内の過去セッション履歴が中心である。

## 追加する層

名称: 制度・算定ナレッジ層

入力:

- 検証済み臨床事実 `canonicalClinicalFacts`
- 病名、主病、疑い、転帰、開始日
- 処方、処方日数、院内/院外
- 診療日、請求月、入外区分、施設区分
- 同月算定履歴、同日算定履歴
- 医科診療行為マスター
- 傷病名マスターまたは当面の対象疾患データ
- 診療報酬改定バージョン

出力:

```json
{
  "proposalId": "specific_disease_management_xxx",
  "ruleId": "B000_specific_disease_management",
  "status": "candidate_high",
  "title": "特定疾患療養管理料の確認",
  "displayName": "特定疾患療養管理料（診療所）",
  "clinicalLabel": "慢性疾患として当院で継続管理中の可能性",
  "candidateLine": {
    "code": "113001810",
    "name": "特定疾患療養管理料（診療所）",
    "points": 225
  },
  "evidence": "療養計画に基づき吸入手技・増悪時対応を説明",
  "monthlyLimit": {
    "maxPerMonth": 2,
    "priorCount": 1,
    "currentOrdinal": 2,
    "previousDates": ["2026-06-14"],
    "status": "within_limit_exactly"
  },
  "requiredInput": "対象疾患、主病管理、療養計画・指導記録、同月算定履歴",
  "warnings": ["同月2回まで。今回は当月2回目です。"]
}
```

## 設計原則

1. LLMにコード、点数、最終算定可否を決めさせない。
2. カルテ本文の特定フレーズだけに反応しない。
3. 病名、主病、処方、履歴、施設、マスターを使う。
4. 対象疾患データはコード直書きにしない。
5. 自動算定ではなく、まずは候補・要確認として出す。
6. 候補には必ず根拠、未確認条件、同月制限、除外理由を持たせる。
7. 同月上限に近い候補は UI に簡潔な注意を出す。
8. 実装は軽量なルール評価にし、追加の LLM 呼び出しは行わない。

## P0 実装範囲

### P0-1. B000/特定疾患処方管理加算ロジックを切り出す

`clinical-calculation-input.js` から以下を独立モジュールへ移す。

- 特定疾患対象病名判定
- 特定疾患療養管理料の候補生成
- 特定疾患処方管理加算の候補生成
- 同月2回制限の履歴評価
- 候補行 fallback

想定ファイル:

- `services/fee-api/src/clinical-billing-knowledge.js`
- `services/fee-api/src/clinical-billing-knowledge-data/specific-disease-targets.json`

### P0-2. 対象疾患データを外部化する

当面は公式マスター完全連携の前段として、対象疾患データを JSON 管理する。

初期データは現在の挙動を維持するための喘息・気管支喘息から開始し、今回の実装で令和8の特定疾患療養管理料に関係する主要な慢性疾患ファミリーへ拡張した。完全な傷病名マスター連携は今後の課題だが、コード直書きではなく JSON データとして管理する。

各レコードは次を持つ。

- `targetId`
- `label`
- `matchTerms`
- `billingFamily`
- `effectiveFrom`
- `effectiveTo`
- `source`

### P0-3. 候補生成ステータスを明示する

候補生成結果には、制度ナレッジ層由来であることと判定状態を明示する。

- `candidate_high`: 対象疾患、管理・指導記録、履歴条件が揃っている。
- `needs_review`: 同月上限到達など、候補としては見せるが注意が必要。
- `excluded`: 疑い病名、既往のみ、過去/他院文脈など。

P0では既存UI互換のため、`candidateProposals` 形式に変換して返す。対象疾患だけがあり、当日の管理・指導記録や長期処方根拠がない場合は、過剰な候補表示を避けるため新規候補を出さない。target-only の `needs_review` は、P1 以降で管理・指導系ファミリーを広げる際に、UIノイズと反例を評価してから追加する。

### P0-4. テストを追加・維持する

追加/維持するテスト:

- 活動性の対象疾患 + 管理記録あり -> 特定疾患療養管理料候補。
- 既往歴だけ -> 候補にしない。
- 対象疾患以外 -> 候補にしない。
- 同月1回済み -> 当月2回目として候補。
- 同月2回済み -> 上限到達として強い確認文を出す。
- 長期処方あり -> 特定疾患処方管理加算候補。

## P1 実装範囲

### P1-1. 対象疾患マスターの拡張

公式告示、傷病名マスターの特定疾患等対象区分、医科診療行為マスターの傷病名関連区分を使い、対象疾患を増やす。

P1では、すべて自動算定にせず、候補・要確認として出す。

### P1-2. 管理・指導系ファミリーを増やす

優先候補:

- 生活習慣病管理料
- てんかん指導料
- 難病外来指導管理料
- 皮膚科特定疾患指導管理料
- 特定薬剤治療管理料
- 在宅自己注射指導管理料
- 在宅酸素療法指導管理料
- 在宅持続陽圧呼吸療法指導管理料
- 心身医学療法

### P1-3. 医事会計・算定済み履歴の入口を作る

同一アプリ内の過去セッションだけでなく、将来のレセコン・会計連携を受けられる contract を作る。

例:

- `billingHistory.claimedItems`
- `billingHistory.sameMonthCounts`
- `billingHistory.externalClaimSource`

## 反例を避ける条件

以下の場合は候補にしない、または要確認に落とす。

- 疑い病名のみ。
- 既往歴、家族歴、過去治療のみ。
- 他院管理の記載。
- 初診直後や退院直後など制度上の除外条件がある。
- 同月上限に到達済み。
- 入院中など外来前提に合わない。
- 併算定不可の医学管理料が同月にある可能性がある。
- 管理・指導の当日記録がない。

## パフォーマンス方針

制度・算定ナレッジ層は追加 LLM を呼ばない。対象疾患データ、同月履歴、マスター候補をメモリ上で評価する。

重くしないために、1算定内で以下をキャッシュする。

- マスター検索結果
- 対象疾患データ
- 同月履歴の集計結果

## 今回の実装結果

今回の修正では P0 を完了し、P1 の一部も実装した。

### 実装済み

1. docs に本計画を追加した。
2. B000/特定疾患処方管理加算ロジックを `clinical-billing-knowledge.js` へ切り出した。
3. 対象疾患を `clinical-billing-knowledge-data/specific-disease-targets.json` に外部化した。
4. 管理・指導・在宅・心身医学系の候補ルールを `clinical-billing-knowledge-data/management-signal-rules.json` に外部化した。
5. 特定疾患療養管理料、特定疾患処方管理加算、在宅持続陽圧呼吸療法指導管理料、心身医学療法などを review-only の増点候補として出せるようにした。
6. 同月2回制限について、前回日、当月何回目か、上限到達かを候補に保持するようにした。
7. 候補には `source`、`ruleId`、`knowledge`、`monthlyLimit` を保持し、監査・テスト・UI詳細で根拠を追えるようにした。
8. 既存の算定変換では、制度ナレッジ候補を自動点数には入れず、`candidateProposals` として表示する方針を維持した。

### 反例対策として入れた制約

- 管理・指導・在宅・心身医学系の候補は、原則として対応する病名がある場合だけ出す。
- 過去、他院、予定、否定、未実施の文脈は候補化しない。
- 管理・指導の当日根拠がない場合、特定疾患療養管理料は候補化しない。
- 特定疾患処方管理加算は、対象疾患に加えて長期処方または日数根拠がある場合だけ候補化する。
- すべて review-only とし、LLMやNLPだけで自動算定に移さない。

### 追加した代表テスト

- 既往歴だけの喘息では特定疾患療養管理料を出さない。
- 活動性の気管支喘息 + 管理記録で特定疾患療養管理料と特定疾患処方管理加算を候補化する。
- マスター検索が外れても fallback の候補行を保持する。
- 同月1回済みの場合、本日を当月2回目として表示する。
- 虚血性心疾患など、公式対象疾患ファミリーの候補化を確認する。
- CPAP、心身医学療法などの管理シグナルを、対応病名つきで候補化する。

### 残課題

- 傷病名マスターの特定疾患等対象区分との直接連携。
- 医科診療行為マスターの傷病名関連区分との直接連携。
- 外部レセコン・会計データからの算定済み履歴取り込み。
- B001/C100/I004 系をさらに広げる場合の、施設基準、算定済み履歴、併算定不可の精密化。
