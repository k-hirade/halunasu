# 改革1〜3 STG再計測（mock_partner 5患者）

## 結論

患者1001、1004、1006、1007、1012をSTGで各3回再計測した。評価経路は次のとおりである。

`カルテ本文 -> セッション算定 -> 患者 x 月の月次レセプト案 -> 既存UKE比較`

5患者・15患者Run・45算定はAPIエラーなく完走し、45算定すべてでOpenAI抽出と`fee-clinical-events-v14`が使われた。

改革1の主目的である確定点数の安定性は、前回の`3/5患者`から`5/5患者`へ改善した。特に1004と1012で発生していた外来管理加算由来の52点単位の揺れは再発しなかった。

一方、既存UKE 32コードに対する確定一致は各反復`0/32`、候補を含む検知は各反復`2/32（6.25%）`である。抽出イベント数が3回同一だった受診は`1/15`、確認事項集合が安定した患者は`0/5`であり、改革3は確定点数への波及を抑えたものの、抽出自体の安定化には至っていない。

## 改革ごとの判断

1. **改革1（確定ゼロ揺れ）**: 今回の5患者では成功。確定点数は全患者で3回一致した。
2. **改革2（検索基盤）**: 1007の在宅酸素で曖昧コードを安全な1グループにできた。候補全体の安定性は未達である。
3. **改革3（抽出契約v14）**: v14は全45算定で利用されたが、イベント数が安定した受診は1/15、確認事項集合が安定した患者は0/5だった。確定点数への揺れ伝播は止めたが、抽出の揺れ自体は残っている。

## 制約

- 全セッションを`outpatient`として実行しており、在宅中心の既存UKEと入力契約が一致していない。
- 既知受診歴と施設基準キーを投入していない。
- レビュー候補は採用していないため、最終レセプト精度ではない。
- `candidateResultStable`は評価スクリプト上、確定月次明細の署名を見ている。候補安定性の判断には候補点数と`candidateLines`を使用した。
- `gcloud`のローカル再認証が切れていたためrevision名は取得できなかった。実行前の`/readyz`、全45算定のprompt version、API応答で対象環境を確認した。

測定手順は[methodology.md](./methodology.md)、匿名化済みraw結果は各患者の`raw/result.json`を参照する。
