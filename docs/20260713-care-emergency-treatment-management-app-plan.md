# 緊急時治療管理・介護報酬点検アプリ構想

作成日: 2026-07-13  
更新日: 2026-07-16
状態: 保留（現行単位の再調査・構想まで。実装未着手）

現行単位、旧通知との不一致、医科500点との制度差、コード実装状況の監査結果は、[緊急時治療管理の現行単位・実装状況調査](./20260716-emergency-treatment-management-current-fee-audit.md)を参照する。

## 1. 結論

西山病院から提示された「緊急時治療管理 518単位」は、2026年7月16日時点の現行介護報酬と一致する。これは現在のハルナスが扱う医科診療報酬とは別制度であるため、現在の医科算定画面へ算定候補として混在させず、利用者からは別の業務アプリに見える「介護報酬点検」として設計する。

ただし初期段階から別のCloud Runサービスへ分割する必要はない。認証、組織、患者識別、監査ログ、UI基盤は共通利用し、介護報酬のマスタ、判定、単位計算、月次請求は独立した境界に置く。

初期ゴールは介護請求全体の自動化ではない。緊急時治療管理について、次を安全に支援する。

1. 急変記録の構造化
2. 算定要件、不足情報、月内重複、連続日数、併算定制限の点検
3. 算定日に有効な介護報酬マスタの単位数と対象日数による候補提示（現行マスタは518単位/日）
4. 医師または担当者による確認と月次確定
5. 西山病院の院内確認票に相当する印刷・出力

AIやルールが記録にない事実を補完して自動算定する機能は作らない。

## 2. 制度上の前提

### 2.1 対象制度

介護老人保健施設では「緊急時施設療養費」、介護医療院では「緊急時施設診療費」の中に「緊急時治療管理」が置かれている。短期入所療養介護等にも同種の規定がある。

2026年7月16日時点の主な要件は次のとおりである。

- 518単位/日
- 同一入所者について月1回
- 1回の急変につき連続3日まで
- 救命救急医療が必要な重篤状態であること
- 投薬、検査、注射、処置等を実際に行うこと
- 特定治療、所定疾患施設療養費等との併算定制限があること
- 短期入所では総合医学管理加算との同日算定制限も確認すること

518単位は2019年10月1日に511単位から改定され、令和8年6月1日版の算定構造でも維持されている。一方、厚生労働省サイト上の旧留意事項には500単位という記載が残っている。単位数は算定日現在の告示・算定構造・サービスコード表を優先し、留意事項は対象状態や併算定制限の解釈に用いる。

対象となる重篤状態は、通知上、次の6区分である。

1. 意識障害又は昏睡
2. 急性呼吸不全又は慢性呼吸不全の急性増悪
3. 急性心不全（心筋梗塞を含む）
4. ショック
5. 重篤な代謝障害
6. その他薬物中毒等で重篤なもの

参考:

- [厚生労働省・介護老人保健施設等の介護報酬基準](https://www.mhlw.go.jp/web/t_doc?dataId=82aa0255&dataType=0)
- [厚生労働省・算定に関する解釈通知](https://www.mhlw.go.jp/web/t_doc?dataId=00ta4383&dataType=1&pageNo=2)
- [厚生労働省・短期入所療養介護の報酬基準](https://www.mhlw.go.jp/web/t_doc?dataId=82aa0253&dataType=0)
- [厚生労働省・介護報酬の単位換算](https://www.mhlw.go.jp/web/t_doc?dataId=82ab4582&dataType=0&pageNo=1)
- [厚生労働省・令和8年6月版 介護報酬の算定構造](https://www.mhlw.go.jp/content/001675193.pdf)
- [WAM NET・令和8年6月1日適用 介護サービスコード資料](https://www.wam.go.jp/gyoseiShiryou/detail?ct=020050010&gno=22560)

### 2.2 医科の類似項目との分離

医科診療報酬には、別制度の「緊急時施設治療管理料 500点」が存在する。現在の医科マスタにもコード`180709110`として収載されているが、今回の介護報酬518単位とは対象、請求制度、算定要件が異なる。医科500点は、所定の転換老健に併設される保険医療機関の医師が夜間又は休日に緊急往診した場合の項目で、1日1回、月4回までである。

そのため、名称の類似だけで介護記録を医科コード`180709110`へ解決してはならない。入力時点で次を必須の判定軸とする。

- `billingScheme`: `medical_insurance` / `long_term_care_insurance`
- `careServiceType`: 老健、介護医療院、短期入所療養介護等
- `facilityId`と算定日に有効な施設設定

参考: [厚生労働省・医科診療報酬点数表](https://www.mhlw.go.jp/web/t_doc?dataId=84aa9729&dataType=0)

### 2.3 金額換算

介護報酬は医科のような一律の`点数 × 10円`ではない。地域区分とサービス別の1単位単価を使用する。そのため介護の518単位を、現在の医科点数合計へ混ぜてはならない。

初期版では単位数までを確定し、金額表示は地域区分・サービス別単価が有効期間付きで設定された場合だけ行う。

### 2.4 現在の実装状況

現行コードには、介護報酬の緊急時治療管理を算定する機能はまだない。

できていること:

- 医科マスタに`180709110 緊急時施設治療管理料 500点`が収載されている
- 認証、組織、患者、監査、月次画面等の共通基盤がある
- 本文書と現行単位監査Docで要件・境界を整理している

できていないこと:

- 医療保険と介護保険を分けるデータモデル
- effective-datedな介護サービスコード・単位マスタ
- 6状態、実施行為、月1回、連続3日、併算定制限の判定
- 介護向けUI、API、Firestore永続化、月次請求
- 西山病院の確認票取込・印刷
- 医科`180709110`の専用適格性チェック

したがって、現時点のアプリで介護518単位を候補化してはならない。

## 3. 西山病院の確認票に対する評価

提示された確認票には、6つの重篤状態、急変時病名、処置、検査、処方、急変状況が含まれており、構造化入力の基礎として利用できる。西山病院グループには介護医療院、介護老人保健施設、短期入所があるため、確認票だけで対象サービスコードを一意に決めることはできない。利用施設・事業所・サービス種別を別途確認する必要がある。

機械判定と監査証跡のため、次の項目を追加する必要がある。

- 施設・サービス種別と保険制度
- 急変発生日時、治療日、開始日、終了日
- 救命救急医療が必要と判断した根拠
- 入院が必要だったか、施設内治療を行った理由
- 処置・検査・注射・投薬の実施時刻、内容、結果
- 連続日数と同一エピソードの識別
- 当月の既算定有無
- 同日の特定治療、所定疾患施設療養費等
- 判断した医師、確認者、確認日時
- 搬送依頼、搬送結果、転帰

確認票にあるケアマネジャーや主治医への報告は業務上有用だが、確認できた公的資料では算定のハード要件としては確認できなかった。請求可否を止める必須条件ではなく、施設内ワークフロー項目として管理する。

## 4. プロダクト境界

### 4.1 利用者に見せる構成

```text
ハルナス
├── 医科診療報酬算定
└── 介護報酬点検
    ├── 緊急時治療管理
    ├── 月次確定
    ├── 取込履歴
    └── 施設設定
```

初期URLは`fee.halunasu.com/care`のような同一アプリ配下でもよい。医科の患者セッション画面や算定候補へは混在させない。

### 4.2 内部構成

初期は既存のデプロイ単位を増やさず、コード境界を分ける。

```text
apps/fee-web/app/care-fee/
packages/care-fee-core/
services/fee-api/src/care-fee/
```

永続化も医科セッションと分離する。

```text
careFeeEpisodes
careFeeMonthlyClaims
careFeeFacilitySettings
careFeeAuditLogs
```

介護報酬全体、外部連携、介護給付費請求まで拡張する段階で、`care-fee-web`と`care-fee-api`への物理分割を再検討する。

## 5. 画面設計

### 5.1 第一画面: 月次点検一覧

この項目は月1回、連続3日、同月・同日併算定を確認するため、患者入力画面ではなく請求月単位の一覧を第一画面にする。

```text
緊急時治療管理             [2026年7月 ▼] [老健本館 ▼] [＋ 急変記録]

すべて 12　要確認 4　算定候補 3　確定 5　対象外 0

入所者       急変日時       重篤状態       対象日数   判定       不足情報       単位
山田 花子    7/10 14:20     急性呼吸不全   2日       要確認     医師確認       1,036
佐藤 一郎    7/12 02:10     ショック       1日       算定候補   なし             518
鈴木 次郎    7/15 09:00     発熱           0日       対象外     重篤条件なし         0
```

表示方針:

- フィルターは請求月、施設、サービス種別、判定状態
- 増収額ではなく算定可否、不足情報、重複、併算定制限を中心に表示
- 行を選ぶと詳細画面を開く
- 一括操作は印刷・CSV出力までとし、一括算定確定は行わない
- 状態は`要確認`、`算定候補`、`確定`、`対象外`に限定する

画面の518は表示例であり、実装では固定値にせず、サービス種別と算定日に有効なマスタ値を表示する。

### 5.2 急変記録・判定画面

デスクトップでは、左側に根拠記録、右側に判定結果を固定表示する。

```text
┌──────────────────────────────┬─────────────────────┐
│ 急変記録・実施内容            │ 算定判定             │
│                              │                     │
│ サービス種別                  │ 要確認               │
│ [介護老人保健施設 ▼]          │ 医師確認が必要です    │
│                              │                     │
│ 急変日時 [7/10] [14:20]       │ 算定見込             │
│                              │ 現行518単位 × 2日    │
│ 重篤状態                      │ 合計 1,036単位       │
│ ☑ 急性呼吸不全               │                     │
│ ☐ 急性心不全                 │ 条件確認             │
│ ☐ ショック                   │ ✓ 対象状態           │
│                              │ ✓ 処置実施           │
│ 急変時病名                    │ △ 医師確認           │
│ [誤嚥性肺炎による呼吸不全]    │ ✓ 月内初回           │
│                              │ ✓ 連続3日以内         │
│ 実施処置                      │                     │
│ ☑ 酸素投与 [3L/分]            │ 併算定確認           │
│ ☑ 末梢点滴 [開始時刻]         │ 所定疾患療養費 なし   │
│ ☑ 採血     [結果]             │ 特定治療 なし         │
└──────────────────────────────┴─────────────────────┘

[対象外にする]                 [下書き保存] [算定候補として確定]
```

重篤状態は複数選択可能なチェックボックス、二値条件はチェックまたはトグル、サービス種別は選択メニュー、日時・数量は専用入力を使う。自由記載だけに依存しない。

### 5.3 日別管理

```text
日付       実施内容             判定       単位
7月10日   酸素、点滴、採血     対象        518
7月11日   酸素、点滴           対象        518
7月12日   経過観察のみ         要確認        0
7月13日   処置継続             上限超過      0
```

同一エピソードか、連続日か、3日上限内かを日付から決定論的に判定する。非連続の3日を一つの対象期間として扱わない。

### 5.4 施設設定

施設側が管理する項目:

- 施設種別
- 提供している介護サービス
- 地域区分
- 介護保険事業所番号
- 設定の適用開始日・終了日
- 算定候補を確定できる職種・権限

法令上の単位、対象状態、上限、排他ルールは施設ユーザーに編集させず、改定対応マスタとして管理する。

## 6. 入力と判定

### 6.1 入力経路

初期版では次の3経路を想定する。

1. 紙様式を見ながら手入力
2. カルテ記載の貼り付けと項目抽出
3. CSVによる複数件取込

AI抽出値には出典と抽出状態を保持する。ただし重篤性、救命救急医療の必要性、施設内治療の妥当性をAIだけで確定しない。不足情報は具体的に提示し、実施していない処置や存在しない判断を生成しない。

### 6.2 判定結果

算定候補にするための基本ゲート:

1. 対象となる介護サービスである
2. 6つの重篤状態のいずれかが記録されている
3. 救命救急医療が必要であることを確認できる
4. 投薬、検査、注射、処置等の実施事実がある
5. 同一入所者の当月初回である
6. 同一急変の連続3日以内である
7. 同日・同期間の併算定不可項目がない
8. 必要な医師確認が完了している

不足時は自動的に0単位へ落とすだけでなく、`要確認`として不足項目を明示する。要件を満たしても初期版は自動確定せず、算定候補に留める。

## 7. 実装フェーズ

### P0: 制度混同の防止

- 医療保険と介護保険をデータモデルで分離する
- 算定日・サービス種別付きのeffective-datedな介護サービスコードマスタを追加する
- 介護の緊急時治療管理を医科コード`180709110`へ自動解決しない
- 医科500点を使用する場合も専用の適格性チェックを通す
- 医科点数と介護単位を同じ合計へ足さない

### P1: 緊急時治療管理の点検支援

- 急変エピソード入力
- 6状態、救命救急判断、実施内容の確認
- 月1回、連続3日、併算定制限の決定論点検
- 有効なマスタ単位×対象日数の候補表示（現行は518単位/日）
- 月次一覧、確定、監査履歴
- 西山病院の院内確認票に相当する印刷

### P2: 介護報酬請求への拡張

- 改定資料からの介護サービスコードマスタ取込・世代管理
- 地域区分・サービス別の単価計算
- 利用者負担を含む金額計算
- 入所者×月の介護報酬集計
- 介護給付費請求データの作成・検証

## 8. 受入テスト例

| ケース | 期待結果 |
| --- | --- |
| 急性呼吸不全、酸素・点滴を2日連続実施 | 要件確認後、1,036単位の候補 |
| 軽度発熱、水分補給のみ、救命救急判断なし | 対象外 |
| 同一急変で4日連続実施 | 最初の3日まで候補、4日目は上限超過 |
| 同月に別の非連続急変が発生 | 2回目は月1回制限として要確認または対象外 |
| 所定疾患施設療養費を同日算定 | 併算定競合を表示して確定不可 |
| 記録に酸素投与の記載がない | 実施したと推測せず、不足情報として表示 |
| 医療保険セッションで介護518単位を検索 | 候補化しない |
| 医科500点の専用条件を満たす | 介護518単位とは別の医科候補として判定 |

## 9. 保留理由と再開条件

現時点では、対象施設・請求業務・データ連携範囲が確定していないため実装を保留する。再開前に西山病院へ次を確認する。

1. 確認票を利用している施設は老健、介護医療院、短期入所のどれか
2. 現在どの介護請求システムで518単位を請求しているか。また、画面に表示されるサービスコード名称とコードは何か
3. 紙様式の誰が、いつ、どの情報を記入・承認するか
4. 急変記録、処置、検査、投薬、介護請求実績をどの形式で取得できるか
5. 必要なのは院内確認票の作成支援か、介護請求データ作成までか
6. 地域区分、事業所番号、サービス種別を施設単位で取得できるか
7. 既存システム上で月内重複や併算定不可をどこまで確認できるか

上記が確定した時点で、P0とP1を実装対象として再見積もりする。
