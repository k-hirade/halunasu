# mock_partner カルテから月次レセプト比較 STG再走

## 結論

新規デプロイ後のSTG revision `fee-api-stg-00154-788`に対し、患者1001、1006、1013をそれぞれ3回再走した。Cloud Runのトラフィックは実行後の確認時点で同revisionに100%向いていた。

評価経路は次のとおりである。

`カルテ本文 -> セッション算定候補 -> 患者×月の月次レセプト案 -> 既存UKEとの比較`

全9患者RunはAPIエラーなく完走し、全算定でOpenAI抽出が使われた。一方、既存UKEの15コード・25算定回・8,752点に対し、**自動採用されて月次明細へ入ったコードの一致は、各反復とも0/15コード**だった。新revisionでも、在宅系コードを外来コードとして処理する主要な差分は解消していない。

改善が確認できた点もある。患者1013では、前回は検知できなかった次の2明細が、3反復すべてでレビュー候補に出た。

- `180000710` 傷病手当金意見書交付料
- `113004310` 療養費同意書交付料

ただし、これらは安全側のレビュー候補であり自動採用されていない。そのため、現在の「採用済み月次明細だけ」を比較する評価では一致に数えられない。**候補検知の改善と、確定月次明細の一致率は分けて評価する必要がある。**

## コード差分の要点

既存UKE側で未再現の中心は、3患者に共通する次の在宅系コードである。

- `114001110` 在宅患者訪問診療料（1）1
- `114057970` 在宅データ提出加算
- `180725910` 外来・在宅ベースアップ評価料（1）3（訪問診療時）イ

当社側は代わりに、外来として次のコードを月次明細へ入れた。

- `111000110` 初診料
- `112007410` 再診料
- `180819910` 物価対応料1（初診時）イ
- `180820010` 物価対応料1（再診時等）ロ
- `112011010` 外来管理加算（1006と1013、一部反復）

したがって、今回の0一致は単純なマスター不足だけではなく、診療区分を`outpatient`に固定した評価契約と、カルテから在宅診療条件を月次明細へ接続する経路の不足による。

## 判断

1. 新revisionでも、確定月次明細の既存UKE再現率は改善していない。
2. 文書交付イベントからレビュー候補を作る経路は改善した。
3. レビュー候補を自動採用して一致率を上げるべきではない。実施事実と算定要件を確認したうえで採用する現在の安全原則を維持する。
4. 次の評価では、`確定明細一致`に加えて`レビュー候補まで含む検知率`を別指標にする。
5. 在宅診療の精度を評価するには、評価スクリプトとセッション契約で`home_visit / house_call`を表現し、外来固定を解消する必要がある。
