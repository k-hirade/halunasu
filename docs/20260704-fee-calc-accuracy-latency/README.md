# カルテ→診療報酬算定 精度・レイテンシ改善 研究レポート

作成日: 2026-07-04
対象: halunasu の「カルテ本文 → 臨床イベント抽出(LLM) → 決定論算定(Python) → 点検」パイプライン。
目的: **より高精度・より低レイテンシ**な算定を系統的に設計する。

---

## 0. エグゼクティブサマリ

現行は「LLMは点数を出さず抽出のみ、算定はPython決定論」という正しい骨格。改善余地は**構造（並列化・キャッシュ）と検証（点検ゲート・較正）**に集中している。

**レイテンシ Top3（効果順）**
1. **準備段の逐次実行**：DBの前処理5段（施設/設定/患者履歴/外部請求/オーダ強化）を**LLM抽出の前に直列**で回している。LLM（長い処理）と独立DB処理を**並列化**すれば体感が最も変わる。
2. **プロンプトキャッシュの取りこぼし**：巨大な静的instructionsは自動キャッシュ対象だが、低頻度運用だと5–10分で失効。**静的先頭の徹底＋ウォームアップ**で TTFT を大きく短縮できる（一般に最大80%）。
3. **Python算定の接続/スキーマ毎回再作成**：`api.py` が算定ごとに SQLite 接続＋`initialize_schema`。読み取り専用で使い回せる。

**精度 Top3（効果順）**
1. **決定論点検を"算定の検証ゲート"に**：今回作った適応/禁忌/併用/算定もれ点検を、算定候補に対して回し、**過大算定を needs_review へ**。精度＝安全性に直結。
2. **コードリンキングの多段化**：現状の名称ベース master_search に、ベクトル近傍＋リランカ＋構造検算を足して**誤コードを削減**。
3. **抽出の確信度較正 → 3値昇格**：LLM抽出の確信度が低いイベントを自動採用させず needs_review へ（過信防止）。

**設計の芯**：**「速いパス」と「確実なパス」を、ノート複雑度・金額インパクトで動的に選ぶ**。低インパクトは軽量・即返し、高インパクト（高額/査定頻発/DPC資源病名）は検証・二重抽出に倒す。

> **⚠️ 実測で判明した訂正（[04](04-eval-findings-20260704.md)）**
> - **レイテンシは総時間の83–90%がLLMコール**（master/py/ruleは合計<0.5秒）。→ 「準備段の並列化(L1)」は効果極小。**唯一のレバーはLLM出力の短縮**（`max_output_tokens`未設定・イベントschema過大・char offsets等）＋並列抽出＋モデル階層化。
> - **精度の"ズレ"の一部は正解データ(gold)の陳腐化**：`180820010 物価対応料`(2026改定・2点)をengineが正しく付与し、goldが未更新。+2系はこれ。→ 深追いしない。
> - **本物の漏れ**：CTが`past_or_external`に誤分類され欠落(Δ-1018)、院内処方の抽出漏れ(Δ-61)。→ **時制/source誤分類の是正**と**画像/院内処方の想起強化＋決定論補完**が要改善。

## 優先ロードマップ（精度×レイテンシ）

### フェーズ1（すぐ効く・低リスク）
- **L1 準備段の並列化**（独立DB前処理をPromise.allに＋LLM抽出と重ねる）
- **L3 Python接続/スキーマ再利用**
- **A1 決定論点検の算定ゲート化**（今ある資産の接続）
- **測定の可視化**（stageTimings/TTFT/キャッシュ率のダッシュボード）

### フェーズ2（構造改善）
- **L2 プロンプトキャッシュ最大化＋ウォームアップ**
- **L4 shadow計算をhot pathから外す**
- **A4 抽出確信度の較正→needs_review昇格**
- **A3 版固定（廃止コード/旧点数の排除ガード）**

### フェーズ3（精度の底上げ）
- **A2 コードリンキング多段化（ベクトル＋リランカ）**
- **L5/L6 split条件化・モデル/effort動的選択**
- **A5 非対称評価（SafetyRatio）＋CIゲート**
- **速い/確実パスの動的ルーティング**

> 各施策の詳細・効果・コスト・実装箇所は各詳細ドキュメントを参照。
