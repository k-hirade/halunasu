## 1. サンプルカルテ・算定結果の収集設計

病院から「個人情報を削除したサンプルカルテ＋診療報酬算定結果」を集め、算定システムの精度検証に使う。
依頼テンプレート実体: `tmp/halunasu_sample_chart_and_billing_request_nishiyama.txt`

### 1.1 収集単位は「患者 × 1か月」
- レセプトは元々「患者 × 1医療機関 × 1暦月」が1単位。これに合わせるのが最も自然。
- この単位なら、1セット内で以下が検証可能:
  - 月合計点数・一部負担金（会計）
  - 同月制限・併算定不可
  - 管理料の月1回・検査の同月重複
  - 月次点検（`buildMonthlyClaimSummary` は `patientId + claimMonth` で名寄せ）
- 「受診1回バラバラ」だと同月2回目以降の制限・併算定が検証できない。

### 1.2 過去分（縦覧）の要否は「ケースタイプ」で決まる
- 急性・単発 → 過去分不要（1受診でよい）。
- 慢性・医学管理料・定期検査 → 同一患者の連続 3か月（理想6か月）。
  - 根拠: 算定回数制限の月窓が `2月/3月/4月/6月/12月`（電子点数表）。初再診継続は1〜3か月。
  - 実装上 `loadPriorFeeSessionsForPatient` は直近10件を参照。
- 5年窓（がん検査等）は収集非現実的のため対象外。

### 1.3 個人情報・匿名化
- 記載しない: 氏名・生年月日・住所・電話・患者ID・保険者番号・記号番号。
- 患者属性は「年代＋性別」のみ。患者は「患者A」等の符牒。
- 施設名・医療機関コード・医師名も記載しない。
- 保険は「負担割合の区分（3割/後期高齢1割 等）」のみ。番号は不要。
- 利用目的＝精度検証のみ、第三者提供なし、検証後削除（テンプレに明記済み）。
- 連続データ（同一患者の複数月）は再識別リスクが上がるため匿名化を一段厳格に。

### 1.4 記入負担の見積もりと許容性
- 1患者×月あたり: 電子カルテ／レセコンから貼り付け可なら 10〜20分、手入力なら 30〜45分。
- 5患者 ≈ 1〜1.5時間、20患者 ≈ 4〜7時間（月末月初多忙の医療事務には重い）。
- 対策（テンプレ反映済み）:
  - まず 3〜5患者で十分。
  - 「そのまま貼り付けOK・整形不要」を最上部に明記。氏名等だけ消す。
  - きれいに埋めなくてよい・空欄可。
  - 急性（5分）＋慢性（20分）を混在させると幅と負担のバランスが良い。

### 1.5 テンプレートの構成（最終形）
- 冒頭: 趣旨 / ご負担を抑えるために / 個人情報の取り扱い / 記入の単位
- 基本情報: 患者符牒・診療区分・請求月・診療科・患者属性・保険区分・届出済み施設基準
- 受診ごと: S/O/A/P・既往歴/生活歴/家族歴/アレルギー/内服歴・初診/再診・時間区分・処方(院内/院外)・検査/処置/画像
- 月の算定結果: 合計点数＋算定明細（レセコン実値）
- カルテにあるがレセに出ない情報（症状詳記の元）
- 査定・返戻（あり/なし）
- 記入例: 60代男性・高血圧/2型糖尿病/脂質異常症・同月2回受診・医学管理料・査定あり例（病名漏れA査定→病名追記して再審査）

## 2. カルテ作成アプリ(charting)の出力構造と実カルテとのギャップ

### 2.1 現状の charting 出力（コード事実）
`packages/medical-core/src/soap/format-profile.js`:
- `SOAP_SECTION_KEYS = ["subjective","objective","assessment","plan"]`
- `DEFAULT_SOAP_FORMAT_PROFILE` は S/O/A/P の4区分テキストのみ。
- 既往歴・アレルギー・内服歴・病名・処方・検査は「独立した構造化フィールドとしては出さない」（O/Pの文章中に混ざる）。

### 2.2 実カルテの構造
- SOAP記録 ＋ 患者サマリ(既往/アレルギー/内服/プロブレムリスト) ＋ 病名(別管理) ＋ オーダー(処方/検査/処置=構造化) を別々に管理。
- サンプル収集テンプレで既往/アレルギー等を別立てにしたのは、実カルテ構造に沿っている（charting出力とは別物でよい。テンプレはfee検証用のため）。

### 2.3 実カルテ出力に近づけるための改修（優先度順）
- P1: 病名候補(レセ病名)の構造化出力（A評価から。疑い/確定を区別）。fee連携・算定に直結。
- P1: オーダー(処方/検査/処置)の構造化出力（用量・薬剤名はLLM創作禁止、候補提示）。
- P2: 付帯情報セクションの器を追加（主訴/現病歴の区別、既往/アレルギー/内服/家族歴/生活歴）。中身は会話で言及があったものだけ（幻覚防止、なければ空）。
- P3: バイタルの構造化（体温/血圧/脈拍/SpO2/体重）。
- 実装の形: `format-profile.js` の sections 拡張 or 「SOAP本文（医師が読む下書き）」と「カルテメタ（病名候補・オーダー候補・付帯情報の構造化）」を分離するのが安全。
- 原則: chartingは「医師が確認・転記する下書き」。器は実カルテ準拠に増やすが、中身は会話根拠のあるものだけ。やりすぎ（それっぽく埋める）は医療安全リスク。

### 2.4 連携上の含意
- fee が charting の出力(S/O/A/P文章)を入力にする場合、病名・処方・既往・アレルギーは SOAP本文からLLM抽出する前提（構造化で渡ってこない）。
- 「charting→fee 自動連携」を本気でやるなら、charting側で病名・オーダーも構造化して持つ拡張が効く。

## 3. SOAPからの診断(病名)生成 — 精度センシティブ領域

誤ると査定・誤記に直結する順:

1. 疑い vs 確定の区別（「〜を疑う」を確定病名にしない。確定/疑い/除外ステータス保持）
2. 否定・除外の検出（「肺炎を示唆する所見は乏しい」→ 肺炎を病名化しない）
3. 既往 vs 現在の病名（時制。治癒済み・既往を今回病名にしない）
4. 他者の病名の混入（家族歴「父 高血圧」等を患者本人病名にしない）
5. 部位・左右（site/laterality。診療内容の部位と整合。`claim-risk-knowledge.js` に検知あり: body_laterality / claim_risk_body_site_mismatch）

レセ病名としての質:

6. 病名標準化 / ICD-10・病名マスタへのマッピング（表記ゆれ・略語 DM=糖尿病 等の正規化、造語禁止）
7. 修飾語（急性/慢性・部位・病期）の取り違え
8. 病名↔診療内容の整合（病名漏れ／根拠なし病名の両方）

取りこぼし・重複:

9. 仮定・予定・条件文の除外（「肺炎なら抗菌薬」を確定病名にしない）
10. 重複・同義の統合（高血圧症/本態性高血圧 等）、主病名/副病名の区別
11. 検査値からの自動診断は慎重に（数値→病名の飛躍は幻覚源。医師記載のAを根拠に）

実装の勘所:
- これらの多くは fee の決定論NLP前処理（否定・時制・他者・メタ文の `clinicalLineCues`）と重複。診断生成でも同じガードを通す。
- 「Aセクションを主たる根拠」にする（S/Oの単語だけで病名化しない）。
- 出力は 病名＋ステータス(確定/疑い/除外)＋部位/左右＋修飾語＋根拠引用(SOAP内) を構造化。根拠なしは出さない。
- 最優先対策は 1〜5（疑い/否定/時制/主体/部位）。

## 4. 令和8改定をふまえた増収機会の4観点フレーム

増収候補を提示する際のタグ体系（Phase 7 改善ダッシュボードの分類軸候補）。
※令和8(2026)改定は +3.09%、柱は「物価・賃上げ対応」「医療DX」。

- ① ほとんどの病院が気にかけていないが簡単につけられるも（自動取得・取りこぼし防止）
  - 物価対応料（届出不要・基本料連動・全機関自動）／乳幼児加算／必須コメント付与漏れ／同月・併算定の取りこぼし。決定論で自動付与・外部AI不要。アプリの一丁目一番地。
- ② 令和8改定で新しく使えるようになったもの
  - 外来・在宅物価対応料、電子的診療情報連携体制整備加算、外来データ提出/管理加算、機能強化加算の要件変更。effective-datedマスタで改定月に自動提示。届出要件があるものは「提案＋要件チェック」。
- ③ 少しの設備投資で投資対効果が出るもの
  - 医療DX系体制加算（電子的診療情報連携、オン資格確認、電子処方箋）、電子画像管理加算。Phase 8（施設基準）＋Phase 9（改定影響）で「届出すれば月◯点／回収◯か月」を提示。増収を煽る表示は要件充足＋コンプラ文言必須。
- ④ 大きめの投資で投資対効果が出るもの
  - 入院基本料区分・看護必要度・DPC等。外来中心の現製品では費用対効果を出しにくく、当面review-only縮退。外部ベンチが無いと投資対効果を出しにくい（doc通り後回し）。

戦略: 主戦場は ①②（決定論で自動付与/即提示・PII安全・即効）。③は提案型で差別化。④は線引き。

## 5. 留意（コンプラ・医療安全）

- 「逸失収益／改善余地点数」の表示は過剰請求を誘発しうる。必ず「実施事実があり要件を満たす候補のみ・自動算定しない」を明記。
- 医師別/患者別の横断集計は機微（RBAC・利用目的限定）。
- AI生成（診断・SOAP・詳記）は下書きであり、確定は人。根拠なき記述は出さない。
- 外部AI（OpenAI）に診療データを送る現実と「国内リージョン/学習禁止」要件の整合は製品横断で要決着。
