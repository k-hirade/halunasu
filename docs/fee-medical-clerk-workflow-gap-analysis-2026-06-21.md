# 診療報酬算定アプリ 医療事務ヒアリング反映方針

作成日: 2026-06-21

## 目的

医療事務へのヒアリング結果を、現行の `fee-web` / `fee-api` / `fee-core` / Python 算定エンジンの実装に照らして整理し、診療報酬算定アプリとして次に修正・追加すべき実装方針を決める。

この文書は、単なる機能要望一覧ではなく、実務フロー上のボトルネックから優先順位を決めるための判断材料とする。

Phase 1〜6 の後続方針は、診療報酬算定コンサルの公開サービス調査を反映した `docs/fee-reimbursement-consulting-benchmark-2026-06-23.md` に整理する。

## ヒアリングから見えた業務課題

受付業務では、受付・患者案内、料金計算、会計が中心であり、時間がかかるのは最初の患者案内と料金計算だった。

月末月初のレセプト業務は、以下の流れで進む。

```text
レセ出力
  ↓
レセ点検
  ↓
医師へ病名出し
  ↓
病名回収
  ↓
データ作成
  ↓
請求
```

患者 1 人あたりの算定から提出までの時間は、通院回数、検査、投薬、処方、修正有無によって大きく変わる。つまり、平均時間を短くするよりも、時間がかかる患者・レセプトを早く特定し、確認作業を減らすことが重要である。

迷いやすい算定・査定返戻の課題として、以下が挙がった。

- 過去に C 査定を受けたもの
- 査定理由が分かりづらい
- A 査定、B 査定
- 疾患部位と画像部位の相違
- 右左など左右の相違
- 検査や投薬が算定要件を満たしているか
- レセプトからは読み取れないが、カルテには実施妥当性が記載されているケース
- 病名漏れ
- 保険適応外の治療
- 支払基金からの照会や詳記依頼

既存導入システムとしてはレセ点検システムがあり、人手不足を背景に導入された。追加で欲しい機能は、査定項目のありがちな理由が分かり、返戻査定対策に使えることだった。

生成 AI については、個人情報の取り扱いに敏感であり、病院内での利用は限定的である。したがって、AI を前面に出すよりも、根拠を残し、人間が確認できる形で支援することが重要である。

## 現行実装でできていること

現行アプリは、1 患者・1 診療日単位の算定候補作成には一定対応している。

- `fee-web` で患者、カルテ本文、病名、手入力オーダーを入力できる。
- `fee-api` で fee session を作成し、非同期算定ジョブを実行できる。
- `detail-lite` を使って算定中ステータスをポーリングできる。
- `fee-core` が算定結果から `receiptDraft`、`reviewItems`、`candidateWorkbench` を生成する。
- 画面上で算定候補、要確認、増点提案、レセプト案を確認できる。
- レビュー項目を「算定する」「算定しない」として採否できる。
- CSV 出力とレセプト電算 UKE 出力の下書きがある。
- 病名未入力時に査定リスク warning を出せる。
- `monthly-summary` API で患者 x 請求月の集計を作れる。
- 同月・同日履歴を calculation options に反映する土台がある。

ただし、主導線は「1 件の算定記録を開いて候補を作る」体験であり、ヒアリングで重要だった月次レセ点検、病名出し、病名回収、査定理由理解にはまだ十分に接続されていない。

## 修正方針

### 1. 月末月初のレセ点検ワークスペースを作る

現状の `monthly-summary` API は土台として存在するが、`fee-web` の主導線では使われていない。

修正方針:

- `fee-web` に請求月単位のレセ点検ダッシュボードを追加する。
- 患者別に、受診回数、合計点数、未算定、要確認、病名不足、詳記候補、出力可否を一覧化する。
- 問題がない患者は確認済みにし、問題がある患者だけ詳細画面に進む導線にする。
- 月末月初の作業では、個別レコード作成画面よりもこの月次画面を入口にする。

狙い:

- 平均時間ではなく、時間がかかる患者・レセプトを先に見つける。
- 医療事務が「今月どこまで終わっているか」を把握できる状態を作る。

### 2. 病名出し・病名回収ワークフローを作る

現状は病名入力欄と病名未入力 warning があるが、医師への確認依頼と回収管理がない。

修正方針:

- `fee_session` に病名確認ステータスを追加する。
- 病名候補、確認依頼理由、依頼先医師、依頼日時、回収日時、回収結果を保持する。
- 月次ダッシュボードに「病名出し待ち」「医師確認中」「回収済み」を表示する。
- 医師向けには、患者名、診療日、カルテ根拠、候補病名、必要理由だけを絞った確認画面または確認キューを用意する。

狙い:

- 病名漏れを単なる warning で終わらせず、月末月初の実務タスクとして管理する。
- 医師への確認依頼を散発的な口頭・チャット依頼から、状態管理できる業務フローに移す。

### 3. 査定・返戻理由の知識ベースを作る

現状の warning は算定エンジン由来の説明が中心で、過去査定や「ありがちな理由」を体系化していない。

修正方針:

- 査定・返戻・再審査の知識ベースを追加する。
- A 査定、B 査定、C 査定、返戻、照会、詳記依頼を区別する。
- `procedureCode`、診療行為名、傷病名、部位、左右、保険適応、必要コメント、過去査定理由を紐づける。
- `candidateWorkbench` の要確認項目に、査定されやすい理由と確認すべき根拠を表示する。
- 医療機関ごとに過去査定メモを登録できるようにする。

狙い:

- 「なぜ確認が必要か」が分かる状態にする。
- 返戻査定対策に使える説明を、算定候補と同じ画面で見せる。

### 4. 部位・左右・適応病名チェックを強化する

ヒアリングでは、疾患部位と画像部位、右左の相違が具体的に挙がった。

修正方針:

- カルテ本文、病名、画像オーダーから部位・左右・目的を抽出する。
- 画像系、処置系、検査系の候補に対して、病名部位・オーダー部位・カルテ根拠を照合する。
- 不一致または不明の場合は `reviewIssue` として出す。
- 右左、上下、部位、疾患名、画像部位を構造化して保存し、監査可能にする。

狙い:

- 単に「画像候補」と出すのではなく、査定されやすい部位不一致を事前に検知する。
- カルテには妥当性が書かれているがレセプトからは読み取れないケースを拾う。

### 5. 詳記・コメント作成を業務フローに接続する

現状の UKE 下書きには CO コメントや SJ 症状詳記の構造があるが、画面で作成・確認する導線が弱い。

修正方針:

- 要確認項目から、必要コメント・詳記候補を作成できるようにする。
- カルテ根拠、検査値、投薬理由、画像実施理由を引用して、医療事務が確認できる詳記下書きを作る。
- 詳記が必要なレセプトを月次ダッシュボードで一覧化する。
- UKE 出力時に、確定済みコメント・詳記だけを含める。

狙い:

- 支払基金からの照会や詳記依頼に備える。
- レセから読み取れない妥当性を、請求前に補える状態にする。

### 6. レセ電出力は実請求品質の検証を通す

現状の UKE 出力は構造実装として有用だが、コード上でも近似を含むことが明記されている。実請求に使うには仕様検証が必要である。

修正方針:

- 当面は「レセ電下書き」として扱い、実請求可能と表現しない。
- レセプト電算処理システムの記録条件仕様に対するフィールド単位の検証テストを追加する。
- 医療機関情報、保険、公費、本人家族区分、高齢区分、診療識別、負担区分、コメント、症状詳記を優先して検証する。
- 実レセコンに取り込める形式は、接続先レセコンごとの CSV / UKE / API 仕様を確認してから対応する。

狙い:

- 下書き支援と請求データ作成を混同しない。
- 実運用で危険な「提出できるように見えるが仕様未検証」の状態を避ける。

## カルテ一括算定の判断

カルテ一括算定は必要である。ただし、目的は全自動請求ではない。

実装すべき形は、請求月単位の一括候補化とリスク順レビューである。

```text
請求月を選択
  ↓
対象患者・受診分を一覧化
  ↓
未算定または再計算が必要なものを一括候補化
  ↓
病名漏れ、部位左右不一致、適応疑義、詳記必要、過去査定類似で優先順位付け
  ↓
問題なしだけ提出候補へ進める
  ↓
要確認は医師確認キューまたは詳細画面へ送る
```

やってはいけない形:

- カルテを大量投入して全件自動確定する。
- 根拠や要確認を隠して点数だけ出す。
- AI が最終判断したように見せる。
- 仕様未検証の UKE を実請求データとして扱う。

## 優先順位

### Phase 1: 月次レセ点検の入口を作る

最優先で、月次レセ点検ダッシュボードを実装する。

実装範囲:

- `GET /v1/fee/monthly-summary` のレスポンスを拡張する。
- `fee-web` に `/monthly` または `/sessions?view=monthly` を追加する。
- 請求月、患者、受診回数、合計点数、ステータス、要確認件数、病名不足、詳記候補、出力可否を表示する。
- 患者行から詳細の fee session に遷移できるようにする。

この段階では、一括算定はまだ実行しない。まず月次全体の見える化を完成させる。

### Phase 2: 病名出し・回収キュー

実装範囲:

- fee session に病名確認ステータスを追加する。
- 病名候補と確認理由を保存する。
- 月次画面で病名確認タスクを一覧化する。
- 医師確認用の軽量画面またはフィルタを用意する。

### Phase 3: 査定理由ナレッジと部位左右チェック

実装範囲:

- 査定返戻ナレッジのデータ構造を作る。
- review issue に査定理由カテゴリを追加する。
- 画像・処置・病名の部位左右チェックを追加する。
- UI で「なぜ査定リスクか」「何を確認すべきか」を表示する。

### Phase 4: 詳記・コメント作成

実装範囲:

- コメント・詳記下書きを fee session に保存する。
- 要確認項目からコメント・詳記を作る導線を追加する。
- 月次画面で詳記未対応を確認できるようにする。
- UKE 出力に確定済みコメント・詳記を接続する。

### Phase 5: 一括候補化

実装範囲:

- 請求月内の未算定・再計算対象を抽出する。
- 一括ジョブを作成する。
- 進捗、失敗、再実行、キャンセルを管理する。
- リスクなしの候補だけを一括確認済みにできるようにする。

一括候補化は Phase 1 から必要に見えるが、月次ダッシュボード、病名キュー、査定理由カテゴリがない状態で実装すると、単に未確認の候補を大量生成するだけになる。そのため Phase 5 に置く。

### Phase 6: レセ電実請求品質検証

実装範囲:

- UKE フィールド単位の仕様テストを追加する。
- レセコン取込 CSV の接続先仕様を確認する。
- 出力前バリデーションを実装する。
- 実請求可能な状態と下書き状態を UI 上で明確に分ける。

## 実装しない、または優先度を下げるもの

### 患者案内機能

受付の患者案内は時間がかかる課題だが、診療報酬算定アプリの中核とは少し離れる。現時点では、算定アプリ内に患者案内ワークフローを大きく作らない。

ただし、会計見込み、保険確認、未入力項目チェックなど、料金計算に直結するものは将来的に別タブとして扱う余地がある。

### AI 前面訴求

生成 AI は、個人情報の取り扱いへの懸念が強い。機能名や UI では、AI が最終判断するように見せない。

採用する表現:

- 候補作成
- 要確認
- 根拠確認
- 下書き
- 医師確認

避ける表現:

- 自動請求
- AI が確定
- 完全自動算定
- そのまま提出

## 決定事項

1. 次の実装の主軸は、個別算定画面の追加改善ではなく、月次レセ点検ワークスペースに置く。
2. カルテ一括算定は必要だが、最初に作らない。月次ダッシュボード、病名確認、査定理由カテゴリを先に作る。
3. 病名漏れは warning ではなく、医師確認タスクとして扱う。
4. 査定・返戻対策は、算定エンジンの点数計算だけでなく、理由知識ベースとして実装する。
5. 部位左右・適応病名チェックを、レセ点検機能の中核リスクとして扱う。
6. UKE 出力は当面下書き扱いとし、仕様検証が完了するまで実請求可能とは扱わない。
7. AI は補助に留め、根拠引用、人間の確認、監査可能性を必須にする。

## 次に作る実装計画

次の実装計画では、Phase 1 の月次レセ点検ダッシュボードを対象にする。

最初の実装単位:

1. `monthly-summary` のレスポンスに、患者別の `needsReviewCount`、`missingDiagnosisCount`、`symptomDetailCandidateCount`、`readyForClaimCount`、`blockedCount` を追加する。
2. `fee-web` に月次レセ点検画面を追加する。
3. 算定一覧から月次レセ点検画面へ移動できる導線を追加する。
4. 月次画面から患者別 session 詳細へ遷移できるようにする。
5. まずは読み取り専用で実装し、一括算定・一括確定は入れない。

この順序なら、既存の `fee_session` と `monthly-summary` を活かしながら、ヒアリングで明らかになった月末月初の業務課題に最短で接続できる。
