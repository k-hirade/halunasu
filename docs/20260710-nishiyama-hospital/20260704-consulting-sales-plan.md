## 0. 戦略（このライン）

1. **まず診療報酬コンサルで売上改善提案を行い、バリューを出す**（無料/低額の一括診断）。
2. バリューを示したうえで「**このバリューを最大化するにはカルテとレセプトを全量ください**」とデータをもらう。
3. **全量を安全にもらうため、匿名化を"院内で・我々のツールで"行う**（生PHIを外に出さない）。これが信頼の決め手。

> このラインは、業界コンサルの王道手法（後述）と一致し、かつ先方の制約（閉域網・PHI敏感・断片データ）を**逆に強みに変える**構成。

## 1. 業界コンサルの手法（調査結果）— これを"型"として借りる

診療報酬請求精度コンサルの王道は**多段点検**（[システム環境研究所](https://syskan.com/menus/main_services/medicalfees.html) 等）:

| 段階 | 内容 | 我々の武器 |
|---|---|---|
| **1次点検** | レセプト（電算データ）を機械点検し、算定漏れ・査定リスク・未取得の施設基準を洗い出す | **決定論点検レイヤー**（適応/禁忌/併用/算定漏れ）＋ baseline-diff |
| **2次点検** | 電子カルテと突合し、レセに反映されていない実施を特定 | カルテ側データ（処置/検体/リハビリ）を取込み突合 |
| **ヒアリング** | 医事課・診療部門に要因を確認、研修で横展開 | レポート＋当日ヒアリング |
| **成果物** | 「算定状況の現在地」＋「増収/算定漏れ対策の糸口」＋概算増収額 | **売上改善診断レポート**（下記3） |

- 成果例（公開情報）: 救急医療管理加算 加算2→加算1 引上げで**年200万円**（[ニチイ](https://medi.nichiigakkan.co.jp/service/medical/santei_sensor/)）、大規模病院で**年数千万〜1億円**の請求漏れ特定（[MRP](https://column.mrp-spd.co.jp/column-medical/hospital-medicalaffairs)）。
- 算定漏れの主因: カルテ記載漏れ／レセ入力漏れ／チェック不足／改定理解不足／教育不足（[ニチイ コラム](https://medi.nichiigakkan.co.jp/oyakudachi/column/140/)）。
- **我々の差別化**: 「人手コンサル」ではなく**決定論エンジンで機械的・網羅的・再現可能**に点検。外部AI不要でPHIを外に出さない。

## 3. 匿名化の設計（P-1・最重要）

### 3.1 方針
- **院内・オフラインで実行**（VPN閉域網に置ける単体ツール。ネット送信ゼロ）。
- **直接識別子は除去**、**擬似ID化でファイル間の紐付けは保持**（患者/病名/処方…が別ファイルでも同一患者として突合できないと点検にならない）。
- **アルゴリズムに必要な項目は残す**（病名コード・診療行為コード・点数・性別・年齢・診療月/相対日）。
- **復元鍵は先方が保持**（我々は擬似IDしか持たない＝再識別できない）。

### 3.2 具体（列単位ルール）

| 区分 | 元データ | 匿名化後 |
|---|---|---|
| 直接識別子 | 氏名・カナ・住所・電話・保険者番号・被保険者記号番号・受給者番号・カルテ番号 | **削除** |
| 患者キー | 患者ID/カルテ番号 | **擬似ID**（`HMAC(salt, 患者ID)` の先頭N桁。saltは先方保持。全ファイル横断で一貫） |
| 生年月日 | YYYYMMDD | **年齢**（診療月時点）＋必要なら**年齢帯** |
| 日付 | 診療日 | **診療月**（YYYY-MM）＋**相対日**（初診からの日数）※縦覧に必要な範囲だけ |
| 臨床・算定 | 病名コード・修飾語・診療行為コード・点数・回数・性別 | **そのまま保持**（点検に必須。非識別） |
| 自由文 | 病名の自由記載・コメント | **原則除外**（氏名等の混入リスク）。必要時のみ簡易マスキング |

### 3.3 実装形
- `python/.../deidentify.py`（新規・単体スクリプト or 小モジュール）:
  - 入力: 先方CSV群 ＋ 列マッピング設定（どの列が氏名/患者ID/生年月日/病名コード…か）
  - 処理: 上記ルールで脱識別＋擬似ID化
  - 出力: 匿名化済みCSV群（我々に渡す用）＋ 対応表（先方だけが保持）
- **決定論・外部送信なし**。halunasu既存原則と一致。
- 法的位置づけ: 完全な「匿名加工情報」（[次世代医療基盤法GL](https://www8.cao.go.jp/iryou/hourei/pdf/guideline_1.pdf)）まで作り込むと厳格すぎるので、実務は**「委託＋脱識別＋安全管理措置」**で運用（契約・媒体暗号化・分離環境・削除規定）。要配慮個人情報の扱い（[PPCガイダンス](https://www.ppc.go.jp/personalinfo/legal/iryoukaigo_guidance/)）に準拠し、**基幹システムからの分離・暗号化・多層防御**を満たす（[匿名レセGL](https://www.mhlw.go.jp/content/12400000/000923325.pdf)）。

## 4. データ受領スペックシート（P-5・先方に渡す）

「点検のために、各ファイルで最低限これをください」を1枚に:

| ファイル | 必須項目 | 目的 |
|---|---|---|
| 患者 | 擬似患者ID・性別・年齢（生年月日でも可、匿名化で年齢化）・保険種別区分 | 突合キー・年齢性別条件 |
| 病名 | 擬似患者ID・傷病名コード（無ければ名称）・修飾語・診療開始日・転帰・主傷病フラグ | 適応/禁忌・疑い放置・病名整理 |
| 処方 | 擬似患者ID・医薬品コード・数量・投与日数・診療月 | 適応/禁忌/併用・処方料もれ |
| 検体検査 | 擬似患者ID・診療行為コード・実施日・回数 | 判断料もれ・適応・回数 |
| 処置・リハビリ | 擬似患者ID・診療行為コード・実施日・回数・部位/単位 | 適応・回数・背反 |
| レセ（あれば） | ML-AのUKE/レセCSV・**返戻/増減点連絡書** | baseline突合・査定傾向分析 |

- **コード体系**（標準レセ電コードか自院ローカルか）を必ず確認。ローカルなら**対応表**を依頼。
- サンプルは**匿名で数レセ分**を当日 or 直後にもらえるのが理想（形式確定→取込アダプタ実装）。

## 5. 売上改善診断レポート（P-3・成果物の型）

「増収」を煽らず、**確認対象・概算影響額（点数×10円ベース・按分なしと明記）**で誠実に出す（自社の既存方針と一致）:

1. **サマリ**: 対象レセ数・期間・**算定もれ候補 ◯件/概算◯円**・査定リスク ◯件・未取得施設基準 ◯件
2. **算定もれ候補**（検査判断料・処方料・加算 等。根拠＝マスタ/告示）
3. **査定・返戻リスク**（適応病名なし・禁忌・併用・疑い病名放置。根拠付き）
4. **施設基準の取得余地**（届出で取れる加算の未算定）
5. **次の一手**（優先度・概算効果・必要な運用/届出）

- 出力: HTML/PDF＋Excel（医事課の作業用）。既存 baseline-diff 出力を土台に拡張。

## 6. タイムライン（10/18 切替を活かす）

- **〜9月（切替前）**: 現行レセの**一括診断（無料アセスメント）**でバリュー提示 → 全量データ受領の合意。切替クランチにぶつけない。
- **10/18 切替**: 手を出さない（先方の移行に集中させる）。
- **11月〜（切替直後）**: 新レセコン(ML-A)立ち上げ直後は**算定設定ミス・漏れが最多**の時期 → 「新システムの算定を安全に立ち上げる保険」として点検を回す（切替イベントに紐づく強い提案）。

## 7. 運用・契約・コンプラ（P-1と一体）

- **媒体**: ネット非接続のため USB/院内媒体。**暗号化必須**。受渡し記録。
- **契約**: 個人情報の取扱い委託契約 or 覚書（安全管理措置・再委託禁止・**利用後削除**・監査）。
- **保管**: 分離環境・アクセス最小権限・保持期間明記・完了後削除（証跡）。
- **外部AI**: このサイトでは**使わない**（決定論のみ）。「外部送信なし」を契約前でも明言できるのが強み。
