# yamamoto-demo-stg 追加3患者評価

## 結論

2026-07-16に、mock_partner患者`1002 / 1003 / 1005`を専用組織`yamamoto-demo-stg`で
各1回算定した。3患者・7受診はAPIエラーなく完走し、全7算定でOpenAI抽出と永続化済み施設設定を使用した。

既存UKE 21コードに対し、確定月次で件数・点数まで一致したコードは`2/21`（9.52%）、確認候補まで
含めて一致したコードは`5/21`（23.81%）だった。完全一致患者は`0/3`である。

点数差は逸失収益を意味しない。今回の1002・1003には電話再診が混在するが、現行CLIは全受診へ
単一の`home_visit`を指定しており、受診条件の不一致が差分へ含まれる。

## 観察結果

1. **施設設定は有効**: `114057970`在宅データ提出加算は3患者すべてで確認候補へ到達した。
2. **単純な訪問診療は再現可能**: 混在受診のない1005では、訪問診療料と訪問診療時ベースアップ評価料が一致した。
3. **受診区分の混在を再現できない**: 1003の電話再診を訪問診療として扱い、訪問診療料とベースアップ評価料を
   既存1回に対して2回計上した。
4. **同一建物区分を再現できない**: 1002の既存UKEは同一建物向けコードだが、当社は同一建物居住者以外向け
   コードを生成した。
5. **在宅機器・管理料が未検知**: 1002の人工呼吸器管理料・人工呼吸器加算等は候補にも到達していない。

## 判断

1005は現行の施設設定と`home_visit`入力で比較しやすい。一方、1002・1003の結果をそのまま
「算定エンジンの精度」とみなすのは不適切である。受診単位の電話再診・定期訪問・同一建物区分を
評価入力へ追加した後に再測定する必要がある。

測定条件は[methodology.md](./methodology.md)、詳細は患者別READMEと`raw/result.json`を参照する。
