## 施設設定の効果

Yamamoto施設には、在宅データ提出加算とベースアップ評価料の施設基準、および訪問診療関連の
3自動算定ルールが登録されている。

- `114057970` 在宅データ提出加算は5患者すべてで確認候補として検知した。
- 同コードは安全側の`candidate`設定なので、確定月次明細へは自動採用されなかった。
- `180725910` 訪問診療時ベースアップ評価料イは、既存UKEにも同コードがある3患者で確定一致した。
- 1004は既存UKEが同一建物向けの`180726010`ロ、当社は同一建物居住者以外向けの
  `180725910`イを出した。現在の共通入力`home_visit`だけでは居住形態を区別できない。

施設設定が読み込まれ、候補化へ影響することは確認できた。一方、施設設定だけで患者・受診固有の
診療条件を補うことはできない。

## 主な差分

1. **受診条件の情報不足**: 1004では施設入居者向けコード、1012では定期訪問・臨時往診等の区別が
   必要だが、今回のCLIは全受診へ単一の`home_visit`を渡している。
2. **候補と確定月次の件数差**: 1004・1006では候補面に訪問診療料4回がある一方、確定月次には1回だけ
   入った。候補の採否状態と月次集約の接続を追加調査する必要がある。
3. **在宅管理・機器加算の未検知**: 気管切開、在宅酸素機器、在宅移行、包括的支援、情報連携等の
   複数コードが候補にも到達していない。
4. **安全側候補化は機能**: 在宅データ提出加算、がん性疼痛緩和指導管理料、在宅酸素療法指導管理料は
   一部で候補まで到達し、自動確定はされなかった。
