# 患者1006 カルテから月次レセプト比較 E2E

## 結論

合成患者1006のカルテ4件を使い、STGで次の経路を3回実行した。

`カルテ本文 -> セッション算定候補 -> 患者×月の月次レセプト案 -> 既存UKEとの比較`

処理経路は3回とも完走した。既存UKEは6コード・12算定回・4,326点、当社月次結果は5コード・9算定回・579点で、既存コードとの完全一致は0件だった。月次結果と候補明細は3回とも同一だが、確認事項は17件、18件、16件と揺れた。

重要なのは、既存UKEの`がん性疼痛緩和指導管理料`を見落としているだけではなく、毎回200点の確認候補として意味的に検知できていた点である。ただし、自動採用せずレビュー候補に留めたため月次明細には入っていない。在宅診療・施設基準・同月履歴を確認できない現状で自動採用しない判断は安全側である。

## 評価

### 検知できたもの

カルテには悪性腫瘍、疼痛評価、オピオイド、レスキュー使用、副作用対策、家族への説明が記載されている。既存の管理料知識ルールはこれを拾い、`がん性疼痛緩和指導管理料の確認`を毎回表示した。これはカルテから算定候補へつなぐ経路が一部機能している根拠である。

一方、既存UKEでは同月1回の明細なのに、4受診すべてで提案している。同月履歴による重複抑制は、レビュー候補にも適用する必要がある。

### 検知できなかったもの

現行のfee契約が持つ診療区分は `outpatient / inpatient` の2値だけで、4受診を訪問診療として表現できない。このため初診・再診・外来管理加算へ流れた。特に外来管理加算52点は既存UKEに存在せず、在宅患者を外来扱いした副作用である。

在宅移行早期加算、在宅データ提出加算、在宅医療情報連携加算、ベースアップ評価料は、カルテ本文だけでなく、算定開始日、届出済み施設基準、連携体制・実績などの構造化情報が必要である。今回のSTG施設基準キーは0件だった。

## 安定性

候補明細と月次579点は3回とも同一だった。一方、OpenAIが抽出したイベント数は各反復で `12/4/5/11`、`6/4/5/9`、`5/8/6/7` と変動した。画面上の確認事項も17・18・16件で、タイトルや重複数が一致していない。

管理料候補5件の構成は3回とも同じだった。内訳は、がん性疼痛緩和指導管理料4件と、最終受診の特定疾患療養管理料1件である。
