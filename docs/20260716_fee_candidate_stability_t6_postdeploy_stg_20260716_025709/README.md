# 候補安定化 T6 デプロイ後STG再計測（3患者）

## 結論

fee-api STG revision `fee-api-stg-00159-qzl`に対し、患者1001、1006、1007を各3回測定した。

評価経路は次のとおりである。

`カルテ本文 -> セッション算定 -> 患者 x 月の月次レセプト案 -> 既存UKE比較`

3患者、9患者Run、24算定はAPIエラーなく完走し、24算定すべてでOpenAI抽出が使われた。
確定月次点数は3/3患者で安定した。一方、月次候補集合が安定したのは1001だけで、1/3患者だった。

今回のデプロイでT1/T2/T5はSTGへ反映された。T4のコードも実行されているが、Cloud Runへ同梱された
マスタDB成果物に病名・適応行為データが入っていないため、病名駆動候補を1件も生成できなかった。
したがってT6はまだ完了していない。

## T1/T2の効果

1007の在宅酸素療法は、前回のSTGでは次の2行が併存していた。

- `codeCandidates = [114003710, 114004110]`、2,400点
- 同じ`codeCandidates`、0点

今回は同じ集合の候補が1行に統合され、全反復で0点になった。1007の候補点数は
デプロイ前の`2,740 / 2,675 / 2,450点`から`115 / 340 / 50点`へ低下した。
これは算定機会の減少ではなく、コード未確定の2,400点を合計から除外した意図どおりの安全化である。

ただし1007では`140005610 酸素吸入`65点と`113001810 特定疾患療養管理料`225点が反復ごとに出没し、
候補集合全体の安定性は未達だった。

## T4が候補を生成できない原因

正式測定の24算定で病名レーンは毎回実行され、所要時間は中央値11ms、最大66msだった。
しかし`diseaseIndicationCandidateCount`は全24算定で0件だった。

1006を診断用に追加1回実行し、病名本文を保存せず件数だけ取得したところ、4受診すべてで
`resolvedDiagnosisCount=0`、`unresolvedCount=2`だった。

同じコード・同じ病名・同じ2026-06-15マスタ世代で、DB成果物を比較した。

| テーブル | ローカル完全DB | Cloud Run同梱gzipを展開 |
| --- | ---: | ---: |
| `diseases` | 27,684 | 0 |
| `disease_modifiers` | 2,390 | 0 |
| `cc_act_indications` | 218,375 | 0 |
| `medical_procedures` | 11,746 | 11,746 |

Cloud Runと同じ`python/data/master/standard-master.sqlite.gz`を別ファイルへ展開し、
`disease_act_candidates`を直接実行しても全病名が未解決になった。一方、完全DBでは同じ入力から
特定疾患療養管理料などの候補を返した。
`/readyz`が返したgzip SHA-256とローカル配布物は、どちらも
`0451c6eca0554ec36c72a8368f86ab79bc6589ed994cb645e9ae919daf0b8839`で一致している。

したがって原因はT4のPython/Node配線ではなく、**デプロイ成果物のgzipへ病名マスタと
`cc_act_indications`を収録していないこと**である。完全DBからgzipを再生成し、Cloud Runへ反映する必要がある。

診断用raw: [result.json](./diagnostics/1006-one-run/result.json)

## T5で判明したボトルネック

| エンドポイント | サーバー合計中央値 | セッション取得 | その他 |
| --- | ---: | ---: | ---: |
| 月次集計 | 4,702ms | 4,637ms（98.6%） | 候補制約63ms、集計3ms |
| 既存レセ差分診断 | 4,537ms | 4,493ms（99.0%） | 監査49ms、比較構築4ms |

測定中の月内セッション取得件数は339件から361件へ増えた。患者別月次レセプトAPIでも、
`listSessionsForMonthlyView`へ患者IDを渡さず組織・請求月の全セッションを取得し、その後に患者で絞っている。
今回の4.6秒前後はPython制約照合や集計ではなく、ほぼセッション一覧取得で消費されている。

改善対象は次のとおりである。

1. 患者別月次レセプトはFirestore/storeへ`orgId + claimMonth + patientId`を渡して取得段階で絞る。
2. 差分診断はbaselineに含まれる患者ID集合を取得条件へ渡す。
3. 請求月、患者ID、更新日時に対応する複合インデックスを用意する。
