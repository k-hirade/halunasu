# 測定方法

## 対象

モックHOMIS由来の合成患者1001、1006、1007を使用した。請求月は2026-06。

## 手順

1. `/readyz`で新revisionの起動を確認。
2. 1007を1反復実行し、マスタDB展開とT1/T2/T4/T5の応答を確認。
3. 1001、1006、1007を各3反復、直列で実行。
4. 確定明細、月次候補、確認事項、抽出イベント、既存UKE比較、区間時間を保存。
5. 1006を追加1反復し、病名レーンの解決件数・未解決件数を診断。
6. Cloud Run同梱gzipをローカルの別DBへ展開し、完全DBとテーブル件数・病名逆引きを比較。

正式測定コマンドの形式:

```bash
npm run eval:fee-monthly-chart-e2e -- \
  --patient-dir <patient_sources>/<patient-id> \
  --repeat 3 \
  --seed-known-prior-history \
  --encounter-setting home_visit \
  --output-dir <report-dir>/<patient-id>/raw
```

## 評価条件

- STGへ直接接続
- 組織: `nishiyama-demo-stg`
- 既知受診歴をseed
- 施設基準キーは未登録のため空
- 候補は採用せず、確定一致と候補込み検知を分離
- 正式測定前にマスタをウォームアップ

## 情報管理

raw結果には患者氏名、カルテ本文、ログインID、パスワードを保存していない。
診断用rawも病名本文を除去してはいないが、対象は合成患者であり、患者氏名とカルテ本文は含まない。

## 制約

- T4のデータ欠落により、T4適用後の候補安定性はまだ評価できない。
- 施設基準キーを投入していない。
- Cloud Run、Firestore、OpenAIの負荷を固定していない。
- STGの月内セッション数は測定ごとに増えるため、長期比較では一覧取得時間への影響を考慮する。
