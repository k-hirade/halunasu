## P0 — これが無いと実務が回らない/誤請求・会計不能に直結

### P0-1. 一部負担金(患者窓口負担)の計算【医療事務・小〜中】
- 現状: 完全に未実装(負担割合・窓口金額の計算なし)。
- なぜ必須: クリニックの**会計はこれが無いと成立しない**。点数×10円×負担割合(3割等)の算出は外来の最終出力。
- 規模: 計算自体は小だが、負担割合(年齢・保険種別・公費)を持つ前提が要るためP0-3と連動。

### P0-2. 保険・公費情報の入力と算定反映【医療事務・中】
- 現状: `session.insurance` がフィールドとして存在するだけで、入力UI・負担割合・公費併用・保険者番号がない。
- なぜ必須: レセプト・負担金は保険種別/負担割合/公費で変わる。**全請求の前提**。
- 規模: 中(患者・受診への保険情報モデル + 入力UI)。

### P0-3. レセプト電算(レセ電/CSV)出力【医療事務・中〜大】
- 現状: receiptDraft はテキストコピーのみ。レセプト電算(UKE/CSV)・オンライン請求形式の出力がない。
- なぜ必須: **月次請求(レセプト提出)に必須**。コピーだけでは実際の請求ができず、レセコンへの再入力が要る=導入価値が半減。
- 規模: 中〜大(レセ電フォーマット準拠)。MVPとしてレセコン取込用CSV出力でも可。

### P0-4. 物価対応料の自動配線【医療事務・施設・小】
- 現状: 令和8新設・**届出不要で全機関自動算定**なのに、マスタにコード(180819910初診2点/180820010再診2点)があるのに未配線。
- なぜ必須: 全外来で確実に算定すべき令和8の新点数を**取りこぼしている**。届出不要なので決定論で確実に付与できる。
- 規模: 小(基本料連動で積むだけ)。

### P0-5. 乳幼児加算の自動算定【医師・医療事務・小】
- 現状: 年齢から「小児加算の確認」レビューを出すのみ。基本料の乳幼児加算(初診75/再診38)を**点数化しない**。
- なぜ必須: 小児科の**最頻加算**。生年月日という構造化事実から決定論で算定可能なのに毎回レビュー行きはレビュー負荷が過大。
- 規模: 小(年齢→infant→基本料加算)。時間外/夜間/休日/深夜区分は受付時刻が要るため当面レビューでよい。

## P1 — 精度・査定リスク・運用効率に必須

### P1-1. 併算定不可・算定回数制限の全領域適用【医療事務・中】
- 現状: electronic_exclusions/frequency_limits は lab(検査)のみ適用。処置・画像・処方等に未適用。
- なぜ必須: **査定の主因**。同月制限・併算定不可は全領域で効く。マスタは既にあるので適用範囲拡大が主。
- 規模: 中。

### P1-2. 同月算定制限の実照合【医療事務・中】
- 現状: 「同月内検査確認」「同月履歴確認」はreviewに出すだけ。priorSessions照合はあるが**管理料の月1回・検査の同月重複を判定で落としていない**。
- なぜ必須: 月1回制限(管理料等)・同月検査重複は査定直結。履歴照合の仕組みはあるので判定に繋げる。
- 規模: 中。

### P1-3. 施設基準の登録・管理UI【施設・中】
- 現状: facilityStandardKeys は算定で読むが、ユーザーが届出加算を**登録・管理するUIが弱い**。機器区分・電子画像管理・入院料区分も同様。
- なぜ必須: 施設基準で算定可否が変わる(検体検査管理加算・画像診断管理加算・各種体制加算)。施設ごとに一度設定すれば毎回のレビューが消える。
- 規模: 中(施設プロファイル編集UI + CT機器区分/電子画像管理の属性化)。

### P1-4. 病名と算定の整合チェック・病名管理【医師・医療事務・中】
- 現状: 病名は抽出/補完するが、**医学管理料の対象疾患チェック・病名漏れによる査定**の能動検出が弱い。
- なぜ必須: 「この算定にはこの病名が要る」は査定対策の核。レセプトコメントと並ぶ事務の主作業。
- 規模: 中(対象疾患マスタ × 算定要件)。

### P1-5. レセプトコメント(症状詳記・コメントコード)【医療事務・中】
- 現状: comment_master/comment_links はマスタにあるが、必須コメント・症状詳記の付与が弱い。
- なぜ必須: コメント欠落は返戻・査定の主因。算定にコメントが必須の項目がある。
- 規模: 中。

### P1-6. 算定漏れの能動サジェスト【医師・医療事務・中】
- 現状: candidateWorkbenchの「提案(proposal)」枠がほぼ常に空。
- なぜ必須: 「取れるのに取れていない」加算・管理料(例: 喘息→特定疾患療養管理料225点)を提示。競合(Medicom/CLINICS)が訴求する中核価値。
- 規模: 中(対象疾患×要件で提案生成)。

## P2 — 対応領域拡大・中長期の運用基盤

### P2-1. 月次レセプト集計・点検【医療事務・大】
- 現状: 無い。セッション単位のみ。
- なぜ必須: 請求は月次。患者×月の名寄せ、月内の整合点検、提出前一括チェック。
- 規模: 大。

### P2-2. 入院基本料の令和8対応・DPC【施設・大】
- 現状: 急性期一般入院料1〜6の旧体系ハードコード。令和8でA/B再編。DPCテーブルは0件。
- なぜ必須: 有床・急性期/DPC病院では必須。当面はreview-onlyで安全縮退、本対応は段階的。
- 規模: 大。

### P2-3. 返戻・査定フィードバックループ【医療事務・大】
- 現状: 無い。
- なぜ必須: 返戻/査定結果を取り込み、ルール・病名・コメントを継続改善。長期の精度向上基盤。
- 規模: 大。

### P2-4. 高額療養費・公費複数併用・限度額【医療事務・中〜大】
- 現状: 無い(P0-1/2の負担計算の発展)。
- なぜ必須: 高齢者・公費対象患者の窓口計算に必要。
- 規模: 中〜大。

## まとめ(優先順の考え方)

- **P0は「会計・請求が成立するか」**: 一部負担金・保険公費・レセ電出力が無いと、点数が出ても実務の最終成果物(会計・月次請求)に届かない。物価対応料・乳幼児加算は令和8の取りこぼし/最頻加算で小コスト即効。
- **P1は「査定されない・レビュー負荷を下げる」**: 併算定/回数制限/同月/病名/コメントは査定対策、施設基準UIと能動サジェストはレビュー負荷削減と差別化。
- **P2は「対応領域の拡大と継続改善」**: 月次・入院/DPC・返戻ループは規模が大きく中長期。

最小で実務価値を出す順: **P0-4(物価対応料)→ P0-5(乳幼児加算)→ P0-1+P0-2+P0-3(会計・請求の最小一式)→ P1群**。前2つは小コストで令和8の取りこぼしを止め、後者で「算定して終わり」から「会計・請求できる」へ引き上げる。
