# feeアプリ「全カルテ対応」計画 — 算定マスタ全域と実装の突合に基づく100%化ロードマップ

作成: 2026-07-14。対象commit: 作業ツリー（P0-A〜P1-E＋レビュー修正適用後）。

## 0. 「100%」の定義

制度上、純粋な自動確定100%は不可能である（施設基準・実施事実・医学的判断は入力の外にある）。
本計画で目指す「すべてのカルテで完全に動く」は次の4保証の同時達成と定義する。

| 保証 | 定義 | 現状 |
| --- | --- | --- |
| G1 可用性 | どんなカルテ・どんな入力でも落ちず、算定案＋候補＋警告を返す | ほぼ達成（OpenAI不正JSON時のイベント消失が残） |
| G2 誤確定ゼロ | 確定明細に「やっていない/算定できない」行が入らない | 外来系は達成。加算・回数・背反の自動整合が未完 |
| G3 検知100% | 正しいコードが「確定」か「候補」のどちらかに必ず現れる | 実測: 確定6/15＋候補5/15（mock在宅3患者）。構造的な穴が残る |
| G4 到達動線 | 人の確認だけで候補→確定に到達できる（UI含む） | 候補承認はセッション画面のみ。月次からの動線・設定UIが不足 |

### 1.3 goldデータセットの分布の偏り（評価の穴）

v2 gold 391症例の期待コード737件の区分分布: **検査50%・基本18%(111/112)・その他18%・画像9%**。
**113医学管理・114在宅・130注射・150手術・190入院はほぼゼロ**。今回のmock在宅3患者で0/15を長期間検出できなかった直接原因はこの偏りである。

## 3. 改善計画

### P0: 沈黙劣化の根絶（G1/G2） — 小粒・即効

1. **P0-1 背反・回数・包括の確定側自動整合**: 確定明細内で背反ペアが同時に立った場合、rule_kind に従い劣後側を候補へ降格（現在は警告のみ）。回数上限超過分も同様に降格。データは取込済み、エンジン適用箇所も既存。
2. **P0-2 診療月×マスタ適用期間の明示警告**: `effective_from` 範囲外の診療日は「マスタ適用期間外（2026-06〜）」を診断・算定の両方で表示（F3恒久化）。
3. **P0-3 抽出失敗の縮退を可視化＋1回リトライ**: 不正JSONは即時1回再試行し、それでも失敗なら calculationResult に `extractionDegraded` フラグ＋fee-webでバナー表示。
4. **P0-4 platformStore側のdelegate完全性テスト**: F4と同型の穴を全ストアで横展開検査。

### P1: マスタ駆動の汎用エンジン3点（G2/G3の主戦場） — 本計画の核

1. **P1-1 注加算エンジン**（594件を1機構で）: マスタの注加算識別・同一区分番号・年齢条件[40-41]をカラム化し、「親コードが確定明細に立ったら、条件（年齢・時間帯・施設基準）を満たす注加算を自動確定/候補付与」。乳幼児加算・時間外加算(処置/手術/検査)類が一括で閉じる。
   - 受付時刻フィールドをセッション契約＋fee-web入力に追加（時間外・休日・深夜の判定材料）。
2. **P1-2 きざみエンジン**（616件）: raw[30-33]をカラム化し、数量（分・cm²・回）→点数を汎用計算。抽出スキーマは処置面積・時間の器を既に持つ。人工呼吸・時間加算系・酸素等が閉じる。
3. **P1-3 年齢条件エンジン**（531件）: 生年月日×診療日から、年齢限定コードの自動選択と年齢外の自動除外（誤確定防止にも効く）。
4. **P1-4 知識ルール候補のコード解決修正**: がん性疼痛・難病候補が code null になるSTG事象の根治（一致計測とコード基準の畳みに必須）。
5. **P1-5 薬剤総合評価調整管理料ほか知識ルール増設**: 反例コーパス由来の管理料シグナルを随時追加（データ追加のみで済む構造は整備済み）。

### P2: 領域拡張（G3） — 中粒

1. **P2-1 fee-web設定UI**: autoBillingRules編集（コード検索→confirm/candidate/対象区分/必要届出）、施設基準の有効期間編集、受診区分の施設既定値。※恒常ルールはAPIのみ実装済のため、UIが全カルテ運用の前提。
2. **P2-2 月次候補の承認動線**: 月次レセプト案のcandidateLinesから該当セッションのproposal承認へワンクリック遷移（G4）。
3. **P2-3 処置の一般化**: treatment_fees を3種別ハードコードから、140区分のマスタ照合＋きざみ（面積・時間）駆動へ置換。
4. **P2-4 在宅の決定論化**: 施設恒常ルール前提を保ちつつ、在医総管・訪問回数（週3上限は取込済み）・単一建物区分をルール条件語彙に追加。
5. **P2-5 入院の本線化**: 入院基本料（190系2,324件）を inpatient_fees＋施設設定（届出病棟種別）で確定へ。食事療養(197)を日数×食数で追加。DPCはテーブル取込済みのため係数×診断群の算定を接続。
6. **P2-6 手術・麻酔(150系3,464件)は「候補＋記載要件」戦略を維持**: 術式確定は人の領域。Kコード候補＋施設基準＋背反チェックで支援に徹する（自動確定は狙わない）。

### P3: 評価と回帰（G全体の担保）

1. **P3-1 gold分布の拡充**: 113/114/130/150/190系の症例を最低各20件追加し、区分別recallをCI表示。counterexamples корпусへの失敗追加を運用ルール化（今5件）。
2. **P3-2 2段recallのCIゲート**: confirmedCodeRecall／candidateCodeRecall を毎PRで計測し、低下をブロック。
3. **P3-3 STG契約スモーク（夜間）**: login→設定PATCH→GET一致→home_visit算定→月次candidateLines存在、の5点を毎晩実行（F3/F4型の本番乖離を24h以内に検出）。
4. **P3-4 性能予算**: 月次集計・差分診断は現在1.8秒/1.7秒（候補集約で+30%）。制約lookupの結果キャッシュ（org×月）と候補保存のスリム化で1秒未満へ。

## 4. 到達見込み

- P0＋P1完了時: 外来カルテの「確定＋候補」検知は、マスタに名称がある行為について構造的に100%（G3）。誤確定要因（背反・回数・年齢外）は自動整合で排除（G2）。
- P2完了時: 入院・在宅・処置を含む実運用領域で確定率が上がり、残差は「人の確認が制度上必要な項目」のみに収束。
- 手術麻酔・DPC請求の完全自動化は対象外（候補＋点検支援まで）。
