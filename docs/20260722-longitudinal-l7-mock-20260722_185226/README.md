## 結論

`1002 / 1004 / 1006 / 1010 / 1011`を各3反復、1反復4カルテの時系列でSTGへ投入した。
合計60回の算定APIはエラーなく完走し、実行前後ともCloud Run revisionは
`fee-api-stg-00169-6qx`だった。

L7は**未合格**と判断する。縦断メモ自体の基本観測は期待どおりだったが、現行評価CLIでは
受診ごとの区分と同一建物区分を入力できず、訪問診療料とベースアップ評価料が全件で保留された。
このため「確定明細・月次回数が従来どおり」「UKE再現率が導入前より悪化しない」を正式には確認できない。
さらに、月次候補点数が1002と1011で反復間変動し、確認事項数は5患者すべてで変動した。

したがって、Phase 1をこの結果でクローズせず、PRODの`FEE_EXTRACTION_MEMO`は有効化しない。

## 評価入力の不足

現行CLIは全カルテへ同じ`setting`だけを渡し、`encounterDetails.sameBuilding`を送っていない
（`scripts/evaluate_fee_monthly_chart_e2e.mjs:274-283`）。一方、施設ルールは同一建物向けコードを持つ場合、
`sameBuilding`が未入力ならコード選択を保留する
（`services/fee-api/src/server.js:7424-7435`）。施設設定には訪問診療料とベースアップ評価料の
同一建物内外2コードがある（`samples/yamamoto-demo-stg/fee-settings.json:72-95`）。

また1002には電話再診が含まれるが、今回CLIは4カルテすべてを`home_visit`として送った。
したがって、今回の0点をUKE精度や縦断コンテキストの精度として評価してはいけない。
