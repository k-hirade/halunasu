# 導入前 一括レセプト差分診断 方針（改訂版）

作成日: 2026-06-29

## 目的

既存レセコンの確定済みレセプト（baselineClaim）と、同じカルテ/オーダー/病名を当社エンジンで再算定した結果（engineClaim）を**患者×暦月（レセ1枚）単位**で突合し、**算定もれ／要確認／検討**の3分類で月次点検・算定改善レポートを作る「導入前の一括診断」。

- これは「既存レセを正解(gold)として当社を評価する」機能では**ない**。既存レセにも算定漏れ・過剰・コメント/病名不足があり得るため、**差分はすべて「要確認」**として両方向に提示する。
- コンサルの「レセプト診断（カルテ×レセ照合／外来・入院別／算定もれ・確認・検討の3分類）」をAIで自動化し、根拠付き・監査可能にする。

## 既存資産（再利用）
- `python/medical_fee_calculation/claim_batch.py`：actual vs expected の比較（点数差・不足/余剰コード・verdict・優先度）。**record単位**。
- order-csv パイプライン（`run-order-csv-claim-pipeline(-batch)` ＋ 病院別 order-contract）：病院CSV→claim JSONL→決定論算定。
- 注意：`expected_*` / `gold` は**内部名のまま温存**（契約・テスト依存）。**外向き（新モデル・出力・レポート）だけ baseline / 差分診断 用語**に統一する。全面リネームはしない。

## レビューで確定した4つの是正点（設計に必須）

1. **`over` を「過剰」と短絡しない（信頼の核心）**
   `over`（既存にあり当社で再現せず）は2系統：(a) 既存の過剰、(b) **当社の取りこぼし**（SOAP抽出漏れ・辞書未収載・未対応領域）。
   - 出力は「**要確認（理由二分: 当社未対応の可能性 / 既存の過剰の可能性）**」。
   - **既知の当社未対応コード（known_unsupported_codes）**に該当する `over` は (b) 側＝「検討」へ寄せ、過剰と表示しない。

2. **コード突合は正規化／対応表が本体（偽差分の爆発防止）**
   レセはまるめ・包括・加算の親子・名称ゆれ・コード体系差がある。素の集合差では偽差分が爆発する。
   - baselineClaim 化＝**コード正規化＋セマンティック対応表**を adapter の中核に置く。比較器は正規化済みコードで突合する。

3. **再算定の経路と PHI（外部LLM）の両立**
   - 当社の再算定には2経路：**構造化オーダー経路（order-csv → 決定論算定, LLM不使用）** と **自由文SOAP経路（OpenAIでカルテ本文→事実抽出）**。
   - **Phase A は構造化オーダー経路を既定**（外部LLMにカルテ本文を送らない）。SOAP-LLM経路は「匿名化＋同意/BAA前提のオプション」に分離。

4. **「患者×月」集計の経路を定義**
   `claim_batch` は record単位で、患者×月の名寄せ・複数受診マージ・**診療実日数**を持たない。
   - 患者×月の集計（baselineClaim も engineClaim も）を行う層を新設する（Phase A）。

## その他の方針
- **baselineスコープ前処理**：同月・同一医療機関・同一保険に絞り、他院/過去/予定/外注結果など対象外行を除外してから突合。
- **概算影響額**：「点数×10円・総医療費ベースの概算」と明記。負担割合/保険者按分の金額は出さない。
- **Phase A の対象範囲**：**外来×出来高に限定**。入院/DPCは「検討」へ縮退。
- **安全原則**：「増収」表現は使わない（確認対象点数／概算影響額）。実施事実・算定要件・施設基準・病名の確認を前提とし、自動確定しない。根拠（本文/オーダー）を必ず添える。低確信の算定もれ候補は「検討」へ。

## 機能設計

単位＝**患者 × 暦月 ＝ レセ1枚**。

入力：既存レセ(UKE or レセコンCSV) ／ カルテ・オーダー・病名 ／ 施設基準スナップショット ／ マスター世代 ／ 診療月 ／ 医療機関コード・診療科・保険情報。

処理：
```
既存レセ取込 → 正規化 → baselineClaim 化（患者×月）
カルテ/オーダー/病名 → claimContext 化
当社エンジンで患者×月を再算定 → engineClaim 化（決定論・LLM不使用が既定）
baselineClaim ⇔ engineClaim 比較（正規化コードで突合） → 3分類・レポート化
```

出力3分類：
| 分類 | 意味 | 内部verdict対応 |
|---|---|---|
| 算定もれ候補 | 当社で候補だが既存に無い（要件充足後に算定可能） | under |
| 要確認 | 既存にあるが根拠不足/過剰/査定リスク（病名・部位左右・施設基準・コメント・背反） | mismatch ／ over(a) |
| 検討 | ルール・運用・データ不足で保留（CSV列不足・履歴不足・低確信・DPC/入院・当社未対応over(b)） | unsupported ／ low_confidence ／ over(b) |

金額表示例：
```
算定もれ候補: 1,230点
概算影響額: 12,300円（点数×10円・総医療費ベースの概算）
注意: 実施事実・算定要件・施設基準・病名を確認のうえ判断
```

## 実装フェーズ
- **Phase A（実用バッチ診断）**：baselineClaim モデル → UKE/レセコンCSV取込adapter（正規化込み）→ 患者×月集計 → baseline⇔engine 比較器 → 3分類レポート（初回は1形式: XLSX or HTML）→ 手元/STGで実行し病院へ返却。構造化オーダー経路・外来出来高限定。
- **Phase B（STG業務アプリ化）**：fee-web 月次点検に「既存レセとの差分」モード、アップロードUI、患者×月詳細・根拠・分類・確認メモ。PROD一般公開は不要。
- **Phase C（継続診断）**：病院別adapter固定、月次再実行、算定もれ率/要確認率/査定リスク/改善推移、再審査・症状詳記・施設基準の時点管理連携。
- **Phase D（PROD化判断）**：2〜3施設でadapter安定／文言が医療事務に通じる／PHI運用・削除手順確立／「増収」表現排除／得意・縮退範囲明記、が揃ったらPROD化。

## 運用・PHI（STG運用型でも本番相当に扱う）
IP制限、アクセス権、ログに本文を出さない、保存期間、削除手順、PHIをGit/docsに残さない、外部LLMに送らない（構造化経路を既定／SOAP経路は匿名化）。
