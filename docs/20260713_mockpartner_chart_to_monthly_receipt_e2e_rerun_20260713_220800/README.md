# mock_partner カルテから月次レセプト比較 STG再走

## 結論

新規デプロイ後のSTG revision `fee-api-stg-00155-kf4`に対し、患者1001、1006、1013をそれぞれ3回再走した。実行前後とも、Cloud Runのトラフィックは同revisionへ100%向いていた。

評価経路は次のとおりである。

`カルテ本文 -> セッション算定候補 -> 患者×月の月次レセプト案 -> 既存UKEとの比較`

全9患者RunはAPIエラーなく完走した。既存UKEの15コード・25算定回・8,752点に対し、**自動採用された確定月次明細の一致は各反復とも0/15コード**だった。一方、今回の月次レスポンスではレビュー候補も患者×月で集約され、患者1013の次の2コードを各反復で検知できた。

- `113004310` 療養費同意書交付料
- `180000710` 傷病手当金意見書交付料

したがって、候補まで含めた検知一致は各反復`2/15コード（13.3%）`である。これは確定算定ではなく、実施事実と算定要件を人が確認する候補である。

## 新revisionで確認できたこと

### 月次候補の集約

レビュー候補を受診単位のまま並べず、患者×月で集約した結果を取得できた。

- 1006のがん性疼痛緩和指導管理料候補は、受診ごとの3〜4件を月1件へ抑制した。
- 1013の難病外来指導管理料候補は、2件を月1件へ抑制した。
- 1013の文書料2件はコード付き候補として月次へ残り、既存UKEと一致判定できた。

月内重複抑制は機能している。ただし、がん性疼痛緩和指導管理料候補はコードが`null`のため、既存UKEの`113012810`と機械的に一致できていない。

### 不要候補の減少

直前のrevisionで3患者に出ていた`在宅患者連携指導加算（訪問看護・訪問看護（同一））`候補は、今回の全反復で出なかった。既存UKEにもない候補だったため、候補ノイズは減った。

## 抽出失敗

1006の反復1・受診1だけ、OpenAI抽出結果を利用できず`rules_fallback`となり、臨床イベントは0件だった。Cloud Runログに記録された原因は次のとおりである。

`OpenAI structured response JSON parse failed: Expected ',' or '}' after property value in JSON at position 1596`

このリクエストは約34.0秒後にHTTP 201で完了し、基本料は計算されたが、カルテ由来イベントは失われた。24算定中1件のため、今回の`allCalculationsUsedOpenAi`はfalseである。フォールバック自体は可用性を維持したが、精度劣化を伴うため、構造化JSONの再取得または修復と、UI上の明示が必要である。

## 残る主要ギャップ

1. 評価セッションが`outpatient`固定で、在宅患者訪問診療料・在宅加算を正しく評価できない。
2. がん性疼痛緩和指導管理料候補をコード`113012810`へリンクできず、候補検知一致に数えられない。
3. 外来管理加算の採否が同一入力で揺れる。
4. OpenAIの不正JSON時にルールフォールバックへ落ち、カルテイベントが0件になる場合がある。
5. 施設基準キーが0件のため、施設基準が必要な加算の自動採用可否は評価していない。
