# mock_partner 残り10患者 カルテから月次レセプト比較

## 結論

新規デプロイ後のSTG revision `fee-api-stg-00157-mw2`に対し、先行評価した1001、1006、1013を除く10患者を、それぞれ3回実行した。

評価経路は次のとおりである。

`カルテ本文 -> セッション算定候補 -> 患者 x 月の月次レセプト案 -> 既存UKEとの比較`

10患者・28受診・84算定はAPIエラーなく完走し、84算定すべてでOpenAI抽出結果が利用された。一方、既存UKE 60コードに対するコード・回数・点数の確定一致は、反復1が`1/60`、反復2・3が`0/60`だった。確定明細とコード候補集合を合わせた検知は`2/60 -> 1/60 -> 1/60`である。

この低い一致率を、そのまま全算定領域の精度とは扱えない。既存UKEは在宅診療・在宅管理コードが中心だが、比較条件を先行評価と揃えるため、全セッションを`outpatient`、既知の受診歴なし、施設基準キー0件で実行している。その結果、在宅コードを算定できず、各反復の先頭受診を初診として扱う構造的な入力差がある。それでも、同一入力で月次点数が安定した患者は10人中3人だけであり、抽出と外来管理加算の揺れは実装上の課題として残る。

## 判断と次の評価

1. **在宅対応を同じ評価軸へ混ぜない**: 在宅患者は`home_care`相当の診療区分、訪問先区分、同一建物人数、管理料、施設基準を入力して再評価する。
2. **既知受診歴を接続する**: 新規合成患者を毎回作る現方式では先頭受診が初診になる。`patients.csv`の既往受診日を安全にseedした評価を別Runで行う。
3. **外来管理加算の再現性を上げる**: 7患者の月次変動に直接影響した。LLM抽出結果ではなく、診療行為・除外条件から決定論で判定する範囲を増やす。
4. **候補採用後も評価する**: 現在は候補を採用していないため、レビュー運用を含む最終レセプト精度は測れていない。
5. **対応領域別に指標を分ける**: 外来、在宅、入院を分け、入力契約を満たした症例だけで本番精度KPIを算出する。
