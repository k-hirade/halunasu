# レセプトプレビュー最終ゴールと実装計画

作成日: 2026-07-04

## 背景

月次画面に患者 x 請求月のレセプト案を置く方針にしたため、プレビューを「診療セッションごとの明細サマリ」ではなく、実際のレセプト業務に近い形へ寄せる必要がある。

ユーザー提供画像（IMG_6741/6742/6743、2026-07-04 19:40:05 screenshot）は、いずれも医科入院外の紙レセプト例または解説画像である。画像から確認できる構造は以下。

- 診療年月単位の明細で、日ごとに1枚ではない。
- 上部に保険者、公費、記号番号、患者、医療機関、診療年月、入外区分がある。
- 左側に傷病名、診療開始日、転帰、診療実日数、点数欄がある。
- 右側に摘要欄があり、診療行為、日付、時刻、回数、コメント、症状詳記、理由などが入る。
- 点数欄は月次集計、摘要欄は確認・提出に必要な詳細という役割分担になっている。

公式資料では、診療報酬明細書は様式第2等に従って作成すること、同一月でも入院と入院外はそれぞれ1件の明細書として扱うこと、保険者番号等が月途中で変更された場合は変更ごとに別明細書を作成することが示されている。

## 重要な業務前提

### レセプトは日ごとではなく月次

基本単位は「患者 x 診療年月 x 入外区分 x 保険条件」である。日ごとに別レセプトを作るのではなく、月内の診療をまとめた明細書を作る。

ただし、摘要欄では日付・時刻・実施理由・コメントが重要になる。したがって、アプリ内部では月次合計だけでなく、診療日ごとの発生明細を保持する必要がある。

### 入院外・入院・DPCは同じプレビューにしない

医科入院外、医科入院、DPCは、同じ「レセプト」という言葉でも必要項目が違う。

- 医科入院外: 外来日、初再診、医学管理、投薬、注射、処置、検査、画像、摘要コメントが中心。
- 医科入院: 入院日、入院期間、入院基本料、病棟、食事療養、包括/出来高の扱いが必要。
- DPC: 診断群分類番号、医療資源病名、ICD、入院期間、包括評価、出来高部分、EFファイル的な粒度が必要。

この3つを同じコンポーネントで無理に表示すると、存在しない情報をそれらしく見せるリスクがある。医療事務向けのプレビューでは、見た目の統一よりも「どの項目が根拠付きで埋まっているか」を優先する。

## 完全な最終ゴール

最終的には、月次点検画面で患者を選択したときに、次の状態まで到達する。

1. 患者 x 請求月 x 入外区分 x 保険条件ごとに、提出前のレセプトプレビューを表示できる。
2. プレビューは紙レセプトに近い読み方ができ、医療事務が「どこを確認すべきか」を直感的に判断できる。
3. 月次の点数集計と、摘要欄に出すべき診療日別の明細を同時に確認できる。
4. コメント、症状詳記、病名不足、施設基準、算定要件不足、同日併算注意などを、提出前の確認事項として明細上に紐づけられる。
5. 医科入院外、医科入院、DPCを明確に分岐し、未対応種別では推測表示をしない。
6. 将来の歯科、調剤、訪問看護は別様式として拡張できる。
7. プレビューは提出用データそのものではなく、提出前レビュー用であることを明確にする。UKE/電子レセプト出力がある場合はそちらを正式出力とし、プレビューは確認支援に徹する。

## 対象パターン

### P0: 医科入院外

最初に完成度を上げる対象。現在のアプリの入力・算定候補・月次点検と最も近い。

必要表示:

- 診療年月
- 保険者番号、記号番号、公費
- 患者氏名、性別、生年月日
- 医療機関名、医療機関コード、都道府県
- 入外区分
- 診療実日数
- 傷病名、診療開始日、転帰、主病
- 公式区分に寄せた点数欄
- 摘要欄
- コメント、症状詳記、要確認事項
- 請求点数、総医療費、一部負担金

### P1: 医科入院

医科入院外とは別レンダラーにする。初期段階では「簡易プレビュー + 不足項目の明示」に留める。

追加で必要な情報:

- 入院日
- 退院日
- 入院日数
- 入院基本料
- 病棟/病床区分
- 食事療養/生活療養
- 入院中の出来高算定項目
- 入院料と外来分の分離

### P1: DPC

DPCは医科入院よりさらに別扱いにする。初期はDPCプレビューの土台だけを用意し、実装は後続。

追加で必要な情報:

- 診断群分類番号
- 医療資源病名
- 主傷病、副傷病
- ICDコード
- 入院期間
- 包括点数
- 出来高算定部分
- DPC対象外/出来高へ戻す理由

### P2: 歯科、調剤、訪問看護

様式自体が異なるため、現時点のfee-webの中心対象からは外す。将来的には `receiptType` の分岐で追加できるようにする。

## なぜ最初は医科入院外プレビューに絞るのか

「最終ゴールを医科入院外だけにする」という意味ではない。最終ゴールは上記の通り、入院・DPCまで分岐可能な月次レセプトプレビューである。

それでも最初の実装対象を医科入院外に絞る理由は以下。

1. 添付画像がすべて医科入院外であり、ユーザーが直近で確認したい実レセプト例と一致する。
2. 現在のfee-webの診療セッション、算定候補、月次点検は外来ユースケース中心である。
3. 現在の `buildMonthlyReceiptDraft` は外来月次集計に近い構造をすでに持っている。
4. 医科入院・DPCは必要データが多く、現状のセッションデータから推測で埋めると危険である。
5. 先に医科入院外で「月次合計 + 摘要明細 + 要確認」の構造を固めると、入院/DPCにも安全に拡張できる。

したがって、設計は全種別を見据えて `receiptType` で分岐可能にし、実装の最初の完成対象を医科入院外にする。

## 現状実装の確認

現在の月次レセプト案は `buildMonthlyReceiptDraft` で作成されている。

- 月内セッションを患者・請求月で抽出している。
- 月内の明細を `code/orderType/name` 単位で集約している。
- 実日数、傷病名、コメント、症状詳記、合計点数を返している。

課題:

- 明細集約時に診療日ごとの発生情報が失われやすい。
- 摘要欄に必要な日付、時刻、コメント、部位、左右、理由を明細単位で持てていない。
- 入院外/入院/DPCの種別分岐が弱い。
- 点数欄の区分が簡易分類で、紙レセプトの読み方と完全には一致しない。
- 傷病名ごとの診療開始日・転帰の表示が弱い。

## 実装計画

### Phase 1: レセプトデータモデルの再定義

目的: 月次集計と摘要明細を両方扱える構造にする。

追加・整理する概念:

- `receiptType`: `medical_outpatient`, `medical_inpatient`, `medical_dpc`, `dental`, `pharmacy`, `home_nursing`
- `claimKey`: patientId + claimMonth + receiptType + insurance/public payer key
- `lineGroups`: 点数欄用の月次集計
- `lineOccurrences`: 摘要欄用の発生明細
- `receiptWarnings`: 病名不足、コメント不足、施設基準、詳記、併算確認など
- `diagnosisRows`: 傷病名、開始日、転帰、主病、根拠

受け入れ条件:

- 医科入院外の月次プレビューで、同じコードの明細が月内複数日に出ても、合計点数と日付別摘要の両方を保持できる。
- 保険者変更や入外混在を同一プレビューに混ぜない。

### Phase 2: 医科入院外プレビューを実レセプト風に刷新

目的: 添付画像に近い読み方ができるUIへ寄せる。

実装内容:

- A4紙面風のコンパクトなレイアウトにする。
- 上部に保険・患者・医療機関・診療年月を配置する。
- 左側に傷病名と点数欄を置く。
- 右側に摘要欄を置く。
- 点数欄は公式区分に寄せて表示する。
- 摘要欄は行為区分・日付・コメント・理由を読みやすく表示する。
- 不足項目は赤/橙の注意として明細に紐づける。

受け入れ条件:

- 医療事務が、点数の合計と摘要の確認点を同じ画面で追える。
- 「必要コメント: 820...」のような内部コード主体の表示を主文に出さない。
- コードは必要な場合のみ補助情報として表示する。

### Phase 3: 月次点検との統合

目的: 月次点検の右ペイン/モーダルで、患者ごとにレセプト案を確認できるようにする。

実装内容:

- 患者一覧からレセプトプレビューを開ける。
- 要確認件数、症状詳記、コメント不足、病名不足をプレビュー内で確認できる。
- CSV/HTML出力と同じ集計ロジックを使う。
- プレビュー更新は「月次再集計」または明示的な再作成で行い、画面操作のたびに無駄な通信をしない。

受け入れ条件:

- 月次点検から患者単位でレセプト確認ができる。
- 点検リストの要対応理由と、プレビュー上の該当箇所が対応する。

### Phase 4: 医科入院・DPCの安全な分岐

目的: 未対応種別を外来プレビューで誤表示しない。

実装内容:

- `receiptType` に応じたレンダラーを用意する。
- 医科入院は簡易プレビューから始め、不足項目を明示する。
- DPCはDPC専用の必要データが揃うまで、簡易サマリと要確認に留める。

受け入れ条件:

- 入院/DPCを医科入院外として表示しない。
- 不足データを推測で埋めない。

### Phase 5: 出力・監査・正式提出データとの整合

目的: プレビューと正式出力の責務を分離する。

実装内容:

- プレビューは人間確認用と明記する。
- UKE/CSV/HTML出力がある場合、同じ `receiptDraft` から生成する。
- プレビュー上の要確認項目を監査ログに残せるようにする。
- 将来的に支払基金ASP/レセ電データとの突合に接続できる構造にする。

受け入れ条件:

- プレビューだけで「正式に提出できる」と誤認されない。
- プレビュー、CSV、HTML、将来のUKE出力の元データが分岐しない。

## UI方針

- fee-webの既存デザインを基調にする。
- レセプト部分だけは紙面風に寄せるが、過度に装飾しない。
- 情報量を増やしすぎず、点数欄・摘要欄・要確認を優先する。
- 紙レセプトの完全再現より、医療事務が確認しやすい提出前プレビューを優先する。
- 外来/入院/DPCの違いをUI上で明示する。

## 非ゴール

- 初期段階で全様式を完全再現すること。
- プレビューだけで正式な電子レセプト提出データを生成した扱いにすること。
- 入院/DPCの不足データを推測で埋めること。
- 画像の紙レセプトをピクセル単位で再現すること。

## 次にやること

1. `receiptType` と `lineOccurrences` を追加する設計を具体化する。
2. 現在の `buildMonthlyReceiptDraft` が失っている日付・コメント情報を洗い出す。
3. 医科入院外の新プレビューコンポーネントを作る。
4. 月次点検の右ペイン/詳細モーダルに接続する。
5. 入院/DPCは外来プレビューに混ぜず、未対応時の安全な簡易表示を先に用意する。
