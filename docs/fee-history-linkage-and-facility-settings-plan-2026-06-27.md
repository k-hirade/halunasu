# 診療報酬算定: 過去履歴連携と施設別算定設定の方針

作成日: 2026-06-27

## 目的

診療報酬算定アプリで、患者の過去診療・過去算定をより正しく利用し、同月/同日/一定期間内の算定制限、初診/再診候補、管理料・判断料の重複確認を強化する。

同時に、施設ごとの届出状況、履歴の保持状況、運用上の確認方針を `https://fee.halunasu.com/admin?section=settings` の算定設定で管理できるようにする。過去情報の扱いは施設ごとの差が大きいため、履歴ロジックだけを先に固定すると、後で施設差分を吸収しづらくなる。

## 前提整理

### 現状の過去紐付けは部分的

現在の実装は、同一患者の過去セッションから一部の履歴を算定オプションへ変換している。

- `services/fee-api/src/server.js` の `loadPriorFeeSessionsForPatient` は同一患者の過去セッションを読む。
- `buildPriorHistoryOptions` は、過去 `calculationResult.lineItems` から `same_month_history_codes`、`same_day_history_codes`、`procedure_history_events` を作る。
- `rejected`、`blocked`、`includedInTotal === false` の行は履歴から除外するガードがある。
- 一方で、エンジン側が受け取れる `same_week_history_codes`、`already_billed_judgement_groups`、`already_billed_lab_management_same_month`、投薬系の月内既算定フラグなどは、過去結果からまだ自動導出していない。
- 取得件数は直近10件固定で、2/3/4/6/12か月の算定制限窓を常にカバーできない。
- 導入前や他レセコンで算定された過去は、現状では履歴ゼロ扱いになりやすい。

### 初診/再診は病院が自由に定義するものではない

ここは重要。初診料・再診料の診療報酬上の定義は、施設ごとに自由に変えてよいものではない。厚生労働省の診療報酬点数表、実施上の留意事項、疑義解釈に基づくべき領域である。

ただし、施設ごとに差が出る論点はある。

- 過去履歴をどこまで信用できるか。
- 導入前レセ、他システム、紙カルテ、他院情報を持っているか。
- 新疾患初診として扱う可能性があるケースを、候補提示に留めるか、必ず人間レビューに回すか。
- 同日複数科、同日再診、紹介状なし受診、情報通信機器等の入力確認フロー。
- 保守的に候補を抑えるか、候補は広めに出してレビューで落とすか。

したがって、設定画面で持つべきなのは「病院独自の初診/再診定義」ではなく、「公式ルールを前提にした施設別の入力補完・候補提示・レビュー方針」である。

参考: 厚生労働省「令和6年度診療報酬改定について」には、医科点数表、実施上の留意事項、施設基準、疑義解釈等へのリンクが整理されている。
https://www.mhlw.go.jp/stf/seisakunitsuite/bunya/0000188411_00045.html

## P0: 履歴抽出の網羅化

目的: 既にエンジンが受け取れる履歴入力を、過去 `lineItems` から自動生成し、月1回・同日・同週・併算定不可の見落としを減らす。

対象:

- `buildPriorHistoryOptions` を拡張する。
- 過去 `lineItems` から以下を導出する。
  - `same_week_history_codes`
  - `already_billed_judgement_groups`
  - `already_billed_lab_management_same_month`
  - 検体採取料、血液採取料、外来迅速検体検査加算などの同日/同月/同週判定材料
  - 特定疾患処方管理料など投薬系の月内既算定フラグ
  - 管理料・指導料の月内既算定フラグ
- `isHistoryCountableLine` 相当の除外ロジックを、履歴導出対象ごとに徹底する。
- 初診/再診については、過去受診がある場合に「再診候補を優先」「初診候補なら要確認」の警告を強化する。ただし自動で病院独自定義に置き換えない。

期待効果:

- 管理料、判断料、検体検査管理加算、処方管理料など、過去算定に依存する項目では精度改善が見込める。
- 特に「本来は月1回なのに再算定候補が出る」「同月既算定なのに警告が弱い」タイプの誤りを減らせる。

限界:

- 急性単発受診や、過去履歴に依存しない処置・検査では大きな精度改善は出にくい。
- 過去算定自体が誤っている場合、誤りが伝播する。

優先度: P0。低コストで効く範囲が広く、現エンジンの受け口も活用できるため最初にやる。

## P1: 履歴取得窓と外部履歴の拡張

目的: 直近10件固定と Halunasu 内算定済み前提を外し、2/3/4/6/12か月の制限窓を実際に参照できるようにする。

対象:

- `loadPriorFeeSessionsForPatient` の `limit: 10` 固定をやめる。
- 算定候補に応じた最大参照期間を決め、最長12か月まで遡れるようにする。
- 件数上限だけでなく、期間条件を併用する。
- 患者別・月別の履歴サマリを作り、Firestore読み取りを抑える。
- 導入前レセ、他レセコン、CSV、レセ電、手入力の過去算定結果を「履歴シード」として取り込める経路を作る。
- 履歴データに `source`、`confidence`、`effectiveMonth`、`importedAt`、`reviewedBy` を持たせる。

期待効果:

- 受診頻度が高い患者でも、長い算定制限窓を取りこぼしにくくなる。
- 導入直後の病院でも、過去レセを取り込めば履歴ベースの精度が上がる。
- 今集めようとしている「患者 x 複数月」のサンプルが、検証データだけでなく実運用の履歴初期値にもなる。

限界:

- 外部履歴の粒度が荒い場合、細かい算定条件までは復元できない。
- レセプトに残らないカルテ上の状態、説明、同意、診療計画は別途カルテから取る必要がある。

優先度: P1。精度向上の伸びしろは大きいが、データ取り込み・正規化・監査の設計が必要。

## P2: 縦覧レビューと履歴ベースの説明可能性

目的: 単回算定の正否だけでなく、患者単位・月単位・複数月単位で、算定の連続性と査定リスクを見えるようにする。

対象:

- 患者別の「管理料算定月マップ」を作る。
- 判断料、検体検査管理加算、指導料、特定疾患処方管理料などを月次で横断表示する。
- 初診/再診候補と過去受診履歴の不整合をレビューUIに出す。
- `buildMonthlyClaimSummary` の出力を、患者横断・月次レビューに拡張する。
- リアルタイム支援では、候補提示の横に「この候補は過去履歴により抑制」「前回算定日があるため要確認」のような説明を出す。

期待効果:

- 現場レビューで見つけやすい査定リスクを、事前に検知しやすくなる。
- 医事課が見たい「この患者は今月この管理料をもう算定済みか」をアプリ内で確認できる。
- AI候補の根拠を説明できるため、ブラックボックス感が減る。

限界:

- 診療内容そのものの抽出精度が低い場合、縦覧だけでは正しい候補は出せない。
- 病院の運用や医事判断を完全自動化するものではなく、レビュー支援として設計すべき。

優先度: P2。価値は高いが、P0/P1で履歴の質を上げてから効果が出る。

## 算定設定ページの方針

目的: 施設ごとの前提条件を設定し、算定ロジックが「何を公式ルールとして扱うか」「何を施設運用として扱うか」「履歴が不完全なときにどう振る舞うか」を明示できるようにする。

### 結論

このタイミングで算定設定ページを設計するのは妥当。理由は、過去履歴の扱いが施設ごとに大きく違い、履歴ロジックだけを先に実装すると「履歴が完全な施設」「導入前履歴がない施設」「外部レセだけある施設」を同じ前提で扱ってしまうため。

ただし、設定画面でやるべきことは「施設ごとの独自算定ルールを作る」ことではない。診療報酬上の公式ルールはマスター/ルールエンジン側で固定し、施設設定では次を扱う。

- 施設の事実: 施設基準、届出、医療機関種別、DPC対象、地方厚生局、算定開始日。
- 履歴の事実: どの期間の過去算定が存在するか、外部レセを取り込んだか、履歴が完全か一部か不明か。
- 運用方針: 履歴不明時に候補を出すか、レビュー必須にするか、手動上書き理由を必須にするか。
- レビュー方針: 自動反映してよい項目、候補提示に留める項目、医事レビュー必須の項目。

### 設定してよいもの / 設定してはいけないもの

設定してよいもの:

- 過去履歴の参照範囲。
- 外部レセ/CSV/手入力履歴を使うか。
- 履歴が不完全なときの挙動。
- 初診/再診候補に履歴矛盾があるとき、警告にするかレビュー必須にするか。
- 新疾患初診の可能性を候補として出すか、医事確認必須にするか。
- 同日複数科、同日再診、情報通信機器、紹介状なし受診などの確認項目を必須入力にするか。
- 施設基準・届出の有効期間。
- 手動上書き時の理由入力と監査ログ。

設定してはいけないもの:

- 「この病院では初診の定義を独自に変える」のような公式ルールの上書き。
- 「Nか月空いたら常に初診にする」のような単純自動判定。初診/再診は疾患の継続性、治癒/中止、同一医療機関での過去受診、診療内容などを確認する必要があり、単純な月数だけで確定させると危険。
- 履歴がないことを根拠に「初診で確定」とする設定。履歴なしは「不明」であって「初診確定」ではない。
- 施設基準の届出がない項目を、設定だけで算定可能にすること。

### 初診/再診の扱い

ユーザーの「初診と再診の定義って病院ごとに違うよね？」という直感は、半分合っていて半分違う。

違う点:

- 診療報酬上の初診/再診の定義は、病院ごとの自由設定ではない。
- 公式の点数表、実施上の留意事項、疑義解釈に従うべき。

合っている点:

- 実務上、初診/再診を判断するために参照できる過去情報の範囲は施設ごとに違う。
- 「過去レセがあるか」「導入前データがあるか」「同一患者の名寄せができているか」「新疾患初診を誰が確認するか」は施設ごとに違う。
- その差は算定候補の出し方、警告の強さ、レビュー必須条件として設定画面に出すべき。

したがって、算定設定では `初診/再診の定義` ではなく `初診/再診の確認方針` として扱う。

具体的には以下の挙動にする。

- 過去受診が確認できる場合: 再診候補を優先し、初診候補が出る場合は理由確認を必須にする。
- 過去履歴がない場合: 初診確定にはせず、「履歴未確認」の状態を持つ。
- 新疾患初診の可能性がある場合: 自動確定ではなく、候補表示 + 医事/医師確認に回す。
- 手動で初診/再診を上書きした場合: 理由、変更者、変更日時を保存する。
- 履歴が外部レセ由来の場合: 信頼度を表示し、完全履歴と同じ扱いにしない。

### 画面構成案

`/admin?section=settings` は、現在の「算定範囲」「マスター検索」「レビュー方針」のプレースホルダから、以下のタブまたはセクションに広げる。

1. 基本施設情報
   - 医療機関コード、都道府県、地方厚生局、病院/診療所、DPC対象、診療科。

2. 施設基準・届出
   - 届出済み施設基準、有効開始日、失効日、対象診療科、証跡メモ。
   - 既存の `facilityStandardKeys` をここで管理する。

3. 履歴の扱い
   - 参照期間。
   - Halunasu内履歴、外部レセ、CSV、手入力履歴の利用有無。
   - 履歴完全性: 完全 / 一部 / 不明。
   - 履歴不明時の挙動: 候補提示のみ / レビュー必須 / 保守的に抑制。

4. 初診/再診・基本診療料
   - 過去受診ありの場合の候補方針。
   - 履歴なしの場合の扱い。
   - 新疾患初診の扱い。
   - 同日複数科、同日再診、情報通信機器、紹介状なし受診の必須確認。
   - 手動上書き理由の必須化。

5. 検査・判断料
   - 判断料グループの月内重複確認。
   - 検体検査管理加算の月内確認。
   - 採血料/迅速検査加算の同日・同週確認。

6. 管理料・処方加算
   - 特定疾患療養管理料、生活習慣病管理料、特定疾患処方管理加算などの履歴確認方針。

7. レビュー方針
   - 自動反映してよい項目。
   - 候補提示に留める項目。
   - 医事レビュー必須の項目。

8. 監査ログ
   - 設定変更履歴。
   - 変更理由。
   - 有効開始日。
   - 過去計算への影響。

### データモデル方針

施設の基本情報と施設基準は、既存の Platform facility を正とする。現在も `packages/platform-contracts` に `facilityStandardKeys` があり、`services/fee-api` は施設プロファイルから `facilityStandardKeys` を算定へ渡している。

Fee固有の算定設定は、Platform facility に無理に詰め込みすぎない。以下のような別リソースに分ける。

```text
organizations/{orgId}/fee_settings/{facilityId}
```

または、複数施設を持つ組織向けに default と facility override を分ける。

```text
organizations/{orgId}/fee_settings/default
organizations/{orgId}/fee_settings/facilities/{facilityId}
```

設定は必ず versioned/effective-dated にする。

```json
{
  "facilityId": "facility_...",
  "effectiveFrom": "2026-07-01",
  "historyPolicy": {
    "defaultLookbackMonths": 12,
    "externalHistoryEnabled": true,
    "historyCompleteness": "partial",
    "missingHistoryBehavior": "review_required"
  },
  "initialRevisitPolicy": {
    "officialRuleBasis": "mhlw_2024_medical_fee_table",
    "priorHistoryBehavior": "prefer_revisit_candidate",
    "newDiseaseInitialHandling": "candidate_requires_review",
    "manualOverrideRequiresReason": true
  },
  "reviewPolicy": {
    "mode": "standard",
    "autoAddAllowedSources": ["deterministic_order_mapping"],
    "reviewRequiredSources": ["history_conflict", "new_disease_initial"]
  }
}
```

この例は設定の形を示すためのもので、病院側にJSONを入力させるという意味ではない。UIではフォーム、チェックボックス、選択式、日付入力で編集させる。

## 実装順

### P0 設定ページの土台

目的: 今ある `算定設定` プレースホルダを、実際に施設前提を確認できる画面にする。

- `apps/fee-web/components/fee-admin-console.js` の設定セクションを実画面化する。
- Platform facility から基本施設情報と `facilityStandardKeys` を表示する。
- 施設基準の追加/削除/有効日管理のUIを作る。
- 初診/再診については「公式ルールを上書きする設定」ではなく、「履歴不整合時のレビュー方針」として表示する。
- 設定変更は監査ログに残す。

### P1 算定ロジックへの接続

目的: 設定ページの値を、履歴取得・候補提示・レビュー判定に反映する。

- `fee-api` が facility fee settings を読み、`buildPriorHistoryOptions`、履歴取得窓、レビュー警告に渡す。
- 履歴完全性が `partial` または `unknown` のときは、履歴依存項目に「履歴不完全のため要確認」を付ける。
- 初診/再診の手動上書き時に理由を保存する。
- 外部履歴を取り込む場合は、取り込み元と信頼度を履歴イベントに保持する。

### P2 高度な施設別運用

目的: 複数施設・複数診療科・改定期・届出変更に耐える。

- 施設基準を effective-dated な履歴として管理する。
- 診療科別 override を追加する。
- 設定変更前後の算定差分シミュレーションを出す。
- 月次レビューUIに、施設設定に基づく縦覧チェックを表示する。

## 正直な見立て

過去情報との紐付けで精度が大きく上がるのは、履歴依存の算定領域である。具体的には、管理料、判断料、検体検査管理加算、処方管理料、同日/同週制限、初診/再診の不整合検知。

一方で、カルテ本文から疾患、処置、検査目的、指導内容、同意、算定要件を読み取る必要がある領域では、過去履歴だけで劇的に改善するわけではない。履歴連携は「候補を増やす」よりも、「重複・期間制限・不整合を減らす」効果の方が強い。

そのため、P0/P1はやる価値が高い。ただし「点数の増加」だけを目的にするとズレる。正しい目的は、算定漏れ候補の発見、過剰算定候補の抑制、査定リスクの事前検知、レビュー理由の説明可能化である。
