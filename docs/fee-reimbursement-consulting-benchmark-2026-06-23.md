# 診療報酬算定コンサル調査と次期実装方針

作成日: 2026-06-23

## 目的

診療報酬算定アプリの次期改善方針を決めるため、診療報酬算定・病院経営コンサルが公開している支援内容を整理し、現行アプリに足りない機能へ落とし込む。

この文書は、医療事務ヒアリングから作成した `docs/fee-medical-clerk-workflow-gap-analysis-2026-06-21.md` の Phase 1〜6 の次に置く判断材料である。

## コンサルが具体的にやっていること

公開情報から見ると、診療報酬算定コンサルの価値は、単純な「レセプト点検代行」ではなく、以下の組み合わせである。

1. レセプト/DPCデータを集め、算定状況を集計する。
2. 算定できている項目、算定できていない可能性がある項目を抽出する。
3. 施設基準、病棟区分、届出、診療実績、看護必要度などを確認する。
4. 同規模病院や同診療科との比較で、低い算定率・収益機会を見つける。
5. 診療報酬改定時に、点数変更・基準変更が収益へ与える影響を試算する。
6. 部門ヒアリングで、算定漏れの原因が記録不足、運用不足、施設基準未届出、医師確認不足のどれかを切り分ける。
7. アクションプランを作り、運用変更・届出・記録改善・算定ルール変更まで伴走する。

つまり、実務上の成果物は次のようなものになる。

- 算定漏れ候補一覧
- 算定率・請求率・査定率のダッシュボード
- 施設基準・届出による算定可能性リスト
- 診療報酬改定の影響額シミュレーション
- 部門別/医師別/診療科別の改善余地
- 改善アクションと進捗管理
- 返戻・査定の原因分類と再発防止策
- 再審査請求の対象候補、根拠、請求書/ファイル作成支援
- 症状詳記・コメントの下書きと確認ワークフロー

## 現行アプリとのギャップ

Phase 1〜6 の実装で、月次レセ点検、病名出し、査定リスク、詳記・コメント、一括候補化、レセ電下書き検証の入口はできた。

一方、コンサルサービスと比べると、まだ不足しているのは以下である。

| 領域 | 現行 | 足りない点 |
|---|---|---|
| 算定漏れ分析 | 個別セッションの候補化と月次要確認はある。`services/fee-api/src/clinical-billing-knowledge.js` に対象疾患→管理料/処方管理加算などの候補化知識は既にある | 月別・診療科別・項目別に、既存の clinical billing knowledge 由来の「本来算定できそうだが算定されていない」候補を流し込む集計が弱い |
| 算定率/査定率 | レビュー件数や状態は見える | 候補採用率、確認完了率、再審査対象率などのKPI定義がない。査定/返戻率は Phase 10 の実績データ投入後でないと実値化できない |
| 施設基準 | 算定ロジックは `facilityStandardKeys` を読める | 施設基準の登録UI、届出状況、受理日、算定開始日、有効期間、要件維持、失効管理が弱い |
| 改定影響 | マスタ更新・令和8監査ドキュメントはある | 改定前後の点数差、月次影響額、患者/診療行為別の増減シミュレーションがない |
| ベンチマーク | なし | 外部比較は難しいが、院内月次・診療科・医師別の相対比較はできていない |
| アクション管理 | 病名確認や注記確認はある | 収益改善施策として、担当者・期限・効果見込み・実績を追う仕組みがない |
| 返戻/査定フィードバック | 査定リスク知識ベースはある。部位/左右不一致は `services/fee-api/src/claim-risk-knowledge.js` で `claim_risk_laterality_mismatch` / `claim_risk_body_site_mismatch` を検知済み | 実際の返戻/査定結果を取り込み、次回ルールや病名/コメント候補へ戻す仕組みがない。検知済みの部位/左右リスクを月次集計・再発防止へ接続する必要がある |
| 再審査請求 | なし | 査定を受けた後に、再審査請求すべき対象を抽出し、根拠を整理し、請求書/ファイル作成を支援する機能がない |
| 症状詳記作成支援 | `receiptAnnotations.symptomDetails` の保存、確定済み詳記のUKE接続、出力前バリデーションはある | カルテ根拠から症状詳記本文を下書き生成するロジックがない。Phase 7 で不足件数だけ数えても、現場の作成負荷を下げられない |

## 安全・コンプライアンス原則

コンサル調査由来の「収益改善」「改善余地点数」は、プロダクト上では慎重に扱う。

1. 実施事実があり、診療録・オーダー・検査値・病名・施設基準などの要件を満たす可能性があるものだけを候補にする。
2. 候補は自動算定しない。医療事務または医師が根拠を確認して採否を決める。
3. 画面文言は「増収」ではなく「算定確認候補」「要件確認候補」「取りこぼし確認候補」を基本にする。
4. 潜在点数は「改善余地」ではなく「候補点数」として表示し、未確認・不確実・要件不足を明示する。
5. 医師別、患者別、診療科別の横断集計は高機密として扱う。RBACで閲覧権限を分け、医師評価・労務評価へ転用しない利用目的をUIと監査ログで制御する。
6. 患者別・医師別ランキングは既定では非表示または管理者限定にする。一般の医療事務画面では、作業に必要な患者単位のレセ点検に絞る。
7. 返戻/査定/再審査データは、審査結果・請求根拠・医師所見を含むため、通常の算定候補より強いアクセス制御と監査ログを要件にする。

## KPI定義

Phase 7 で導入するKPIは、査定・返戻実績が入る前と後で分ける。

### Phase 7 で実値化するKPI

- `candidatePointTotal`: 候補として抽出された点数合計。自動算定済み点数ではない。
- `acceptedCandidatePointTotal`: 人が採用した候補点数合計。
- `unresolvedCandidatePointTotal`: 未確認または保留の候補点数合計。
- `candidateAdoptionRate`: `acceptedCandidateCount / candidateCount`。分母はアプリが提示した候補数であり、真の算定可能件数ではない。
- `reviewResolutionRate`: `resolvedReviewCount / reviewCount`。
- `clinicalKnowledgeProposalCount`: `clinical_billing_knowledge` 由来の対象疾患・管理料・処方管理加算などの提案件数。
- `claimRiskReviewCount`: 部位/左右/適応など、査定リスク分類のレビュー件数。
- `annotationPendingCount`: コメント・症状詳記の下書き未確定件数。

### Phase 10 後に実値化するKPI

- `denialRate`: 確定請求に対する査定点数または査定件数の割合。
- `returnRate`: 返戻件数の割合。
- `reconsiderationCandidateRate`: 査定のうち再審査候補になった割合。
- `reconsiderationSubmissionRate`: 再審査候補のうち請求した割合。
- `reconsiderationRecoveryRate`: 再審査請求で復活した点数/件数の割合。

Phase 10 までは、査定/返戻率はプレースホルダにし、UIでは「実績未取込」と表示する。

## 次に実装すべき方向

### Phase 7: 月次収益改善ダッシュボード

目的は、医療事務が「今月の請求が通せるか」を見るだけでなく、院長・事務長が「どこに改善余地があるか」を見る状態を作ること。

ただし、画面上の主語は「収益改善」ではなく「算定確認候補」に寄せる。過剰請求を誘発しないため、実施事実・要件・根拠の確認前に確定点数や増収額として扱わない。

実装範囲:

- 月次サマリに以下のKPIを追加する。
  - 総点数
  - 算定候補点数
  - 採用済み候補点数
  - 未採用候補点数
  - 要確認点数
  - 病名不足件数
  - コメント/詳記不足件数
  - 査定リスク件数
- `clinical_billing_knowledge` 由来の候補を月次に流し込む。
  - 特定疾患療養管理料
  - 特定疾患処方管理加算
  - 管理・指導・在宅・心身医学系の review-only 候補
  - 同月履歴、主病、指導記録、施設基準などの不足理由
- `claim-risk-knowledge.js` で既に検知している部位/左右不一致を、患者別・項目別・月次の集計へ接続する。
- 診療行為カテゴリ別、患者別、医師別、診療科別に集計できる構造を作る。
- fee-web の月次画面に「請求点検」と「改善余地」のタブを分ける。
- 医師別/患者別の横断集計はRBACで制限する。
- 査定/返戻率は Phase 10 の実績取込前は表示しないか、「実績未取込」のプレースホルダにする。
- KPIカードには分母と注意書きを併記する。

初期実装では外部ベンチマークは不要。まず院内の月次比較、前月比、項目別ランキングを出す。

### Phase 7.5: 症状詳記ドラフト支援

目的は、「カルテには妥当性があるがレセプトからは読み取れない」ケースを、請求前に短時間で補えるようにすること。

Phase 4 でコメント・症状詳記の保存、確定、UKE接続は作ったが、本文生成は未実装である。Phase 7 で詳記不足件数をKPIにするだけでは、現場の作成負荷は減らないため、Phase 7 と Phase 10 の間に挟む。

実装範囲:

- reviewIssue から症状詳記ドラフトを作成する。
- 入力根拠はカルテ本文、診療行為、検査値、画像・処置の目的、病名、既存コメント、過去査定理由に限定する。
- ドラフトには根拠リンク、引用元、未確認項目を付ける。
- 自動確定はしない。ステータスは `draft` → `confirmed` / `rejected` とし、確定済みだけUKEに含める。
- 定型テンプレートから開始し、生成AIを使う場合も院内ポリシー、PII、監査ログ、プロンプト/出力保存方針を満たすまで既定オフにする。
- UIでは「詳記本文案」であり「提出文確定」ではないことを明示する。

### Phase 8: 施設基準・届出管理

目的は、施設基準が理由でレビューに落ちる項目を減らし、届出すれば算定できる可能性を見つけること。

実装範囲:

- 施設プロファイル画面に、届出済み施設基準の管理UIを追加する。
- `facilityStandardKeys` を後方互換の要約キーとして残しつつ、施設基準本体は effective-dated な `facilityStandards[]` として管理する。
  - `standardKey`
  - `standardName`
  - `acceptanceNumber`
  - `submittedAt`
  - `acceptedAt`
  - `claimStartDate`
  - `effectiveFrom`
  - `effectiveTo`
  - `status` (`active`, `pending`, `expired`, `withdrawn`, `unknown`)
  - `regionalBureau`
  - `evidenceUrl`
  - `maintenanceChecklist`
  - `lastSelfCheckedAt`
- 検体検査管理加算、画像診断管理加算、電子画像管理、入院基本料系など、既にロジックが参照している領域から始める。
- 施設基準不足で除外された候補を、月次改善ダッシュボードに集計する。
- 「届出済みなら算定候補」「未届出なら算定不可」「届出不明ならレビュー」を明確に分ける。
- 受理日・算定開始日・有効期間を、受診日ベースで判定する。
- 要件維持が必要な施設基準は、自己点検チェックリストと次回確認期限を持つ。
- 要件失効・届出取消・期限切れがある場合、当該日以降の候補は算定不可または要確認へ落とす。
- 厚労省/地方厚生局の「施設基準届出チェックリスト」は、提出書類としてではなく院内の届出漏れ・要件維持の自己点検として扱う。
- 支払基金の施設基準ASPチェック例に合わせ、届出と異なる施設基準算定を請求前に検出する。

### Phase 9: 改定影響シミュレーション

目的は、診療報酬改定時に、点数変更・基準変更がどの程度収益へ影響するかを事前に見ること。

実装範囲:

- 同じ月次セッションを、改定前マスタ/改定後マスタで再計算する比較モードを作る。
- 行為別に点数差、患者別差額、月次総影響額を出す。
- 新設/廃止/点数変更/施設基準変更を分類して表示する。
- 令和8のような改定対応では、取りこぼし候補と過剰算定リスクを分けて表示する。
- マスタ、ルール、施設基準、clinical billing knowledge は effective-dated な世代を固定して計算する。
- 同一入力、同一マスタ世代、同一施設基準スナップショットなら同一差額になる決定論リプレイを要件にする。
- 再計算時は、元のカルテ本文、オーダー、病名、保険、公費、施設基準スナップショット、算定オプションを保存し、後から検証できるようにする。

この Phase は実運用価値が高いが、マスタ世代管理と比較計算の安定性が前提になる。

### Phase 10: 返戻・査定フィードバックループ

目的は、実際に返戻・査定された結果をアプリに戻し、次回以降のチェックを強くすることに加え、査定をそのままにせず再審査請求の候補化とドラフト作成まで支援すること。

実装範囲:

- 返戻・査定結果を患者/月/診療行為コードに紐づけて登録する。
- A/B/C査定、返戻、照会、詳記依頼を分類する。
- 理由、審査側コメント、再審査結果、院内メモを保存する。
- 同じパターンが次回出たとき、reviewIssue に「過去査定あり」として表示する。
- 月次ダッシュボードで返戻・査定の件数、点数、再発件数を集計する。
- 再審査請求候補を抽出する。
  - カルテ根拠がある
  - 病名/部位/左右/適応/施設基準を補正できる
  - 症状詳記またはコメントで妥当性を説明できる
  - 過去に同様の査定が再審査で復活している
  - 審査の差異の可視化レポートや審査情報提供事例と照合できる
- 再審査請求ドラフトを作成する。
  - 対象レセプト
  - 査定項目
  - 請求理由
  - カルテ根拠
  - 添付資料要否
  - 医師確認要否
  - 再審査等請求ファイル/紙様式に必要な項目
- 支払基金の医療機関再審査等請求ファイルレイアウトに合わせたエクスポートは、仕様確認済み状態と下書き状態を分ける。
- 添付資料ありの場合は郵送提出が必要になるため、オンライン提出用ファイルとは別に「郵送パック」チェックリストを作る。
- 再審査結果を取り込み、復活/原審どおり/一部復活/取下げを記録する。
- `reconsiderationRecoveryRate` は自院データで算出する。約58%という外部調査値は仮説値として扱い、自院実績で上書きする。

## カルテ一括算定との関係

カルテ一括算定は Phase 5 で土台を作ったため、次は「大量に候補を作る」ことよりも、候補を経営・運用改善に変換することが重要である。

コンサル調査から見ると、一括算定の価値は次の順で出る。

1. 未算定・再計算対象を漏れなく拾う。
2. リスクなしのものは一括で処理する。
3. リスクありのものは病名・コメント・施設基準・部位左右に分類する。
4. 月次でどの分類が多いかを見せる。
5. 改善アクションを作り、翌月に減ったかを見る。

したがって、次の実装では一括算定エンジン自体を大きく増やすより、集計・KPI・施設基準・フィードバックを優先する。

## 優先順位

直近で着手するなら、次の順が妥当である。

1. Phase 7: 月次収益改善ダッシュボード
2. Phase 7.5: 症状詳記ドラフト支援
3. Phase 8: 施設基準・届出管理
4. Phase 10: 返戻・査定・再審査フィードバックループ
5. Phase 9: 改定影響シミュレーション

理由:

- Phase 7 は既存の月次サマリ・候補・レビュー情報に加え、`clinical_billing_knowledge` と `claim-risk-knowledge` の既存実装を再集計するだけで価値が出る。
- Phase 7.5 は、詳記不足を数えるだけでなく、実際の作成負荷を下げるために必要である。
- Phase 8 は施設基準不足レビューを減らし、受理日・算定開始日・要件維持を具体的な運用タスクへ落とせる。
- Phase 10 はヒアリングの「査定理由が分かりづらい」に加え、「査定をそのままにしない」再審査請求の高ROI領域に直結する。
- Phase 9 はコンサル価値として強いが、マスタ世代比較の整備が必要なため Phase 7/8 の後が安全である。

## 実装しない/優先度を下げるもの

### 外部ベンチマーク

GHC や日本経営のようなベンチマークは、外部病院データを持っていることが強みである。現時点のアプリで同じことを再現するのは難しい。

当面は、院内の前月比、診療科別、医師別、項目別ランキングで代替する。

### コンサル報告書の自動生成

将来的には有用だが、先に KPI と原因分類が必要である。まずダッシュボード上で正しい数字を出し、その後に月次レポートPDF/CSVを作る。

### DPC本格コンサル機能

DPC分析は病院向けには重要だが、現行の外来中心の算定補助からは大きく離れる。DPCは当面、改定影響や入院基本料レビューに限定する。
