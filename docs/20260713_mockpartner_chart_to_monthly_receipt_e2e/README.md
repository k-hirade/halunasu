# mock_partner カルテから月次レセプト比較 E2E

## 結論

患者1001の合成カルテ2件を使い、STGで次の経路を3回実行した。

`カルテ本文 -> セッション算定候補 -> 患者×月の月次レセプト案 -> 既存UKEとの比較`

処理経路は3回とも完走し、2受診を同一患者・同一月へ集約できた。一方、既存UKEの1,988点に対して当社月次結果は371点で、既存3コードとの一致は0件だった。したがって、この試験を「カルテから既存レセを100%再現できた」と評価することはできない。

以前の[日付補正後STG再測定](../20260713_comparison_mockpartner_data_dateshift_stg/README.md)は、確定コード付きordersを入力して患者1001の3コード・1,988点を完全再現した試験である。今回初めて、そのordersを与えず、カルテ本文だけから通常のOpenAI抽出経路を通した。この違いにより、決定論エンジンの再現性とカルテ経路の入力・抽出精度を切り分けられた。

## 何が確認できたか

1. カルテ2件から別々の算定セッションを作成し、同一患者の2026年6月分として月次集計できた。
2. 月次APIと差分診断APIは、既存のみ3件・当社のみ4件・双方差分0件という同じ比較結果を返した。
3. 算定候補のコード・点数と月次結果は3回とも同一だった。
4. OpenAI抽出後の確認事項は3件、9件、6件と揺れた。抽出イベント数も各受診で `2/8`、`9/4`、`7/2` と安定していない。
5. `patients.csv` の開始日 `2019-04-20` を既往受診として追加した対照試験でも、月次結果は3回とも371点のままだった。

## 不一致の主因

### 1. 在宅医療は現行カルテ経路で確認扱い

`clinical-calculation-input.js` は在宅医療をreview-only領域として扱い、訪問診療か往診か、訪問日、同月履歴、施設基準を確認できない場合は確定算定へ入れない。今回のカルテから在宅関連イベントを抽出した実行はあったが、在宅診療料には昇格しなかった。

### 2. 現在の受診区分が構造化入力にない

現行のfee契約が受け付ける`setting`は `outpatient` と `inpatient` の2値だけであり、`home_visit` や `house_call` を表現できない。データには在宅療養や次回訪問に関する記載があるが、今回の受診自体が訪問診療か往診か、同一建物居住者か否かを確定する構造化フィールドがない。将来予定の記述を現在の実施事実として扱わない現行方針は安全側である。

### 3. 施設基準が未登録

実測時の施設基準キーは0件だった。在宅データ提出加算と訪問診療時のベースアップ評価料を、カルテ本文だけから施設の届出済み加算として確定することはできない。

### 4. 外部の請求履歴が算定履歴へ接続されていない

評価ではSTGに新規患者を作るため、初回セッションが初診として処理された。既知の開始日を追加しても、履歴参照は最大12か月であり、`buildPriorHistoryOptions` は過去セッションの確定明細を参照する。2019年の開始日だけでは、2026年6月の初診・再診判定に必要な直近請求履歴にはならなかった。

## パフォーマンス

算定6回のクライアント応答時間中央値は12,355.65ms、Cloud Run内処理の中央値は12,026.50msだった。

| 処理 | 件数 | 最小 | 中央値 | 平均 | 最大 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 算定API（クライアント） | 6 | 8,020.68ms | 12,355.65ms | 12,263.84ms | 17,175.36ms |
| 算定処理（Cloud Run） | 6 | 7,826ms | 12,026.50ms | 11,953.17ms | 16,667ms |
| OpenAI provider | 6 | 7,157ms | 8,410ms | 9,978.17ms | 14,473ms |
| Python算定（全件） | 6 | 23ms | 28ms | 1,115.67ms | 6,560ms |
| Python算定（初回除外） | 5 | 23ms | 28ms | 26.80ms | 30ms |
| 月次集計API | 3 | 135.38ms | 219.40ms | 201.88ms | 250.85ms |
| 差分診断API | 3 | 162.24ms | 210.82ms | 195.71ms | 214.08ms |

初回のPython算定6,560msはコールドスタートの影響で、残り5回は23〜30msだった。定常時の主な待ち時間はOpenAIを含む準備段である。6回中5回はプロンプトキャッシュがヒットし、キャッシュ入力比率は68.16〜86.33%だった。

## 改善優先度

### P0

1. 現在の受診区分を `outpatient / home_visit / house_call / inpatient` などの構造化入力として受け取り、訪問日・同一建物区分も明示する。
2. 施設基準を有効期間付きで登録し、算定日に有効な届出だけを在宅系加算へ渡す。
3. 在宅医療を一律review-onlyから外す前に、必要入力と決定論ルールを揃え、不足時は引き続き要確認にする。

### P1

1. 既存レセ・医事会計から直近請求履歴を患者へ紐付け、初診・再診や同月履歴の判定に使用する。
2. 同じカルテで確認事項が3〜9件に揺れる問題を、抽出スキーマの制約、重複排除、決定論的な正規化で抑える。

### P2

1. 本評価をSTGの定期評価へ追加し、全UKE再現率と「入力から導出可能な項目」の再現率を分けて継続計測する。
2. OpenAI準備段の出力安定性とレイテンシを追跡し、低リスクケースの決定論リカバリを拡張する。
