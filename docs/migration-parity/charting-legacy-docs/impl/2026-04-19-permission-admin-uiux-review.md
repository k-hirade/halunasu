# 権限管理画面 UI/UX ギャップレビュー

Date: 2026-04-19
Scope: `apps/web/components/admin-console.js` の `members` タブ（ラベル: 「権限管理」）とその関連モーダル（メンバー追加 / パスワード再設定 / プロンプト割当変更）
Branch: `stg`
関連: `docs/impl/2026-04-16-uiux-b2b-gap-analysis.md`（全体の B2B UX ギャップ）、`docs/core/06-screen-flows.md`、`packages/contracts/src/index.js`（ロール定義）

## 目的

本ドキュメントは、現行の「設定 > 権限管理」画面を **Okta / Auth0 / AWS IAM / Microsoft Entra / Notion / Linear / Salesforce / HubSpot / Epic (Hyperspace) の User & Role 管理 UI** と突き合わせ、B2B SaaS としての最低ラインに到達するために足りていない UI/UX 論点を洗い出す。docs/impl/2026-04-16 が "全画面" 視点だったのに対し、本書は **権限管理のみ** に絞ってディテールを見る。

---

## 現状サマリ

- 一覧: 1 枚のテーブル（6 列 × 各メンバー 1 行）
  - 氏名 / 個人ID / 権限 / 録音方法 / プロンプト / 操作
  - 氏名セル内に "利用中・停止中" と "MFA 登録済み・未登録" を `small` で重畳
  - 権限セルは "primary + secondary" の 2 行テキスト表示（B2B 標準のチップ/バッジではない）
  - 録音方法セルはそのまま `<select>` で即時保存
  - 操作列に 5 ボタン：`割当変更` / `パスワード` / `失効` / `MFAリセット` / `停止|再開`
- フィルタ: 氏名・個人 ID・録音方法ラベル・プロンプト名を含むフリーテキスト検索のみ。ロール、ステータス、MFA、最終ログインでの絞り込み不可
- 並び替え: 列ヘッダークリックソートなし。配列は API 返却順のまま
- 追加: 右上 `メンバーを追加` → モーダル（displayName / loginId / 初期パスワード / 既定の録音方法 / ロールチェックボックス）
- パスワード再設定: モーダルで管理者が 12 文字以上を入力して保存（自動生成・コピー・強度表示なし）
- ロール割当: 一覧上では変更不可。メンバー追加時のみチェックボックス。**既存メンバーのロール変更 UI が見当たらない**
- 組織切替: 画面ヘッダ右側に `<select>`（ `platform_admin` のみ）
- バルク操作 / CSV インポート・エクスポート / 招待メール送付 / 最終ログイン表示 / ロール権限の詳細プレビュー: **いずれも無し**

## 評価フレーム

以下 7 軸で B2B 標準と比較：

1. **一覧の発見性**（数十〜数百メンバー規模で誰が何の権限かが一目で分かるか）
2. **ロールと権限の透明性**（そのロールが何を許可するのかを画面上で確認できるか）
3. **メンバーのライフサイクル運用**（招待 → 再発行 → 停止 → 削除を迷わず進められるか）
4. **ミスを減らす防御設計**（自分の権限剥奪、最後の管理者削除、全員 MFA 解除などの自己破壊を止めるか）
5. **操作効率**（バルク操作、並び替え、キーボード、行内編集）
6. **情報アーキテクチャ**（複数組織・施設・診療科・録音端末との関係が見えるか）
7. **アクセシビリティとモバイル閲覧**（色以外の識別、`<select>` の 44px タップ領域、狭幅レイアウト）

## P0 — B2B 導入の前提として欠けているもの

### 1. 既存メンバーのロール変更 UI が無い

- **現状**: メンバー作成時のチェックボックスが唯一のロール編集 UI。作成後にロールを増減する画面上の導線が無く、`/api/v1/admin/members/:id/roles` 相当の呼び出し口もフロントから到達できない（`admin-console.js` 全体に `roles` を PATCH するコードが見当たらない）。
- **B2B 標準**: Okta / Entra / Notion / Linear いずれも "メンバーを開く → 権限を編集" が 1 操作でできる。ロールは時間と共に変わる前提。
- **何が問題か**:
  - 「医師 → 病院管理者」「看護師 → 医療クラーク」の昇格/降格運用が、画面上はサポート外と解釈される
  - 回避策として「メンバーを削除して作り直す」を選ぶと、`memberId` が変わって監査ログの一貫性・担当セッションの紐付けが切れる
- **提案**:
  - 行アクションに `権限を変更` を追加し、既存のロールチェックボックスをモーダルで再利用
  - もしくは行をクリックしたら右側に "メンバー詳細ドロワー" を開き、その中で `基本情報 / 権限 / セキュリティ / アクティビティ` のタブを切る
  - **最後の `org_admin`/`org_owner` を降格しようとした場合は保存をブロック**（後述 #10）

### 2. 操作列の 5 ボタン並列が危険操作を埋没させる

- **現状**: `割当変更` / `パスワード` / `失効` / `MFAリセット` / `停止` の 5 ボタンが全て `btn--ghost` で同列に並ぶ。破壊系（`失効`=セッション強制ログアウト、`停止`=アカウント無効化、`MFAリセット`=MFA 再登録強制）と日常系（`割当変更`）が視覚的に区別されない。
- **B2B 標準**: Linear / Notion / Salesforce いずれも、破壊系はオーバーフロー（`⋯`）メニュー内の赤字、日常系だけが行頭に露出する。
- **何が問題か**:
  - `失効` と `停止` はラベルが 2 文字で意味の重なりが強く、管理者が取り違えやすい（`失効` はセッション失効、`停止` は is-active=false）
  - 狭幅では行アクション列が折り返して 2 段になり、間違ったボタンを押しやすい
- **提案**:
  - 主要導線 1〜2 個のみ露出（例: `詳細を開く`）、残りは `⋯` メニューへ
  - メニュー内で `アカウントを無効化`（= `停止`）と `ログインセッションを失効`（= `失効`）をフルセンテンスのラベルに変更し、`aria-label` と `title` を併記
  - 破壊系は赤字＋先頭にアイコン（`log-out`, `alert-triangle`）

### 3. "停止" の確認ダイアログが無い

- **現状**: `revokeMemberSessions` / `resetMemberMfa` / `revokeTrustedRecorder` はいずれも `window.confirm` を呼ぶが、`updateMemberStatus(member, "disabled")` は確認なしに即時 API 呼び出し。押し間違いで当該医師がログイン不能になる。
- **B2B 標準**: アカウント無効化は必ず「何が起きるか（ログイン不可、進行中セッション？、データは保持）」を明記した確認モーダル。Okta・Entra は確認＋「理由」入力欄を要求するのが一般的。
- **提案**:
  - 既存の `AdminModal` を使い、`停止` と `再開` の両方で 1 段置く
  - 説明文には「進行中の診療セッションは失効しません / ログイン済みのセッションは直ちに切れません（`失効` と組み合わせて運用）」などの挙動を明示
  - できれば「停止の理由」を任意記入させて `audit-events` に残す（後述 #12）

### 4. パスワード再設定 UI が B2B 水準に届いていない

- **現状**: モーダルで管理者が `minLength={12}` のパスワードを手入力 → 保存 → トースト「再設定しました」で終了。強度表示・生成ボタン・クリップボードコピー・メール送付・ワンタイムリンク配布のいずれも無い。
- **B2B 標準**:
  - Okta / Entra / AWS IAM: 「セキュアなランダムパスワードを生成」「クリップボードにコピー」「メールで送信」「初回ログインで強制変更」から選択
  - 最低でも **"設定後、対象者に伝達する必要があります" の注意書き** と、**生成パスワードの 1 回だけ表示**
- **何が問題か**:
  - 管理者が自分の頭で 12 文字を毎回考える → 弱いパスワードになりやすい（医療 SaaS ではリスク大）
  - 設定後の伝達手段が画面上に示されない → Slack/メール/口頭など非公式チャネル経由で運用されがち
- **提案**:
  - `セキュアに生成` ボタン（`crypto.getRandomValues` で 16 文字以上）＋ `コピー` ＋ `目を開く/閉じる` アイコン
  - 保存後のトーストではなく **インライン表示**: 「新しいパスワードを本人に安全な経路で伝達してください。画面を閉じると再表示できません」
  - `forceChangeOnNextLogin` フラグを将来追加（今回実装しないならドキュメント化）

### 5. メンバー情報の多くが "見えるだけ" で編集不可

- **現状**: 一覧で閲覧できる項目は displayName / loginId / status / MFA / role / recordingSource / prompt だが、編集できるのは録音方法 `<select>` とプロンプト割当だけ。displayName / loginId / role / MFA 方針の編集導線は無い。
- **B2B 標準**: 行を展開 or 詳細ドロワーで `displayName / email / role / department / manager` を編集可能。ID は基本不変で、それ以外は編集可。
- **提案**: 行クリックで開く「メンバー詳細ドロワー」を基盤として用意し、そこに以下のタブを置く：
  - `基本情報`（displayName, 所属/診療科※将来, タイムゾーン※将来）
  - `権限`（ロール チェックボックス、実効権限プレビュー※ #7）
  - `セキュリティ`（MFA 必須トグル、MFA 登録状況、最終ログイン、`ログインセッションを失効`、`パスワード再設定`）
  - `アクティビティ`（このメンバーを対象にした操作ログ、当該メンバーが actor の操作ログ）
- **補足**: 将来の運用拡張（部署・診療科・マネージャ）を受け入れる器として、ドロワー型が折りたたみ式インライン編集より拡張しやすい

### 6. 招待フロー（Invite by Email）が存在しない

- **現状**: メンバー追加モーダルで管理者が `loginId` と初期パスワードを手入力 → 作成完了 → **本人への伝達は画面外**。
- **B2B 標準**: ほぼ全ての B2B SaaS がメール招待（ワンタイムリンク）+ 本人が初回ログイン時にパスワード/MFA 設定。Okta / Notion / Linear / Slack / HubSpot は完全にメール中心。
- **なぜ重要か**:
  - 医療機関では「管理者 → 医師 → 本人」と口頭・紙で初期パスワードが受け渡されがち。初期パスワードがメモ紙や Slack DM に残る運用リスク。
  - メール（あるいは SMS / Teams）での招待＋初回必須変更が最もセキュア
- **提案（短期）**:
  - 現行の「管理者が作成」は残したまま、`メンバーを招待` モードを追加。招待コード（ワンタイムリンク）を表示 → 管理者がそれを対象者に共有
  - `/api/v1/admin/members/invite` のエンドポイント新設が必要。pairing token と似た仕組みで単回使用・短期 TTL の発行
- **提案（中期）**:
  - 将来 `email` フィールドを追加し、Resend / SendGrid などで自動送信

## P1 — 熟達者／大規模運用で必要になる機能

### 7. ロールの実効権限プレビューが無い

- **現状**: ロールは日本語ラベル（`医師` / `病院管理者` / `医療クラーク` / `診療閲覧` …）の列挙のみ。`packages/contracts/src/index.js` では各ロールに 10 前後の `permissions` 文字列が定義されているが、画面上は見えない。
- **B2B 標準**: Entra / Salesforce / HubSpot / Notion はロール名にホバー or クリックで「このロールができること／できないこと」のチェックリストがポップオーバーで出る。Okta には "Role Diff" ビューもある。
- **提案**:
  - `roleDefinition.description` は既にあるので、`member-role-summary` のロールラベルに `aria-describedby` のツールチップを付け、ホバーで `description + permissions` を出す
  - メンバー追加・権限変更モーダルでは、チェックしたロールの和集合でできる主要操作（セッション作成 / 全医師のセッション閲覧 / メンバー管理 / 操作ログ閲覧）を `権限プレビュー` 領域に表示
  - 実装は静的: `permissions` 配列 → 既定の「何ができる」文言マップを 1 本持つ

### 8. 並び替え・フィルタ・保存済ビューが無い

- **現状**: 検索ボックス 1 本のみ。`利用中のみ表示` / `MFA 未登録のみ` / `ロール=医師` などの絞り込み不可。件数 `N 名` ピルのみ表示。
- **B2B 標準**:
  - 列ヘッダークリックでソート（氏名・作成日・最終ログイン・ロール・ステータス）
  - フィルタチップ（`ロール` / `ステータス` / `MFA 登録` / `作成 X 日以内`）
  - 保存済みビュー（`未登録 MFA` / `長期未ログイン`）
- **提案**:
  - まず無料で効くのが **列ヘッダソート**（全てクライアント側で完結）と、**status/MFA/role のチップフィルタ**
  - 後続で localStorage に "保存済みビュー" を 3 枚まで持たせる

### 9. 最終ログイン・ログイン履歴の表示が無い

- **現状**: メンバーに `mfaEnrolledAt` は載るが `lastLoginAt`・`lastSeenAt`・`passwordChangedAt` の表示が無い。
- **B2B 標準**: 監査観点で「90 日ログインしていない管理者がいる」「退職者が残っている」を検出するため、ほぼ全ての B2B SaaS が一覧に `最終ログイン` 列を持つ。医療領域では J-SOX/ISMS 監査の必須項目。
- **提案**:
  - `GET /api/v1/admin/members` レスポンスに `lastLoginAt` を追加（Firestore 側の `members/{id}.lastLoginAt` を確認。無ければ `operator-access-token` 発行時に更新するフックを入れる）
  - 一覧に `最終ログイン` 列を追加。90 日以上前なら `badge--warning` を重畳。ドロワーには簡易タイムライン
- **関連**: 信頼録音端末テーブルには既に `lastSeenAt` があるのにメンバーテーブルに無いのは一貫性が崩れている

### 10. 自己破壊防止（Guardrail）が弱い

- **現状**:
  - 自分自身を `停止` するボタンは無効化されているが、「自分を無効化できません」の文言は出ず、ホバーツールチップも無い（ただ押せないだけ）
  - 自分の `org_admin` ロールチェックを外す UI は無い（=ロール編集 UI 自体が無いため、実質ブロックされている）
  - `失効`（セッション失効）は自分自身にも適用され、実行すると `clearAccess()` で強制ログアウト。押し間違いのリスク大
  - 「組織内最後の `org_admin`/`org_owner` を `停止` する」「最後の `platform_admin` の MFA をリセットする」は現状ブロックなし（API 層での保護はこのレビュー範囲外）
- **B2B 標準**: "Last admin lockout" ガードは B2B SaaS の常識。Notion / Linear / Entra / Okta いずれも、ブラウザ側で警告 + サーバ側で 409 を返す 2 段構え。
- **提案**:
  - disabled ボタンにツールチップ（`自分自身は停止できません`）
  - 自分の `失効` は `確認モーダル` + 「自分を含みます。実行すると再ログインが必要です」と明示
  - 「このメンバーを停止すると、病院内の管理者が 0 人になります」など、**実行前に組織全体の状態をチェックして警告**
  - 最後の管理者の降格/停止は UI で `disabled` + `title` 提示

### 11. 表示中の組織スコープが一覧上で見えづらい

- **現状**: 組織切替セレクタは画面右上ヘッダに `医療機関: <select>` とあるが、一覧テーブルには組織名が無い。`platform_admin` が複数組織を横断する運用では「いま表示しているメンバーは A 病院のもの」と認識しづらく、操作誤爆の元。
- **B2B 標準**: マルチテナント管理画面は上部にテナント名のバッジ常駐、操作系ボタンに `@organization` のコンテキストラベル。
- **提案**:
  - 画面タイトル直下に `●● 病院の権限管理` と大きく。切り替えるとアニメーションでフェード
  - `メンバーを追加` ボタンのラベルを `●● 病院にメンバーを追加` に組織名を差し込む（長ければツールチップでもよい）
  - 一覧が空のときに「●● 病院にはまだメンバーが登録されていません」のメッセージ

## P2 — 信頼感・プロフェッショナル感を上げる装飾

### 12. メンバー単位の操作履歴が追えない

- **現状**: `操作ログ` タブは全体ログをフィルタするだけ。特定メンバー行から「このメンバーに関する操作ログ」への導線が無い。`applyTargetLabel` / `actorLabel` の仕組みはあるのに UI から使えない。
- **B2B 標準**: 一覧の各行から "View activity" でフィルタ済みログに飛ぶのは業界標準。
- **提案**: 行コンテキストメニューに `操作ログを表示` を追加し、`section=audit&memberId=xxx` で audit タブを開き、既存 `auditTypeFilter` と並列に `memberId` フィルタを通す

### 13. バルク操作と CSV インポート/エクスポート

- **現状**: 複数選択の手段が無い。30 人の医師を一斉にプロンプト再割当、という運用は全て手作業。
- **B2B 標準**: Entra / Okta は CSV 一括招待が標準。HubSpot / Linear は行チェックボックス＋一括削除/エクスポート。
- **提案**:
  - `<input type="checkbox">` を行先頭に追加、選択時に `一括操作バー` をテーブル上部に出す
  - 一括で `停止 / 再開 / プロンプト再割当 / CSV でダウンロード` を可能に
  - CSV ダウンロードは `memberId / displayName / loginId / roles / status / lastLoginAt / mfaEnrolledAt` をブラウザで組み立て → Blob → ダウンロード。API 変更なしで実装可能
  - CSV アップロードは middleware 追加が必要なので後続

### 14. 空状態・オンボーディングが薄い

- **現状**: メンバー 0 件時は「条件に一致するメンバーはありません」だけ。新規病院が開設された直後にこのメッセージしか出ないため、何をすべきか分からない。
- **B2B 標準**: Intercom / Notion / Linear は空状態にイラスト＋主 CTA＋副 CTA（ドキュメントへ）。
- **提案**:
  - フィルタヒット 0 件 と、本当に 0 名 を分岐
  - 本当に 0 名のときは「最初の医師を追加しましょう」＋ CTA `メンバーを追加` ＋ `権限モデルを学ぶ`（docs リンク）

### 15. アクセシビリティ・モバイル閲覧

- **現状**:
  - `member-role-summary` はテキスト中心だが、ロールごとの色分けが無いため一覧上でロールを瞬時に識別できない
  - 行アクション列の 5 ボタンは狭幅で崩れる（既存 CSS のメディアクエリはあるが、タップ領域が 36px と小さく WCAG 2.5.5 最小 44px を切る）
  - `<select>` の既定録音方法は即時保存で Undo が無く、読み取り専用モードの可視表現も弱い
  - `badge--finalizing` を「同期中」として使い回しているが、`role="status"` `aria-live` の扱いが明示されていない
- **提案**:
  - ロールにはテキスト + 小さなチップ色（カテゴリ別: platform=紫 / organization=インディゴ / clinical=ティール / operations=グレー / governance=アンバー）。`MEMBER_ROLE_DEFINITIONS.category` が既に分類済みなので配色テーブルを 1 本作るだけ
  - 行アクションボタンは最低 40×40、`⋯` メニューに集約した上で主要 1 ボタンのみ露出
  - 録音方法 `<select>` 変更直後のトーストに `元に戻す` を 10 秒だけ表示
  - `admin-filter-bar` の `同期中` バッジは `aria-live="polite"` を明示し、`prefers-reduced-motion` を尊重

### 16. ラベルの表記ゆれと曖昧さ

- **現状**:
  - `権限` / `ロール` / `医師` / `アカウント` が同じテキスト色 (`var(--ink)`) で並ぶため階層が見えづらい
  - `停止` と `失効` の違いは画面上では説明されない
  - `プロンプト` 列は `assignedFormatName(member, formats)` がフォーマット名そのままで、"なぜこのメンバーにこのプロンプトが付いているのか"（組織標準 or 個別割当）が区別されない
- **提案**:
  - 用語集を 1 枚作って画面と docs を揃える。`docs/core/06-screen-flows.md` に `"停止" = アカウント無効化`、`"失効" = ログインセッションの強制終了` の定義節を追加
  - プロンプト列は `● 個別: <name>` / `○ 病院標準に準拠` の 2 行バリアントにして、ソースが一目で分かるように

## 横断的リファクタの方向

現状、`admin-console.js` は 2,125 行の 1 ファイルに 5 タブ分の UI と 10+ モーダルが全部入っている。このままロジックを増やすと保守不能なため、**権限管理画面だけでも最低限のコンポーネント分割** を先にやっておくのが望ましい：

| 抽出先 | 役割 |
|---|---|
| `components/members/member-table.js` | テーブル本体（ソート、フィルタ、バルク選択） |
| `components/members/member-drawer.js` | 行クリックで開く詳細ドロワー（基本情報/権限/セキュリティ/アクティビティ） |
| `components/members/member-create-modal.js` | メンバー追加（招待モード付き） |
| `components/members/password-reset-modal.js` | パスワード再設定（生成/コピー/表示切替） |
| `components/members/role-assignment-modal.js` | 既存メンバーの権限変更 |
| `components/members/role-permission-preview.js` | ロール → 実効権限の説明ブロック |
| `lib/member-query.js` | `useMembers({ filter, sort, paginate })` の基盤フック |

これを入れるタイミングで docs/impl/2026-04-16 の「#3 ユーザーメニュー」「#5 Autosave Pill」などの共通基盤（`lib/autosave-status.js`, ドロワーのトランジション規約）とも揃える。
