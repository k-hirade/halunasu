# 算定ロジック設計

## 最適な設計方針

最適解は、ルールエンジンを中核にし、LLM を補助に使うハイブリッド型である。

LLM 単体で算定を確定する設計は採用しない。厚労科研の検体検査実験でも、LLM は検査オーダーから診療行為への変換では一定の性能を示したが、総合正答率は症例により大きく崩れた。点数計算、同時算定不可、包括、施設基準の最終判定は、決定的ロジックで処理する。

## 全体アーキテクチャ

```text
診療データ
  -> 正規化・構造化
  -> 候補コード抽出
  -> 算定要件チェック
  -> 包括・同時算定不可・回数制限の解決
  -> 最適な算定セット選択
  -> レセプト形式生成
  -> 根拠・警告・人間確認キュー
```

## 入力モデル

診療録やオーダーをそのまま計算しない。まず算定に必要な事実を構造化する。

```json
{
  "patient": {
    "age": 72,
    "sex": "F",
    "insurance": "social",
    "public_expense": []
  },
  "encounter": {
    "date": "2026-06-03",
    "setting": "outpatient",
    "department": "internal_medicine",
    "first_visit": false,
    "time_category": "regular_hours"
  },
  "facility": {
    "beds": 300,
    "dpc": true,
    "accepted_facility_standards": ["..."]
  },
  "events": [
    {
      "type": "lab_order",
      "name": "HbA1c",
      "specimen": "blood",
      "performed_at": "2026-06-03"
    }
  ],
  "history": {
    "same_day_codes": [],
    "same_month_codes": [],
    "prior_month_codes": []
  }
}
```

最低限必要な入力は次の通り。

- 患者属性
  - 年齢、性別、保険種別、公費、負担割合、死亡有無。
- 受診属性
  - 入外区分、診療日、初診/再診、診療科、時間帯、同日複数科、入院日、退院日。
- 医療機関属性
  - 病床数、DPC対象、特定機能病院、施設基準届出、届出受理日、経過措置。
- 診療行為
  - オーダー、実施、結果、診療録記載、説明同意、手術記録、麻酔記録、輸血記録。
- 履歴
  - 同日、同月、過去算定、前回実施日、退院日、転帰。
- 入院/DPC
  - 入院基本料コード、算定日数、病棟、施設基準略称、DPC対象、DPCコード、医療資源病名、手術、処置、副傷病、医療機関別係数。

## マスターとバージョン管理

すべてのマスター、通知、ルールは `effective_from` と `effective_to` を持つ。

```json
{
  "source_id": "mhlw-r8-medical-fee-table",
  "version": "2026-05-01-correction",
  "effective_from": "2026-06-01",
  "published_at": "2026-05-01",
  "url": "https://www.mhlw.go.jp/stf/newpage_67729.html"
}
```

利用するデータソース。

- 医科診療行為マスター
- 歯科診療行為マスター
- 調剤行為マスター
- 医薬品マスター
- 特定器材マスター
- コメントマスター
- 傷病名マスター
- 修飾語マスター
- 医科電子点数表
- 歯科電子点数表
- DPC電子点数表
- 施設基準届出情報
- 疑義解釈、訂正通知、記載要領

## ルールモデル

点数表や通知の文章は、そのまま実行しない。実行可能な制約として保存する。

```json
{
  "rule_id": "D007-example",
  "effective_from": "2026-06-01",
  "target_codes": ["160000000"],
  "type": "eligibility",
  "requires": [
    "encounter.setting == 'outpatient'",
    "event.specimen == 'blood'",
    "not bundled_by_external_visit_fee",
    "same_month_count(target_codes) <= 1"
  ],
  "adds": ["judgement_fee_if_applicable"],
  "excludes": ["mutually_exclusive_code_x"],
  "source": {
    "kind": "notification",
    "url": "https://www.mhlw.go.jp/stf/newpage_67729.html"
  }
}
```

必要なルール種別。

- `eligibility`: 算定要件。
- `add_on`: 加算。
- `derived`: 判断料、採取料などの自動追加候補。
- `exclusive`: 同時算定不可。
- `frequency_limit`: 月1回、週1回、初回のみなど。
- `bundle`: 包括対象。
- `substitution`: A を算定する場合 B に置換。
- `facility_required`: 施設基準必須。
- `documentation_required`: 診療録記載、説明同意、記録必須。
- `human_review`: 医学的判断または情報不足。

## 候補生成

候補生成では、院内オーダー名や記録から標準コード候補を出す。

1. 完全一致。
2. 院内マスター対応表。
3. 同義語、略語、表記ゆれ辞書。
4. LLM または検索モデルによる候補提示。
5. 候補ごとの信頼度と根拠を保存。

候補生成の出力例。

```json
{
  "event_id": "order-123",
  "candidates": [
    {
      "code": "160xxxxx",
      "name": "HbA1c",
      "confidence": 0.98,
      "evidence": ["院内マスター対応表"]
    }
  ]
}
```

## 算定セットの選択

候補コードをすべて足し合わせない。制約充足と最適化で算定セットを決める。

```text
maximize:
  合法に算定できる点数合計
  + 請求漏れ回避スコア
  - 査定リスクペナルティ

subject to:
  算定要件を満たす
  同時算定不可を破らない
  包括対象を除外する
  回数制限を超えない
  施設基準を満たす
  診療録根拠がある
```

重要なのは点数最大化ではなく、根拠のある範囲での合法最大化である。根拠が不足する候補は自動算定せず、`needs_review` にする。

## LLM の役割

LLM に任せてよいもの。

- オーダー名の表記ゆれ補正。
- 院内名称から標準マスター候補へのマッピング。
- 診療録から算定根拠の抽出。
- 疑義解釈や通知の要約。
- 人間向けの根拠説明生成。

LLM に任せないもの。

- 点数計算の最終確定。
- 同時算定不可の最終判定。
- 包括除外の最終判定。
- 高額手術、DPC、施設基準絡みの最終確定。
- 査定リスクが高い請求の自動確定。

## パターン別設計

### A. 外来出来高

最初に実装しやすい。初再診、検査、投薬、注射、画像、処置を中心に扱う。

主な処理。

- 初診/再診/外来診療料の選択。
- 同月回数制限。
- 判断料、採取料、投薬基本料、注射手技料、画像診断管理加算などの追加候補。
- 同時算定不可の解決。
- 包括される検査や処置の除外。

現在のMVPでは、初再診・外来診療料、検体検査、採取料、外来迅速検体検査加算、投薬基本料、注射手技料、一部処置料、一部画像診断料を、共通の `ClaimContext` から `CalculationResult` にまとめる。入力済み診療行為・医薬品・注射薬・特定器材は `confirmed`、ロジックが追加する候補は `candidate`、情報不足や二重追加回避は `needs_review` または `blocked` のメッセージとして返す。

注射は薬剤コードだけでは注射経路や加算対象を確定できないため、`InjectionOrder` で薬剤総量を計算し、`InjectionOptionContext` で皮内・皮下・筋肉内、静脈内、点滴、中心静脈、関節腔内、硝子体内などの経路と加算フラグを明示する。皮内・皮下・筋肉内注射と静脈内注射は入院中患者では薬剤料側だけにする。輸液量、投与時間、同日複数注射のまとめ、その他の入院中包括は後続の制約解決で扱う。

処置は自然文の処置名から自動確定しない。`TreatmentOrder` で処置種別を明示し、創傷処置・熱傷処置・皮膚科軟膏処置のように面積で点数が変わるものは `TreatmentAreaSizeKind` も必須にする。複数部位の面積合算、在宅指導管理料等による包括、処置薬剤・材料、乳幼児加算、時間外・休日・深夜加算、診療録記載要件は後続の制約解決で扱う。

画像診断も自然文のオーダー名から自動確定しない。`ImagingOrder` で単純撮影、造影剤使用撮影、乳房撮影、CT、MRIを明示し、単純撮影では写真診断区分と撮影方式、CT/MRIでは機器区分、造影有無、電子画像管理を渡す。画像診断管理加算と遠隔画像診断管理加算は、明示フラグと施設基準辞書の `画１` - `画４` / `遠画` から候補化する。複数方向・複数部位の逓減、共同利用施設の施設基準、核医学/PET、造影剤薬剤料・材料は後続の制約解決で扱う。

### B. 検体検査特化

最初の検証対象に最適。

```text
検査オーダー
  -> 検査コード候補
  -> 検体別グルーピング
  -> 項目数算定
  -> 出来高算定
  -> 判断料/採取料/加算追加
  -> 不可項目削除
  -> 合計点数
```

注意点。

- CRP のように包括グループに入らない項目。
- 血液と尿など、検体横断の項目数カウント。
- 同月2回目以降の判断料。
- 検査結果の陽性/陰性で分岐する追加検査。
- 外来診療料や小児科外来診療料による包括。

### C. 画像診断

基本構成は `撮影料 + 診断料 + 加算 + 部位コメント`。

必要な入力。

- 撮影部位。
- 撮影枚数。
- 造影有無。
- 読影有無。
- 電子画像管理の有無。
- コメントマスター。

### D. 手術、麻酔、材料

難度が高く、初期の完全自動化対象にしない。

必要な入力。

- 術式、左右、部位、創数。
- 併施手術。
- 麻酔種別、麻酔時間、監視内容。
- 輸血記録。
- 使用薬剤。
- 特定保険医療材料。
- 手術記録と診療録根拠。

高点数で査定リスクが大きいため、初期は `候補提示 + 根拠表示 + 人間承認` にする。

### E. 入院出来高/DPC

入院出来高は、外来出来高と同じ共通出力を使うが、病棟、入退院日、転棟、入院中包括が絡むため別エンジンとして扱う。2026-05-18時点のStep6では、入院基本料を明示コード入力で候補化する入口を実装済みである。

```text
入院基本料コード + 算定日数
  -> 施設基準略称をrule keyへ正規化
  -> hospital_profile / ClaimContext の施設基準届出を確認
  -> 医科診療行為マスターから点数を解決
  -> candidate行
```

DPC 入院は、出来高とは別系統で処理する。現時点では `dpc_claim` または `dpc_code` を取り込めるが、点数計算はせず `needs_review` に止める。

```text
主傷病/医療資源病名
  -> DPCコード候補
  -> 手術・処置等1/2、定義副傷病、重症度で分類
  -> 入院期間I/II/IIIの点数
  -> 医療機関別係数
  -> 包括範囲と出来高算定可能項目の分離
  -> 合計点数
```

特に注意するもの。

- 入院期間III超過後の扱い。
- 手術、麻酔、放射線治療などの出来高部分。
- 1,000点以上処置。
- 高額薬剤。
- 医療機関別係数。
- DPC対象外患者。
- DPC包括範囲と出来高算定項目の分離。

## 出力モデル

算定結果は、点数だけではなく根拠と警告を含める。

```json
{
  "claim_items": [
    {
      "code": "160xxxxx",
      "name": "HbA1c",
      "points": 49,
      "quantity": 1,
      "status": "confirmed",
      "sources": ["rule:D007-example", "master:medical-procedure:2026-06"]
    }
  ],
  "total_points": 49,
  "warnings": [
    {
      "level": "review",
      "reason": "診療録記載の有無が入力から確認できない",
      "required_action": "human_review"
    }
  ]
}
```
