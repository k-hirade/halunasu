# 診療報酬算定の全体像

## 結論

診療報酬算定は、保険診療で行った医療行為を点数化し、合計点数を金額化して、患者負担分と保険者請求分に分ける仕組みである。原則は `1点 = 10円`。

ただし、実務上は単なる点数表検索ではない。点数表、算定要件、施設基準、届出、包括評価、同時算定不可、疑義解釈、レセプト記載要領、審査上の医学的判断を同時に扱う必要がある。

## 制度構造

診療報酬は大きく次の要素から構成される。

- 基本診療料
  - 初診料、再診料、外来診療料、入院基本料など。
  - 基本診療料には、簡単な検査や処置などが含まれる場合がある。
- 特掲診療料
  - 医学管理、在宅医療、検査、画像診断、投薬、注射、リハビリテーション、処置、手術、麻酔、放射線治療、病理診断など。
- 薬剤
  - 薬価基準に基づく。
- 特定保険医療材料
  - 材料価格基準に基づく。
- 加算、減算
  - 時間外、休日、深夜、乳幼児、施設基準、医療安全、医療DX、地域連携など。
- 包括評価
  - DPC/PDPS、外来診療料、特定入院料、小児科外来診療料など、一部項目を包括する体系。

## 実務フロー

1. 保険診療の対象か確認する。
2. 患者属性を確認する。
   - 年齢、性別、負担割合、公費、保険種別、死亡有無など。
3. 診療属性を確認する。
   - 入外区分、診療日、初診/再診、診療科、時間帯、同日複数科、同月履歴など。
4. 医療機関属性を確認する。
   - 病床数、DPC対象、特定機能病院、施設基準届出、受理日、経過措置など。
5. 診療行為候補を標準コードに変換する。
   - 医科診療行為マスター、医薬品マスター、特定器材マスター、傷病名マスター、コメントマスターを使う。
6. 算定要件を評価する。
   - 対象疾患、実施条件、回数制限、期間制限、診療録記載、説明同意、施設基準など。
7. 包括、同時算定不可、置換、加算、減算を解決する。
8. 合計点数を算出する。
9. 電子レセプト形式に変換する。
10. 不確実なものを人間確認へ回す。

## 完璧な自動算定が難しい理由

全領域、全症例で完璧な算定ロジックを作ることは現実的ではない。

理由は次の通り。

1. 制度が固定ルールだけで完結していない。
   - 厚労省の令和8年度改定ページには、点数表、留意事項通知、施設基準、疑義解釈、訂正通知、記載要領、DPC、給付調整が分かれて掲載されている。
   - 令和8年度改定では、2026年3月から5月にかけて疑義解釈や訂正通知が複数回出ている。
2. 審査には医学的判断が入る。
   - 支払基金は、審査が法令、点数表、通知を踏まえつつ、各審査委員会の医学的見解に基づくと説明している。
   - 公表事例も、すべての個別診療内容に画一的に適用されるものではない。
3. レセプト情報だけでは足りない。
   - 算定要件には、診療録記載、患者状態、医学的必要性、説明同意、検査結果、手術記録、施設基準などが含まれる。
4. 公式側も完全自動ではない。
   - 支払基金のAIは、査定や返戻の可能性が高いレセプトを人の審査へ振り分ける用途で使われている。

## 実装で目指すべき状態

目標は「完璧な自動算定」ではなく、次の状態である。

- 根拠付きで高精度に算定できる。
- ルールで確定できるものは自動確定する。
- 医学的判断や情報不足があるものは `needs_review` にする。
- 請求漏れ、過剰請求、査定リスクを説明可能にする。
- 改定、疑義解釈、訂正通知をバージョン管理して再計算できる。
